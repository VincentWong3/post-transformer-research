# Training Compute-Optimal Large Language Models（2022）

---

## 一、先搞清楚“问题在哪里”

在 2022 年之前，训练大语言模型（LLM）的社区有一个默认共识：**模型越大越好**。GPT-3（1750 亿参数）、Gopher（2800 亿参数）、Megatron-Turing NLG（5300 亿参数）——这些模型的规模每半年翻一倍，所有人都相信更大的模型会带来更好的性能。

但这里有一个非常实际的问题：**计算资源是有限的**。你有一个固定的计算预算（比如 10^23 FLOPs），你要决定怎么花这笔钱——是训练一个更大的模型但用更少的数据，还是训练一个小一点的模型但用更多的数据？

这个问题看似简单，但当时没人能给出一个确切的答案。主流做法是：有多少数据就用多少数据，模型能多大就做多大。这种“有多少资源就堆多大模型”的思路，本质上回避了最核心的权衡问题。

**没有这篇论文，原来的技术路线会卡在哪里？**

会卡在“资源浪费”上。你会花 1000 万美元训练一个模型，但其中 30% 的计算量可能被浪费了——因为模型太大而数据太少，或者数据太多而模型太小。更糟的是，你无法知道哪个方向是错的。整个领域在盲目地堆参数，却没有一个理论框架来指导“给定计算预算，到底应该怎么分配”。

---

## 二、旧路线的真正问题

要理解这个问题的严重性，我们需要看清当时存在的三个基本矛盾。

### 矛盾一：Scaling Laws 的不完备性

2020 年 Kaplan 等人提出的 Scaling Laws 告诉社区：模型性能与模型大小、数据大小、计算量之间存在幂律关系。但他们给出的结论是：

> **模型大小比数据大小更重要**——当计算预算增加时，应该主要增加模型参数，数据量只需要缓慢增加。

具体来说，Kaplan 等人给出的最优分配是：**模型参数增加 8 倍，数据只增加 1 倍**。这个结论直接导致了“模型越大越好”的狂热。

但这里有个致命的漏洞：Kaplan 等人的实验是在模型参数不超过 1.5B、计算预算不超过 10^21 FLOPs 的条件下做的。**这个结论能否外推到 GPT-3 规模的模型？没人知道。**

### 矛盾二：训练不足的隐形成本

如果你按照 Kaplan 的建议去训练，会发生什么？

假设你有 10^23 FLOPs 的计算预算（大约是 GPT-3 的 3 倍），按照 Kaplan 的建议，你会训练一个约 10 万亿参数的模型，但只用 50B 左右的 token。

问题来了：**这个模型会被严重欠训练**。它的参数量巨大，但每个参数见到的数据太少，模型根本没有学到足够的模式。这就像一个学生读了 100 本书但每本书只读第一页——他拥有巨大的“容量”，但输入的信息量不够。

### 矛盾三：无法验证的假设

当时没有任何人做过系统的实验来回答这个问题：

> 给定固定计算预算 F，模型参数 N 和训练数据 D 的最优比例是多少？

原因很简单：**做这个实验太贵了**。你需要训练几十个不同大小、不同数据量的模型，每个都跑到收敛，然后比较它们的性能。这在当时的计算资源下几乎不可能。

所以整个领域都在一个未经验证的假设上运行：模型越大越好，数据够用就行。

### 这个矛盾的后果

直白说，如果你在 2022 年之前训练一个 LLM，你大概率浪费了 20%-50% 的计算资源。你训练了一个 100B 的模型，但用 300B token 的数据训练它可能效果更好——而且更便宜。

这个问题的本质是：

> **我们不知道如何高效地花计算预算。**

这不只是一个学术问题。在 GPT-3 的训练成本高达 1200 万美元的时代，每个百分点的效率提升都意味着数百万美元的节省。

## 三、这篇论文的核心思路

> **这篇论文的核心思路是：给定固定的计算预算，模型参数 N 和训练数据 D 之间存在一个最优比例，这个比例可以用一个简单的公式来预测。**

这个思路看似简单，但它直接推翻了 Kaplan 等人的结论。DeepMind 团队做了大规模的实验，训练了 400+ 个模型，从 70M 到 16B 参数不等，使用不同的数据量进行训练，然后拟合出一个新的 Scaling Law。

**他们推翻了什么旧假设？**

Kaplan 的结论是：模型参数比数据更重要（N 增长快于 D）。而 Chinchilla 的结论是：**模型参数和数据应该等比例增长**——计算预算翻倍时，模型参数和数据量应该同时翻倍。

**这意味着什么？**

如果你有 10^23 FLOPs 的预算，Kaplan 会建议你训练一个 ~10T 参数的模型用 ~50B token，而 Chinchilla 会建议你训练一个 ~400B 参数的模型用 ~2T token。

**哪个更合理？**

Chinchilla 的实验证明：在相同计算预算下，小模型+更多数据的效果比大模型+更少数据的效果更好。他们用 70B 参数的 Chinchilla 模型，在 1.4T token 上训练，在几乎所有 benchmark 上都超过了 280B 参数的 Gopher。

```text
固定计算预算 F
 ↓
Kaplan 的选择：N↑↑  D↑
 ↓
Chinchilla 的选择：N↑  D↑
 ↓
结果是：Chinchilla 胜出
```

这个结论的直接后果是：**整个 LLM 社区开始重新思考模型设计。** 不是“能做多大做多大”，而是“在给定计算预算下，最优的模型大小和数据量是多少”。

## 四、整体方法与实验流程

这篇论文不是提出一个新的模型架构，而是提出一个**确定最优训练配置的方法论**。所以它的“forward 流程”实际上是实验流程。

### 4.1 实验设计

论文设计了三个独立的实验方法来确定最优的 N/D 比例：

#### 方法一：固定模型大小，变化数据量

对于每个模型大小（70M 到 16B 参数），训练多个模型，每个使用不同数量的 token。然后观察 loss 随 token 数量的变化曲线。

```text
模型大小固定（比如 1B 参数）
 ↓
训练多个模型：100B token, 200B token, 500B token, 1T token
 ↓
记录每个模型的最终 loss
 ↓
找到 loss 开始趋于平缓的点——这就是“数据饱和点”
```

#### 方法二：固定计算预算，变化 N 和 D

给定几个不同的计算预算（比如 10^20, 10^21, 10^22 FLOPs），对于每个预算，训练多个模型，每个模型有不同的 N/D 比例。

```text
计算预算固定（比如 10^21 FLOPs）
 ↓
尝试不同的 N/D 组合：
   - 500M 参数 + 200B token
   - 1B 参数 + 100B token
   - 2B 参数 + 50B token
   - 4B 参数 + 25B token
 ↓
找到 loss 最低的那个组合
```

#### 方法三：使用参数化的 Scaling Law 拟合

这是最理论化的方法。假设 loss 可以表示为：

```
L(N, D) = E + A/N^α + B/D^β
```

其中：
- E 是 irreducible loss（数据本身的熵）
- A/N^α 是模型容量不足带来的 loss
- B/D^β 是训练数据不足带来的 loss

然后用所有实验数据拟合这个函数，得到最优的 N/D 比例。

### 4.2 三个方法的一致性

关键结果是：**三个方法给出了几乎相同的结论**。

最优的 token 数量大约是模型参数的 20 倍。也就是说：

> 对于 1B 参数的模型，最优训练数据是 ~20B token。

这个比例远高于当时的主流做法。

## 五、核心公式与数学直觉

### 公式 1：参数化 Scaling Law

```
L(N, D) = E + A/N^α + B/D^β
```

**这个式子表面上是在算 loss，但真正的作用是把“模型容量”和“数据容量”分离成两个独立的项。**

- `E`：无论你有多大模型、多少数据，都无法降低的 loss。这是任务本身的难度上限。
- `A/N^α`：模型太小带来的 loss。当 N 增大时，这一项减小。
- `B/D^β`：数据太少带来的 loss。当 D 增大时，这一项减小。

**关键洞察**：这两个项是独立的——增大模型不能弥补数据不足，增大数据也不能弥补模型太小。所以最优解是在两者之间找到平衡点。

### 公式 2：最优分配条件

给定计算预算 F，假设训练一个参数为 N 的模型需要 6N 个 FLOPs 每 token（前向+反向），那么：

```
F ≈ 6ND
```

代入 loss 公式，对 N 求导，得到最优条件：

```
Aα/N^(α+1) = Bβ/D^(β+1)
```

**这个式子的直觉是**：在最优配置下，增加一个参数带来的边际收益等于增加一个 token 带来的边际收益。如果左边大于右边，说明模型太小；如果右边大于左边，说明数据太少。

### 公式 3：最优比例

从拟合结果，作者得到：

```
N_opt ∝ F^0.50
D_opt ∝ F^0.50
```

**这意味着**：计算预算翻倍时，模型参数和数据量都应该翻倍。这是一个 1:1 的比例。

对比 Kaplan 的结果：

```
Kaplan: N_opt ∝ F^0.73, D_opt ∝ F^0.27
Chinchilla: N_opt ∝ F^0.50, D_opt ∝ F^0.50
```

**为什么会有这个差异？**

Kaplan 的实验范围太小（模型 ≤ 1.5B，计算预算 ≤ 10^21 FLOPs），而且他们只用了方法二（固定计算预算），没有用方法一和方法三做交叉验证。Chinchilla 的实验覆盖了更大的范围，并且用三种独立方法验证了同一个结论。

### 实际意义

假设你有 10^23 FLOPs 的计算预算：

- **Kaplan 的建议**：N ≈ 10T 参数，D ≈ 50B token
- **Chinchilla 的建议**：N ≈ 400B 参数，D ≈ 2T token

Chinchilla 的建议意味着：**你不需要一个 10T 参数的模型，400B 就够了。但你需要 2T token 的数据，而不是 50B。**

## 六、重要实验结果

### 6.1 Chinchilla vs Gopher

最直接的验证是：Chinchilla（70B 参数，1.4T token）与 Gopher（280B 参数，300B token）的比较。

| 模型 | 参数 | 训练数据 | 计算量 | MMLU | Lambada |
|------|------|----------|--------|------|---------|
| Gopher | 280B | 300B token | 同量级 | 60.0% | 74.5% |
| Chinchilla | 70B | 1.4T token | 同量级 | 67.5% | 77.4% |

**Chinchilla 在 1/4 的参数下，用相同的计算量，在所有 benchmark 上都超过了 Gopher。** 这不是小幅度提升——MMLU 从 60% 到 67.5% 是 7.5 个百分点的绝对提升，这在 LLM 基准测试中是非常显著的。

### 6.2 三个方法的交叉验证

论文最令人信服的部分是三个独立方法给出了一致的结果：

```text
方法一（固定模型大小）：最优 token/参数比 ≈ 20:1
方法二（固定计算预算）：最优 token/参数比 ≈ 20:1
方法三（参数化拟合）：最优 token/参数比 ≈ 20:1
```

这种一致性排除了偶然性。如果只有一个方法，你可能会怀疑是实验设计的问题。但三个完全不同的方法给出相同的答案，这个结论就非常可靠了。

### 6.3 实验证明了什么

实验证明了一个核心假设：

> **在给定计算预算下，模型性能对 N/D 的比例非常敏感。偏离最优比例 10 倍，loss 会显著增加。**

换句话说，如果你用 Kaplam 的建议（N/D = 200:1），你的模型会比最优配置（N/D = 20:1）差很多——即使计算量相同。

### 6.4 实验没有证明什么

实验没有证明“Chinchilla 的配置总是最优的”。它只证明：**在 2022 年的训练技术和数据质量下，这个比例是最优的。** 如果训练技术变了（比如更高效的优化器），或者数据质量变了（更高质量的数据），最优比例可能会变化。

## 七、技术谱系位置

### 7.1 它继承了谁？

- **Kaplan et al. (2020) "Scaling Laws for Neural Language Models"**：Chinchilla 直接继承了这个工作的研究范式——用实验拟合 Scaling Law，然后用它指导模型设计。但 Chinchilla 修正了 Kaplan 的结论。

- **OpenAI 的 scaling 研究传统**：从 GPT-1 到 GPT-3，OpenAI 一直在探索“更大是否更好”。Chinchilla 对这个假设提出了挑战。

### 7.2 它修正了谁？

- **Kaplan 的 Scaling Law**：这是最直接的修正。Kaplan 说 N 比 D 重要，Chinchilla 说 N 和 D 同等重要。

- **GPT-3 的设计哲学**：GPT-3 的选择是 175B 参数 + 300B token，N/D 比 ≈ 0.58:1。Chinchilla 认为这个比例太偏向模型了，应该用更小的模型 + 更多的数据。

### 7.3 它替代了谁？

Chinchilla 实际上替代了整个“模型越大越好”的默认假设。在它之后，社区开始问：“给定我的计算预算，最优的 N 和 D 是多少？”而不是“我能堆多大的模型？”

### 7.4 它启发了谁？

- **LLaMA (Meta, 2023)**：LLaMA 直接采用了 Chinchilla 的结论——训练 7B、13B、33B、65B 参数的模型，每个都用 1T+ token 的数据。LLaMA-13B 在大多数 benchmark 上超过了 GPT-3 175B，这正是 Chinchilla 预测的结果。

- **Falcon (TII, 2023)**：40B 参数 + 1T token，同样是 Chinchilla 最优配置的翻版。

- **MosaicML 的 MPT 系列**：明确使用 Chinchilla 的公式来设计模型。

- **几乎所有后来的开源 LLM**：从 2023 年开始，几乎所有开源 LLM 都在使用 Chinchilla 的 N/D 比例（或接近的比例）。

### 7.5 技术谱系图

```text
Kaplan Scaling Laws (2020)
  │
  ├── GPT-3 (2020): 175B + 300B token
  │
  └── Chinchilla (2022): 70B + 1.4T token
        │
        ├── LLaMA (2023): 7B-65B + 1T-1.4T token
        ├── Falcon (2023): 40B + 1T token
        ├── MPT (2023): 7B-30B + 1T token
        └── Mistral (2023): 7B + 8T token (进一步增加数据量)
```

## 八、时间线位置

### 2020-2022：Scaling 狂热期

- **2020年5月**：GPT-3 发布，175B 参数，证明了 scaling 的有效性
- **2021年12月**：Gopher 发布，280B 参数
- **2022年1月**：Megatron-Turing NLG 530B 发布

整个社区在疯狂堆参数，每个人都想做出更大的模型。

### 2022年3月：Chinchilla 发布

这篇论文在 Gopher 论文发布仅 3 个月后发表。它用 Gopher 团队自己的实验数据，证明 Gopher 本身是次优的——Gopher 有 280B 参数但只用 300B token，而 Chinchilla 用 70B 参数 + 1.4T token 就超过了它。

### 为什么是这个时间点？

有几个原因：

1. **计算资源变得足够便宜**：Chinchilla 的实验需要训练 400+ 个模型，这在 2020 年几乎不可能。到 2022 年，DeepMind 有足够的计算资源来做这个系统性的研究。

2. **Gopher 的教训**：DeepMind 自己刚刚花了大量资源训练 Gopher，然后意识到它的配置可能是次优的。这个教训促使他们做这个研究。

3. **数据瓶颈开始显现**：2022 年，高质量文本数据开始变得稀缺。社区开始意识到“数据不够用”可能比“模型不够大”更严重。

### 后续影响

- **2023年2月**：LLaMA 发布，验证了 Chinchilla 的结论
- **2023年7月**：LLaMA 2 发布，进一步验证
- **2023年9月**：Mistral 7B 发布，用 8T token 训练（远超 Chinchilla 的比例），探索“更多数据是否更好”

```python
}

# 训练数据量

    "7B":   "1.0T tokens",
    "13B":  "1.0T tokens",

    "65B":  "1.4T tokens",
}
```

### 9.3 如何在自己的项目中使用

如果你要训练自己的 LLM，可以按照以下步骤使用 Chinchilla 的结论：

1. **确定你的计算预算 F**（以 FLOPs 为单位）
2. **计算最优模型大小**：`N_opt ≈ (F / 6)^0.5`
3. **计算最优数据量**：`D_opt ≈ (F / 6)^0.5`
4. **验证 N/D 比例**：`D_opt / N_opt ≈ 20`

例如，如果你有 10^22 FLOPs 的预算：
- `N_opt ≈ (10^22 / 6)^0.5 ≈ 40B 参数`
- `D_opt ≈ 40B × 20 ≈ 800B token`

## 十、范式意义

### 这是一次“训练策略范式转变”

Chinchilla 没有发明新的架构、新的优化器、新的损失函数。它改变的是**如何分配计算资源的基本策略**。

**范式转换的层级**：训练策略范式转变

**为什么是范式转变？**

在 Chinchilla 之前，社区默认的假设是：模型越大越好，数据够用就行。这个假设指导了 GPT-3、Gopher、Megatron-Turing NLG 等模型的设计。

Chinchilla 推翻了这个假设，建立了一个新的默认假设：**在给定计算预算下，模型参数和数据量应该等比例增长。**

这个转变的影响是深远的：

1. **经济影响**：如果你用 Chinchilla 的建议训练模型，你可以用更少的计算资源达到相同的性能，或者用相同的计算资源达到更好的性能。

2. **研究影响**：社区开始关注“数据效率”而不是单纯的“模型大小”。这直接推动了数据质量管理、数据增强、课程学习等方向的发展。

3. **工程影响**：训练基础设施的设计也需要重新考虑。之前人们设计的是“能装下最大模型的集群”，现在需要的是“能处理最大数据集的集群”。

### 它属于什么类型的节点？

**桥接节点（Bridge Node）**：它连接了“scaling 理论”和“实际训练配置”，把理论结论转化为可操作的指导原则。

**修正节点（Correction Node）**：它修正了之前 Scaling Law 的错误结论。

**不是基础节点**：它没有提出新的理论框架，而是完善了已有的框架。

## 十一、局限性与后续方向

### 局限性

1. **实验规模有限**：最大的模型是 16B 参数，训练数据最多到 1.4T token。这个结论能否外推到 100B+ 参数的模型？论文没有验证。
