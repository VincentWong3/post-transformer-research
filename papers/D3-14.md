## 一、先搞清楚问题在哪里

多模态领域在 2019-2021 年间经历了一波爆发。从 ViLBERT、LXMERT、UNITER 到 CLIP、ALIGN，再到 SimVLM，模型一个接一个，benchmark 一个接一个被刷。但这三年下来，有一个问题越来越刺眼：**每个模型都只解决一部分问题，没有一个模型能同时搞定理解、生成、分类、定位这些事。**

具体来说，当时的现状是：

- **想做 VQA（视觉问答）**？用 UNITER 或者 LXMERT，加一个分类头。
- **想做 Image Captioning（图像描述）**？用 UNITER 不行，得换一种架构，或者给 UNITER 额外加一个解码器。
- **想做 Visual Grounding（视觉定位）**？需要检测头，UNITER 不支持，得换 Oscar 或者同期的 UNICORN。
- **想做 Text-to-Image Generation（文生图）**？以上全废，得用 VQGAN 之类的生成模型。

更让人头疼的是，**每个模型用不同的视觉特征提取器**。UNITER 用 Faster R-CNN 的 region features，CLIP 用 ViT 的 patch features，Oscar 用 object detection 的标签作为辅助输入。这意味着如果你想把一个模型的结果迁移到另一个模型上，你连输入格式都对不上。

这个问题有一个很形象的比喻：**多模态领域像是盖了一堆只有一层楼高的房子，每个房子只能做一件事。你想从 VQA 走到 Image Captioning，得先下楼、过马路、再上另一栋楼。** 整个领域缺的不是某一栋特别高的楼，而是一栋能容纳所有房间的统建楼。

OFA 就是在这样的背景下出现的。它的名字说得很直白——One For All，一个模型搞定一切。

## 二、旧方案的真正问题：为什么之前的模型做不到"统一"

要理解 OFA 提出的必要性，得先搞清楚之前的方案为什么统一不起来。不是大家不想统一，是**旧范式底层的设计决策天然阻断了统一的可能性**。

### 2.1 问题一：任务表示不统一（Task-Specific Output Heads）

这是最核心的问题。下面这个表列举了 2021 年前后四种有代表性的多模态模型——UNITER（encoder-only 融合路线）、CLIP（双塔对比学习）、SimVLM（prefix LM 统一路线）、VL-T5（早期 seq2seq 尝试）——在四个常见任务上的表现差异：

| 模型 | VQA | Captioning | Grounding | 分类 | 
|------|-----|------------|-----------|------|
| UNITER | 分类头 | 需加解码器 | 不支持 | 分类头 |
| CLIP | 不支持 | 不支持 | 不支持 | 对比学习分类 |
| SimVLM | 生成 | 生成 | 不支持 | 分类头 |
| VL-T5 | 生成 | 生成 | 特殊 token | 分类头 |

每个任务有自己独立的输出头。VQA 的答案分类器、Grounding 的坐标回归器、分类的线性分类头——这些输出头在架构上是不兼容的。你在 VQA 上学到的参数对 Captioning 毫无贡献，因为它们的输出空间完全不同。

更隐蔽的问题是：**当你有多个输出头时，预训练阶段的多任务学习需要人工调权重。** MLM（掩码语言建模）的 loss 和 MRM（掩码区域建模）的 loss 量级不同，ITM（图文匹配）的 loss 分布也不同。怎么平衡它们？大多数论文靠手调，少数论文（如 UNITER）用了一个简单的"等权重"，但这显然不是最优解。

### 2.2 问题二：视觉特征表示不一致

OFA 之前的模型几乎清一色用 **Faster R-CNN 提取 region features** 作为视觉输入。

这个选择的代价非常大：

第一，**Faster R-CNN 本身就是一个完整的检测模型**，需要在大规模检测数据集（如 Visual Genome）上单独预训练。这意味着多模态模型的视觉部分早就被"污染"了——它不是从 zero 开始学视觉表示的，而是用了现成的检测器输出。

第二，**region features 的语义粒度和文本 token 完全不同**。一个 region 可能包含"一个戴着红帽子的男孩"这样复杂的语义单元，而文本 token 是一个一个的子词。模型需要学习把不同粒度的东西对齐起来，这个映射本身就很难学。

第三，**region features 是固定数量的**（通常是 36 或 100 个 region），而文本 token 数量是可变的。这种不对称性让统一的序列建模变得很别扭——你得把 region 列表当作一个"包"而不是一个"序列"来处理。

### 2.3 问题三：预训练目标的碎片化

看看 UNITER 的预训练目标：

$$\mathcal{L} = \mathcal{L}_{\text{MLM}} + \mathcal{L}_{\text{MRM}} + \mathcal{L}_{\text{ITM}} + \mathcal{L}_{\text{WRA}}$$

四个目标，各有各的 loss 函数：
- MLM：交叉熵（分类）
- MRM：L2 回归或交叉熵（取决于版本）
- ITM：二元交叉熵（分类）
- WRA：对比学习 loss

**每个目标的数学形式不同，梯度分布不同，优化难度也不同。** 训练过程中，这些目标之间很可能存在竞争——有些任务"想"让模型往某个方向更新，另一些任务"想"往另一个方向。这不是理论推测，VLMO 的消融实验确实发现多目标训练时某些目标的性能会互相拖累。

### 2.4 问题四：微调时还要加新参数

这是最实际的痛点。假设你用 UNITER 预训练了一个多模态模型，现在需要做 Grounding。你得做以下事情：

1. 在 UNITER 编码器后面加一个坐标回归头（通常是一个 2 层 MLP）
2. 随机初始化这个回归头的参数
3. 在 Grounding 数据集上重新微调整个模型（包括 UNITER 的编码器）

这意味着：**每次加一个新任务，你都在改变模型结构。** 这导致了一个非常尴尬的局面——你的"预训练模型"实际上只在预训练阶段是"统一"的，到了微调阶段又变成了各种不同的样子。这削弱了预训练的统一性承诺。

OFA 要解决的就是这四个问题。它的方案听起来简单到近乎天真：**把所有任务都变成"序列到序列"的生成问题。** 但就是这个"把所有问题都规约成一个问题"的思路，恰好同时打击了四个痛点。

## 三、OFA 核心思路

**一句话概括：把跨模态和单模态的所有任务都统一成序列到序列（sequence-to-sequence）的生成问题，用一个 Transformer encoder-decoder 搞定一切，不需要任何任务特定的结构。**

怎么做到的？把每个任务拆成三块：

1. **输入**：图像和/或文本，统一编码成 token 序列
2. **指令**：一段手写的自然语言提示，告诉模型要做什么（"What does the image describe?"、"Does the image describe the text?"、"Which region does the text describe?"）
3. **输出**：任务结果，统一解码成 token 序列（文本、离散化的坐标、图像 code）

用例子来看更清楚：

- **Image Captioning**：输入 = 图像 token + 指令 "What does the image describe?"，输出 = "a dog playing with a ball"
- **VQA**：输入 = 图像 token + 问题，输出 = "yes"（或任何答案）
- **Visual Grounding**：输入 = 图像 token + 文本 + "Which region does the text describe?"，输出 = `<bin_200><bin_300><bin_400><bin_500>`（离散化的坐标序列）
- **ImageNet 分类**：输入 = 图像 token + "What is the image?"，输出 = "goldfish"
- **Text-to-Image**：输入 = 文本 + "Generate an image of ..."，输出 = 图像 code 序列

注意这几个例子中的关键设计：**连坐标都是离散化的 token，连分类都是生成类别名称。** 没有任何回归输出，没有任何分类头。输出空间统一成词表上的概率分布。

这意味着：
- 训练目标只有一个：**next token prediction（交叉熵）**
- 模型结构只有一套：**Transformer encoder-decoder**
- 不需要任何 task-specific head、adapter、soft prompt

## 四、整体结构与 forward 流程

### 4.1 输入输出表示（I/O）

> ⚠️ **本节导览**：OFA 使用五种不同类型的 token 表示——文本子词、图像 patch token（输入端）、图像 code token（输出端）、坐标 bin token（定位任务）、位置编码。这五种 token 共享同一个词表，是 OFA 模态无关设计的基础。读完本节后你会对它们各自的功能和格式有清晰的理解。

OFA 的输入由三个模态组成，全部编码成 token 序列。

**文本 token**：用 BPE（Byte Pair Encoding）编码，和 GPT/BART 一样。词表大小 65536（包含子词、图像 code token、位置 bin token）。

**图像 patch token**：
- 输入图像经过 ResNet 的前三个 block（ResNet-50 的 conv1 + bn + relu + maxpool + layer1 + layer2）下采样，输出 16× 下采样的特征图
- 预训练用 256×256，输出 16×16 = 256 个 patch token
- 每个 patch 对应 16×16 的原始像素区域
- 用 ResNet 而不是 ViT 的线性投影来做 patch 特征提取，因为 ResNet 的前三层能更好地保留局部结构信息

**图像 code token**（目标端）：
- 使用 VQVAE/VQGAN 的 codebook 把图像量化成离散 code 序列
- 256×256 的图像被编码为 16×16 = 256 个 code，每个 code 对应 codebook 中的一个索引
- 这些索引共享词表空间，是词表的一部分

**坐标 token**（Visual Grounding、Object Detection 输出端）：
- 连续的边界框坐标被离散化为整数 bin（0-999）
- 每个坐标值映射为 `<bin_0>` 到 `<bin_999>` 的 token
- 一个边界框 (x1, y1, x2, y2) 被表示为 4 个 token

**位置编码**：
- 文本使用 1D 绝对位置 embedding，图像使用 2D 绝对位置 embedding
- 位置 embedding 与 token/patch content embedding 解耦（各自独立加）
- 注意力计算中额外加 1D/2D 相对位置偏置（relative position bias）——帮助模型感知 token/patch 之间的空间关系

**统一词表**：所有 token 类型（子词、图像 code、坐标 bin、特殊标记）共享同一个词表。这意味着图像 code `<code_42>` 和文本 token "dog" 在 embedding 空间中处于同一层——它们可以被同一个 embedding 矩阵编码。

### 4.2 网络架构

OFA 使用的是一个**标准的 Transformer encoder-decoder 架构**，和 T5 / BART 同族。

架构参数（论文中明确提到的）：
- **OFA-Base**：约 93M 参数 [NEED VERIFICATION: 论文 Table 2 显示为 93M，待确认层数]，6 层编码器 + 6 层解码器
- **OFA-Large**：约 470M 参数，层数更多
- **OFA (最大)**：约 930M 参数

架构细节：
- 编码器：self-attention + FFN，双向注意力
- 解码器：self-attention + cross-attention + FFN，因果注意力（只能看当前位置之前的 token）。（cross-attention 的 query 来自解码器自注意力，key/value 来自编码器输出，是生成阶段从输入序列提取信息的核心通道——将在 6.2 节详细解释）
- 对标准 Transformer 做了几处调整：
  - **head scaling**：对 self-attention 的 head 维度做了缩放 [NEED VERIFICATION: 该细节需对照论文 Section 3.1 确认]
  - **post-attention LayerNorm**：attention 输出后接 LN [NEED VERIFICATION: 同上]
  - **第一个 FFN 层后也接 LN** [NEED VERIFICATION: 同上]
  - 使用两个独立的绝对位置 embedding（文本用 1D，图像用 2D）
  - 使用 1D 相对位置偏置（文本）、2D 相对位置偏置（图像）

这些调整并不激进——基本上是在标准 Transformer 上做了一些微调，用在了 T5/BART 已经验证过的架构上。

### 4.3 前向流程（以 VQA 为例）

**Step 1 — 输入编码**

输入图像（256×256 或 480×480）经过 ResNet 前三个 block：
- 256×256 → conv1 + maxpool → 64×64 → layer1 → 64×64 → layer2 → 32×32 → layer3 → 16×16
- 输出是 16×16 的 patch 特征图，每个 patch 特征维度由 ResNet 的 channel 数决定
- 每个 patch 特征通过一个线性层投影到模型的 hidden size

输入文本问题：
- "What color is the dog?" → BPE 分词 → [What, color, is, the, dog, ?]
- 每个 token 查 embedding 表得到向量

**Step 2 — 序列拼接**

编码器输入序列是扁平化的 token 序列：

```
[CLS] What color is the dog ? [IMG] patch1 patch2 ... patch256 [SEP]
```

加上位置编码（文本的 1D 位置 + 图像的 2D 位置）。

**Step 3 — 编码器前向**

序列经过 6 层（Base）Transformer 编码器。每个 token 通过 self-attention 看到所有其他 token（双向注意力）。

编码器输出的 hidden states 包含了图像和文本的联合语义表示。

**Step 4 — 解码器前向**

解码器输入是目标序列（shifted right）：

```
[BOS] black [EOS]
```

解码器做自回归生成：
1. 第一步：看到 `[BOS]`，通过 cross-attention 从编码器获取信息，预测第一个 token
2. 第二步：看到 `[BOS] black`，预测下一个 token
3. 直到生成 `[EOS]` 停止

每一步都使用 **causal attention mask**（只能看历史 token）和 **cross-attention**（看编码器输出）。

**Step 5 — 输出解码**

生成的 token 序列 "black [EOS]" 直接作为 VQA 的答案。

### 4.4 预定义的 8 个任务

OFA 在预训练阶段使用了 8 个子任务，分三组：

**跨模态任务（5 个）：**

| 任务 | 输入 | 输出 | 指令示例 |
|------|------|------|---------|
| Visual Grounding | 图像 + 文本 | 坐标 bin 序列 | "Which region does the text describe?" |
| Grounded Captioning | 图像 + 区域坐标 | 文本描述 | (输入包含坐标 token 的图像) |
| Image-Text Matching | 图像 + 文本 | "Yes" 或 "No" | "Does the image describe the text?" |
| Image Captioning | 图像 | 文本描述 | "What does the image describe?" |
| VQA | 图像 + 问题 | 答案 | (问题本身作为指令) |

**单模态视觉任务（2 个）：**
- Image Infilling：mask 掉图像中间部分，生成被 mask 区域的稀疏 code 序列
- Object Detection：给定图像，生成物体位置和类别标签

**单模态语言任务（1 个）：**
- Text Infilling：和 BART 一样的去噪语言建模——随机 mask 15% 的文本 token，让模型预测

这 8 个任务覆盖了理解（ITM、VQA）、生成（Captioning、Text Infilling）、定位（Grounding、Detection）、重建（Image Infilling）四种能力类型。**任务多样性（Task Comprehensiveness）** 是 OFA 设计时明确强调的——只有足够多样的预训练任务，模型才能学到跨任务泛化的能力。

## 五、核心创新点

OFA 的创新点不像 Swin 那种"设计了一个新模块"的类型，它的创新在**范式层面**：用一套规则统合所有东西。具体来说有四个核心设计。

### 5.1 创新一：用 seq2seq 统合所有任务

这是 OFA 最本质的贡献。

在 OFA 之前，有没有人试过用 seq2seq 做多模态？有。VL-T5（Cho et al., 2021）就是一个例子。但 VL-T5 只是把部分任务转成了 seq2seq 格式，分类任务仍然用分类头，回归任务仍然用回归头。[NEED VERIFICATION: VL-T5 的具体输出设计细节待对照原文确认]

OFA 把这条路走到底了：**连分类和回归都用生成来做。**

这意味着：
- **分类 = 生成**：输入图像，生成 "goldfish" 这个 token 序列 → 没有分类头，没有对比学习，只是普通的 next token prediction
- **回归 = 生成**：输入图像和文本，生成 `<bin_200> <bin_300> <bin_400> <bin_500>` → 没有坐标回归 MLP，只是普通的 next token prediction

这个设计的巧妙之处在于：当所有任务都用同一套接口时，**任务之间的知识迁移变得极其自然**。Captioning 中学到的"如何描述视觉内容"，可以直接用于 VQA；Grounding 中学到的"文本中的名词对应图像中的什么位置"，可以直接用于 Object Detection。因为这些能力都存储在同一个 Transformer 的参数里，而输出都是同一个词表上的 token 分布。

### 5.2 创新二：统一的预训练目标

在 OFA 之前，多模态预训练模型通常使用 3-4 个不同的预训练目标（MLM + MRM + ITM + ...），每个目标有自己的 loss 函数和权重。

OFA 只用了一个目标：**交叉熵（cross-entropy loss）**，作用于所有预训练任务。

公式上：

$$\mathcal{L} = -\mathbb{E}_{(x,y) \sim \mathcal{D}} \left[ \sum_{t=1}^{|y|} \log p_\theta(y_t | y_{\lt t}, x) \right]$$

$y_t$ 是目标序列的第 $t$ 个 token，$y_{\lt t}$ 是前 $t-1$ 个 token，$x$ 是输入序列（可能包含图像 token 和/或文本 token）。

不同的任务只是"损坏"方式不同：
- Text Infilling：mask 掉 15% 的文本 token
- Image Infilling：mask 掉图像中心的 code
- Visual Grounding：mask 掉坐标 bin
- ITM：mask 掉"Yes"/"No"

但底层 loss 完全一样。这意味着**不需要人工平衡不同目标之间的权重**，不需要担心任务冲突。模型只需要学会一件事：根据输入序列预测下一个 token。

### 5.3 创新三：模态无关的表示空间

OFA 把文本 token、图像 patch token、图像 code token、坐标 bin token 全部放进**同一个词表**。

这意味着一个 embedding 矩阵同时 encode 了三种模态的信息。图像 patch token 和文本 token 在 embedding 空间中处于同一层——它们之间的点积有直接的语义意义。

为什么这很重要？看一个具体的例子。

假设模型在做 Text-to-Image Generation：输入是 "a red apple"，输出是一个图像 code 序列。模型需要做的事情是：

1. 理解文本 token "red" 和 "apple" 的语义
2. 在解码时生成图像 code tokens

为什么同一个空间能让它们的 embedding 更接近？因为词表中的所有 token（包括文本子词和图像 code）在训练时都被同一个交叉熵目标梯度更新，模型会自然地把语义相关的输入和输出映射到相近的embedding区域。这和 word2vec 中'国王-女王≈男人-女人'的类比推理本质上是同一个原理——embedding 空间中的距离编码了语义关系。

如果文本和图像 code 在不同的 embedding 空间，模型需要额外学习一个"从文本空间映射到图像 code 空间的转换"。但如果它们在同一个空间，那么 "apple" 的 embedding 和 `<code_42>`（某种红色苹果形状的 code）的 embedding 天然更接近——模型只需要利用这个空间中的距离关系。

实际上这等价于形成了一个**跨模态的语义空间**，所有模态的信息都编码在同一个 embedding 矩阵中。这是 OFA 能够实现跨模态迁移的重要原因。

### 5.4 创新四：指令学习（Instruction-based Learning）

OFA 的另一个重要设计是：**在预训练和微调阶段都使用指令（instruction）**。

每个任务有一个手写的自然语言指令：
- Image Captioning 用 "What does the image describe?"
- Visual Grounding 用 "Which region does the text describe?"
- ITM 用 "Does the image describe the text?"

指令的作用是告诉模型"现在要做什么任务"。在预训练阶段，模型在 8 个任务上训练，每个任务有不同的指令。在微调阶段，OFA 通过更换指令来适配新任务——不需要改模型结构。

这比传统方法（通过任务 ID 或 token type 来区分任务）更灵活。原因有二：
1. 指令是自然语言，人类可以创造新指令来定义新任务
2. 指令本身也参与训练——模型学会了理解指令的语义，而不仅仅是简单匹配

这为零样本迁移奠定了基础：**如果给一个未见过的任务写一个指令，模型有可能在没有见过这个任务的情况下完成任务。**

## 六、核心公式

### 6.1 统一的预训练目标

OFA 的预训练目标是最标准的 seq2seq 负对数似然：

$$\mathcal{L}(\theta) = -\mathbb{E}_{(x,y) \sim \mathcal{D}} \left[ \log p_\theta(y | x) \right]$$

其中 $x$ 是输入序列（文本 + 图像 patch token），$y$ 是目标序列（文本 / 坐标 bin / 图像 code）。

展开自回归分解：

$$\log p_\theta(y | x) = \sum_{t=1}^{|y|} \log p_\theta(y_t | y_{\lt t}, x)$$

这里 $y_{\lt t}$ 是前 $t-1$ 个已生成的 token。解码器在生成每个 token 时，都能通过 cross-attention 看到编码器输出的全部信息。

这个公式和 GPT 的 autoregressive LM、T5 的 denoising objective 本质上是同一个东西。OFA 的创新不在于公式本身，而在于**把所有任务都规约成了这个公式可以处理的形式**。

### 6.2 跨模态注意力

解码器的 cross-attention 是把多模态信息从编码器传到解码器的通道：

$$\text{CrossAttn}(Q, K, V) = \text{softmax}\left(\frac{Q K^T}{\sqrt{d}}\right) V$$

- $Q \in \mathbb{R}^{L_d \times d}$：解码器当前位置的 query（自注意力后的结果）
- $K, V \in \mathbb{R}^{L_e \times d}$：编码器输出的 key 和 value
- $L_d$：解码器序列长度
- $L_e$：编码器序列长度

这个计算中，$Q K^T$ 的每个元素 $Q_i K_j^T$ 度量了解码器位置 $i$ 对编码器位置 $j$ 的注意力强度。

**这个公式解决了什么问题？** 假设解码器正在生成 "dog" 这个词的最后一个 token "g"，此时 cross-attention 计算的结果应该让模型重点关注图像中"狗"所在的 patch 区域。如果模型训练得好，图像中对应狗区域的 patch token 在 $\text{softmax}$ 归一化后会有较高的权重。

关键在 $K$ 和 $V$ 的组成：它们同时包含文本 token 和图像 patch token。所以 cross-attention 不仅让解码器"看"图像，还能"看"输入序列中的文本——这是 OFA 能处理图文混合输入的原因。

### 6.3 坐标离散化公式

在 Visual Grounding 和 Object Detection 中，连续的边界框坐标需要被映射为离散的 token：

$$\text{bin}(v) = \lfloor v \cdot (B-1) \rfloor$$

其中 $v \in [0, 1]$ 是归一化的坐标值，$B$ 是 bin 的数量。

对于一组边界框坐标 $(x_1, y_1, x_2, y_2)$：

$$
\begin{aligned}
\text{out}_{x1} &= \lfloor x_1 \cdot (B-1) \rfloor \\
\text{out}_{y1} &= \lfloor y_1 \cdot (B-1) \rfloor \\
\text{out}_{x2} &= \lfloor x_2 \cdot (B-1) \rfloor \\
\text{out}_{y2} &= \lfloor y_2 \cdot (B-1) \rfloor \\
\end{aligned}
$$

OFA 使用 $B=1000$，因此量化误差最大为 $1/999 \approx 0.001$（归一化空间），对应 480×480 图像上约 0.5 像素。对于大多数视觉任务来说，这个精度足够了。

**有趣的点**：这个离散化看起来很简单，但它带来的好处是**把回归问题转化成了分类问题**。模型不需要预测连续坐标值（需要 MLP 回归头），而是在 1000 个 bin 上做分类——这正好可以用 cross-entropy loss 来优化。

### 6.4 图像编码公式

图像编码过程：

$$I_{patch} = \text{ResNet\_blocks}(I)$$

$I \in \mathbb{R}^{H \times W \times 3}$ 是输入图像，ResNet 的前三个 block 输出 $I_{patch} \in \mathbb{R}^{\frac{H}{16} \times \frac{W}{16} \times C_r}$，其中 $C_r$ 是 ResNet 的 channel 数。

然后投影到模型的 hidden size：

$$E_{img} = I_{patch} W_{proj} + b_{proj}$$

其中 $W_{proj} \in \mathbb{R}^{C_r \times d}$，$E_{img} \in \mathbb{R}^{\frac{H}{16} \times \frac{W}{16} \times d}$。

展平后作为 token 序列的一部分输入编码器：

$$\text{input} = [E_{txt}; \, E_{img}]$$

### 8.2 模型结构关键代码

`models/ofa_model.py` 中的 `OFAModel` 核心结构：

```python
class OFAModel(FairseqEncoderDecoderModel):
    def __init__(self, args, encoder, decoder):
        super().__init__(encoder, decoder)
        self.args = args
```

编码器和解码器是标准的 Transformer 结构，继承自 fairseq 的 `TransformerEncoder` 和 `TransformerDecoder`。

但 OFA 的编码器有一个关键修改：**支持多模态输入**。具体来说，在 `TransformerEncoder.forward()` 中，输入序列是扁平化的 token 序列，但位置编码被分成了文本 1D 和图像 2D 两部分：

```python
# 简化后的伪代码
def forward_embedding(self, src_tokens, ...):
    # 文本 token embedding
    x = self.embed_tokens(src_tokens)
    
    # + 绝对位置编码（文本 1D）
    if self.text_position_embed:
        x = x + self.text_position_embed[src_tokens]
    
    # + 绝对位置编码（图像 2D - 对图像 patch 位置使用独立的编码器）
    if has_image_patches:
        x = x + image_position_embed
    
    return x
```

### 8.3 任务定义的关键设计

`tasks/ofa_task.py` 中定义了 OFA 的训练流程。核心是 `train_step` 函数：

```python
def train_step(self, sample, model, criterion, optimizer, ...):
    # 构造输入序列
    net_input = model.get_net_input(sample)
    
    # 前向传播
    output = model(**net_input)
    
    # 计算 loss (cross-entropy on target tokens)
    loss = criterion(output, sample['target'])
    
    return loss
```

`get_net_input()` 负责将不同任务的输入格式统一为模型能够处理的 `src_tokens`。具体来说，它把图像 patch token、文本 token、特殊 token 拼接成一个扁平的序列：

```python
def get_net_input(self, sample):
    # 文本 token 序列
    text_tokens = sample['text_tokens']
    
    # 图像 patch token  
    image_patches = sample['image_patches']
    
    # 拼接: [CLS] text_tokens [IMG] image_patches [SEP]
    src_tokens = self.concat_with_special(text_tokens, image_patches)
    
    return {'src_tokens': src_tokens, 'src_lengths': ...}
```

### 8.4 坐标离散化实现

`utils/` 中实现了坐标到 bin 的转换：

```python
def coord2bin(coord, num_bins=1000):
    """
    coord: 归一化的 [0, 1] float
    return: bin index [0, num_bins-1]
    """
    return int(coord * (num_bins - 1))
```

反变换：

```python
def bin2coord(bin_idx, num_bins=1000):
    """
    bin_idx: bin index [0, num_bins-1]
    return: 归一化的 [0, 1] float (bin 中心)
    """
    return (bin_idx + 0.5) / num_bins
```

## 九、技术谱系位置

### 9.1 技术树中的位置

```text
[Transformer, Vaswani et al., 2017]
   │
   ├── [BERT, 2018] ← encoder-only, 理解优先
   │      │
   │      ├── [UNITER, 2019] ← 多模态 encoder, region features
   │      ├── [VL-BERT, 2019] ← 同上
   │      └── [Oscar, 2020] ← + object tags 辅助
   │
   ├── [GPT, 2018] ← decoder-only, 生成优先
   │
   ├── [T5, 2019] ← encoder-decoder, seq2seq → ★ 核心前驱 ★
   │      │
   │      ├── [VL-T5, 2021] ← T5 做多模态, 部分统一
   │      │      │
   │      │      └── [OFA, 2022] ← ★ 本篇 ★ 全部统一
   │      │             │
   │      │             ├── [Flamingo, 2022] ← 冻结 LLM + adapter
   │      │             ├── [BLIP-2, 2023] ← Q-Former
   │      │             ├── [Kosmos-1, 2023] ← 统一多模态
   │      │             ├── [PaLI, 2022] ← Google 大规模版本
   │      │             └── [LLaVA, 2023] ← 视觉 instruction tuning
   │      │
   │      └── [BART, 2019] ← denoising autoencoder
   │
   └── [CLIP, 2021] ← 对比学习, 双塔
```

OFA 处于 **"统一 multimodal seq2seq 范式"** 这条路线上。它的直接前驱是 VL-T5（2021），同期竞品是 SimVLM（2022）和 BEiT-3（2022），后续影响覆盖了 Flamingo、BLIP-2、PaLI、LLaVA 等一系列工作。

### 9.2 前驱工作

**T5（Raffel et al., 2019）**：OFA 最重要的灵感来源。T5 证明了"把所有 NLP 任务统一成 text-to-text 格式"在 NLP 上的可行性。OFA 的自然语言到 ... 扩展："能不能把多模态任务也统一进来？" T5 的 denoising objective 直接成为 OFA 的预训练目标。

**BART（Lewis et al., 2019）**：OFA 的架构原型和初始化来源。OFA 使用了 BART 的 Transformer encoder-decoder 架构和 denoising 类型的预训练。

**VL-T5（Cho et al., 2021）**：最直接的先驱。VL-T5 展示了用 T5 做多模态任务的可行性，但统一程度有限——仍然保留了部分任务特定的输出格式。OFA 在 VL-T5 的基础上推到了极致。

**ViT（Dosovitskiy et al., 2021）**：OFA 没有用 ViT 的线性 patch projection，而是用 ResNet 提取 patch 特征，但两者共享同一个核心思想——用 patch 替代 region，避开 object detection 的依赖。

### 9.3 同期竞品

**SimVLM（Wang et al., 2022）**：和 OFA 几乎是同时期的工作（arXiv 同在 2022 年 2 月）。SimVLM 也用 prefix language modeling 做统一预训练，但它用的是单峰 prefix LM（没有 encoder-decoder 的 cross-attention），而且不支持 Visual Grounding 和 Object Detection 这类定位任务。OFA 的 encoder-decoder 架构在这方面更灵活。

**BEiT-3（Wang et al., 2022）**：微软的工作，使用 Multiway Transformer（同类型 Transformer encoder）做统一预训练。它也支持多模态，但架构是 encoder-only，生成任务需要额外适配。OFA 的 encoder-decoder 架构对生成任务天然友好。

**Flamingo（Alayrac et al., 2022）**：DeepMind 的工作，走的是"冻结视觉 encoder + 冻结语言模型，用 cross-attention 连接"的路线。Flamingo 的优势是可以利用超大规模语言模型（70B），但架构不是端到端的，不够统一。OFA 的优势是简洁——一个模型，一次训练。

### 9.4 后续影响

**Kosmos-1（Huang et al., 2023）**：微软的后续工作，直接延续了 OFA 的"统一 seq2seq"思路，但用了 decoder-only 架构（基于 LLM）。在 OFA 的基础上加上了视觉 grounding 和语言理解的统一。Kosmos-1 在架构上可以看作"OFA 的思想 + 大语言模型的路线"。

**PaLI（Chen et al., 2022）**：Google 的响应，也是 encoder-decoder 架构，但规模更大（17B 参数）。PaLI 验证了 OFA 的路线可以在更大的尺度上工作。

**BLIP-2（Li et al., 2023）** / **LLaVA（Liu et al., 2023）**：这两条线走的是"预训练视觉 encoder + 大语言模型 + 轻量连接"的路线，而不是 OFA 的"端到端统一"。它们选择了更大的语言模型，但放弃了统一训练的简洁性。这其实是两条路线的分化：
- OFA 路线：统一训练 → 更强的小模型泛化
- LLM 连接路线：大模型 → 更强的生成能力

**后续工作对 OFA 的评价**：OFA 被认为是第一个系统性地验证"完全统一 seq2seq 多模态"可行性的工作。它的价值不在于把 benchmark 刷到最高，而在于证明了一条路——**用最简单的 cross-entropy loss 和最标准的 encoder-decoder 架构，就能做到之前需要复杂多任务设计才能做到的事情。**

## 十、局限性

### 10.1 图像输入尺度固定且单一

OFA 使用固定的下采样比例（16×），预训练用 256×256（16×16 个 patch），微调用 480×480（30×30 个 patch）。

问题是：对于包含微小物体的图像（比如一张多人照片中的人物），16×16 的 patch 粒度可能不够。一个区域对应 16×16=256 个像素，对于 256×256 的输入，整个图像只有 256 个 patch。如果图像中有多个小物体，每个物体可能只覆盖 1-2 个 patch——信息量不够。

后续工作（如 BEiT-3、PaLI）使用了多尺度训练或更大的输入尺寸来缓解这个问题。

### 10.2 生成式分类的效率问题

OFA 把 ImageNet 分类做成了生成类别名称——输入图像，生成 "goldfish" 这样的 token 序列。这虽然统一了任务接口，但效率很成问题：

- 分类头只需要一次前向传播 + 一次 argmax（O(1)）
- 生成式分类需要自回归解码多个 token（O(L)，L 是类别名称的 token 长度，最多 10 个左右）

在 ImageNet 1000 类分类上，分类头一次推理和 beam search 5 步解码的时间差距大约是 5-10 倍。虽然论文认为"统一的好处大于效率损失"，但在需要低延迟的应用场景（如实时分类）中，这个差距可能是不可接受的。

### 10.3 坐标离散化的精度硬限制

使用 1000 个 bin（$B=1000$）的假设是把图像归一化到了 [0, 1] 区间。对于 480×480 的微调输入，每个 bin 覆盖的像素范围约 0.48 像素——看起来够用。

但问题出在边界上。假设一个物体的边界框正好落在两个 bin 的边界上（比如坐标 0.5005，bin 500 是 [0.500, 0.501)），量化误差约为 0.5 个 bin。对于 480 像素，0.0005 的归一化误差对应约 0.24 像素的偏移。对于大多数任务这不是问题，但对于需要亚像素精度的任务（比如医学图像分割），这个误差是硬限——它不随模型规模增加而缩小，是离散化本身的固有限制。

### 10.4 预训练数据的争议点

OFA 的预训练数据只有 **2000 万图文对**，远少于 CLIP（4 亿）和 SimVLM（18 亿）。论文把"小数据也能出 SOTA"当作一个卖点。

但这个卖点有前提条件：OFA 用了 **BART 的预训练权重初始化**。也就是说，OFA 的语言能力不是从零开始的——BART 已经在海量文本上预训练过了。OFA 的多模态预训练相当于在 BART 的基础上"增加视觉能力"。

如果严格从零开始训练（没有 BART 初始化），2000 万图文对是否够用？论文没有回答这个问题。

### 10.5 语言模型的规模天花板

OFA 的最大版本（~470M 参数）和同期的大语言模型（GPT-3 175B、PaLM 540B）相比，在语言能力上差了两个数量级。

这是因为 OFA 走的是 **encoder-decoder 端到端训练**路线——视觉和语言能力在一个模型里同时学习，没有利用已有的超大语言模型。这个路线在 2022 年还行得通，但 2023 年之后，社区普遍转向了"冻结 LLM + 视觉 adapter"路线（Flamingo、BLIP-2、LLaVA），因为直接利用 GPT-4、LLaMA 等模型的生成能力远强于端到端训练的 470M 模型。

### 10.6 不支持多语言

OFA 只支持英文。BPE 词表基于英文子词，图像 caption 和 VQA 数据也全是英文。对于多语言应用（如中文看图说话），OFA 需要额外的多语言扩展。

## 十一、小结

OFA 是一篇**"思路极其简单，效果出人意料好"**的论文。

它的核心贡献可以用三句话概括：

1. **一个架构**：标准 Transformer encoder-decoder，没有任何 task- 或 modality-specific 的组件
2. **一个目标**：next token prediction（cross-entropy），所有任务共用
3. **一个接口**：输入 = token 序列，输出 = token 序列，分类和回归都被离散化成生成

OFA 的价值不在于某个 benchmark 上刷了多少点——虽然它确实在多个任务上达到了 SOTA——而在于**它系统地验证了一条路线的可行性**：把多模态任务全部统一成 seq2seq 生成，这条路是可以走通的。

在 OFA 之前，多模态领域默认的范式是：不同任务用不同架构、不同目标、不同输出头。OFA 证明了这个"默认"不一定是正确的。这给了后续工作一个清晰的信号：你不需要为每个任务设计专门的东西，一个简单的框架就够了。

当然，OFA 也不是没有代价的。**统一带来了简洁性，但牺牲了某些方面的效率和上限。** 生成式分类慢于分类头，坐标离散化回归精度有限，端到端训练限制了语言模型的规模。这些问题在 OFA 中没有得到完美解决，而是由后续工作（PaLI、BLIP-2、LLaVA）在不同的方向上进行改进。

但回过头来看，OFA 在 2022 年出现的时间点有特殊意义：**它标志着多模态领域从"碎片化"进入了"统一化"的探索阶段**。在它之后，多模态模型的设计范式从"task-specific"转向了"unified"，这个转变的影响一直延续到今天。

这篇论文在技术树 D3 分支中的定位是**"奠基性工作"**。它不是这个分支的起点（T5 和 VL-T5 在前面），但它把这个分支推到了一个新的高度——证明了"全部统一"不仅是可行的，而且是有优势的。后续的 Kosmos-1、PaLI、甚至 LLaVA，都可以看作是 OFA 这条大路的不同变体或改进。
