# E1-3：Longformer — 滑动窗口 + 全局注意力，线性复杂度的长文档 Transformer

> **论文：** Longformer: The Long-Document Transformer  
> **作者：** Iz Beltagy, Matthew E. Peters, Arman Cohan — Allen Institute for AI  
> **发表：** arXiv:2004.05150, April 2020 (v2 December 2020)  
> **仓库分类：** E1. 如何降低 attention 复杂度？ → 范式 E1-1：高效 attention 范式

---

## 一、先搞清楚"问题在哪里"

Transformer 在自然语言处理领域取得了统治性地位，但它的自注意力机制有一个硬伤：计算量和内存占用随序列长度 **二次方增长**（O(n²)）。这意味着 BERT 类模型最多只能处理 512 个 token 的序列，超出这个长度就得截断或分块处理。

这个限制对于长文档任务尤其痛苦。问答系统要把问题和多个文档拼在一起，长度很容易超过几千 token；共指消解决定需要全局视野来分辨代词指代的对象；长文本分类需要理解整篇文章而非片段。现有的解决方案要么直接截断（信息丢失），要么分块处理后再用复杂架构拼接（工程复杂、仍有信息损失）。

Longformer 的核心贡献是提出了一种 **线性复杂度** 的自注意力机制，可以在 GPU 上直接处理数千乃至上万 token 的文档，而且是 **即插即用式**（drop-in replacement）的替代方案——也就是说，它可以直接替换 BERT/RoBERTa 的自注意力层，无需改变整体架构。

更关键的是，Longformer 不仅做了自回归语言模型实验（这是当时大多数长文档 Transformer 的标配），还真正做了 **预训练→微调**（pretrain-finetune）的完整 pipeline：从 RoBERTa 继续预训练，然后在多种下游任务上微调，并取得了当时多个数据集上的 SOTA。

此外，Longformer 还有一个 Encoder-Decoder 变体 **LED**，专门用于长文档摘要等 seq2seq 任务。

## 二、旧路线的真正问题

### 标准自注意力的复杂度

给定一个长度为 n 的序列，Transformer 的自注意力机制计算每个 token 对所有其他 token 的注意力分数：

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

其中 Q、K、V 的维度都是 n × d。矩阵乘法 QK^T 的复杂度是 O(n² × d)。对于长度为 512 的序列这没问题，但当 n 增长到 4096、8192 甚至 32768，O(n²) 的代价就变得不可接受。

### 实际影响

具体来说，BERT-base（12 层，768 hidden）处理 512 token 需要约 1.2 GB GPU 显存；处理 4096 token 时，标准自注意力需要约 **75 GB**——远超任何单 GPU 的显存上限。

现有的折中方案：

1. **直接截断**：只保留前 512 token，丢掉长尾信息。最粗暴但最常用。
2. **分块处理**：把文档切成 512 token 的块，各自过 BERT 后拼接激活值，再用任务相关模块组合。这是当时共指消解和 QA 的主流做法，但跨块信息交互只能靠后续模块解决，效果受限。
3. **检索-阅读两阶段**：先检索相关段落再阅读理解。级联误差是绕不开的问题。

Longformer 的目标就是让 Transformer **直接处理长序列**，避免这些折中。

## 三、这篇论文的核心思路

Longformer 的核心思想是 **稀疏化注意力矩阵**：不计算完整的 n×n 注意力矩阵，而是根据预设的"注意力模式"只计算部分位置的注意力分数。

整体注意力模式由两部分组成：

1. **滑动窗口注意力**（Sliding Window Attention）—— 每个 token 只关注邻近的 w 个 token，复杂度 O(n×w)。
2. **全局注意力**（Global Attention）—— 少量预设的"全局 token"关注整个序列，也让整个序列关注它们，复杂度 O(n)。

两者相加，整体复杂度仍然是 O(n)（w 是常数，全局 token 数量也是常数）。

这种设计的背后直觉是：局部上下文对理解文本至关重要（这在 BERT 注意力分析论文中有充分证据），而全局信息可以通过少量关键位置（如 [CLS] token 或问题 token）来传递。多层叠加的窗口注意力也能通过逐层传播逐级扩大感受野，类似 CNN。

## 四、整体结构与 forward 流程

### 基本形式

给定窗口大小 w，每个 token 只关注自己左右各 w/2 个 token。复杂度 O(n×w)。当 w 是常数（如 512），这就是线性复杂度。

在一个有 ℓ 层的 Transformer 中，顶层 token 的感受野大小为 ℓ × w——因为每层窗口注意力都在向外传播信息。这意味着即使每层只看到很小的局部，多层的累积效果也能覆盖整个序列。

### 窗口大小配置

Longformer 在自回归语言模型实验中采用了 **逐层递增** 的窗口大小：

- 低层使用小窗口（如 32），聚焦局部特征
- 高层使用大窗口（如 8192），捕获全局信息

消融实验（Tab. 4）清楚展示了这种配置的优势：

| 配置 | Dev BPC |
|---|---|
| Decreasing w（从 512 递减到 32） | 1.24 |
| Fixed w (= 230) | 1.23 |
| **Increasing w（从 32 递增到 512）** | **1.21** |

递增 > 固定 > 递减，这个次序很合理：低层需要捕捉紧密的局部上下文来建立基础表示，高层则需要更广的视野来整合全局信息。

### 在预训练/微调中的窗口设置

在从 RoBERTa 继续预训练时，Longformer 将窗口大小直接设为 512——和 RoBERTa 的 512 序列长度一致。这样保持了相同的计算量，只是把原来 512 token 的完整注意力换成了 512 窗口的滑动注意力。


## 五、核心模块逐个解释：空洞滑动窗口

### 动机

滑动窗口注意力的感受野大小是 ℓ × w。为了在不增加窗口大小（即不增加计算量）的情况下扩大感受野，Longformer 引入了 **空洞滑动窗口**——在窗口中留下空洞，只关注间隔为 dilation d 的位置。

这直接借鉴了 dilated CNN（如 WaveNet）的思路。一个带空洞的窗口虽然宽度还是 w，但覆盖范围变成了 d × w。多层叠加后，感受野达到 ℓ × d × w——即使 w 和 d 都很小，也能覆盖上万 token。

### 多头配置

Longformer 在多头注意力中使用了 **不同 head 不同 dilation** 的配置：

- 部分 head **无空洞**（dilation=0），聚焦紧密的局部上下文
- 部分 head **带空洞**，捕获远程依赖

具体配置（small model, 12 层，8 heads）：

- 层 0-5：dilation = 0（所有 head 都是纯窗口）
- 层 6-7：dilation = 1
- 层 8-9：dilation = 2
- 层 10-11：dilation = 3

只有 **2 个 head** 使用空洞，其余 head 保持无空洞的窗口注意力。

消融实验（Tab. 4）显示，在 150K 步时，加 2 个 head 的 dilation 将 BPC 从 1.21 降到 1.20，有改进但幅度不大。

### 实现限制

空洞滑动窗口的实现在工程上比普通滑动窗口更难。Longformer 提供了三种实现（详见第八节），其中只有 **CUDA kernel** 支持空洞模式。chunks 实现不支持空洞。

## 六、核心公式与数学直觉

### 为什么需要全局注意力

窗口注意力（包括带空洞的）本质上还是局部处理。对于很多 NLP 任务，仅靠局部上下文是不够的：

- **分类任务**：需要聚合整个序列的信息到 [CLS] token
- **问答任务**：问题 token 需要看到文档中的所有相关信息
- **MLM 预训练**：预测 masked token 时需要跨区域的信息

### 设计

Longformer 的全局注意力是 **对称** 的：一个被标记为"全局"的 token 可以看到整个序列中的所有 token，同时序列中的每个 token 也都能看到这个全局 token。

全局 token 的数量是 **少数且固定** 的，和序列长度 n 无关，所以整体复杂度仍然是 O(n)。

### 任务特定的配置

不同任务使用不同的全局注意力配置：

| 任务 | 全局 token |
|---|---|
| 文本分类 | [CLS] token |
| MLM 预训练 | 所有待预测的 masked token |
| WikiHop QA | 所有问题 token + 所有答案候选 token |
| TriviaQA | 所有问题 token |
| HotpotQA | 问题 token + 段落标题 token + 支持事实相关的句子 token |
| LED 摘要 | 第一个 \<s\> token |

这种任务特定的设计是一种 **归纳偏置**（inductive bias）——告诉模型哪些位置值得关注全局信息。论文强调，这比现有的复杂任务特定模型（如多阶段检索+阅读）要简单得多。

### 独立线性投影

Global attention 使用 **独立的 Q/K/V 投影矩阵**：

- Qs、Ks、Vs：用于滑动窗口注意力
- Qg、Kg、Vg：用于全局注意力

两套投影在初始化时共享相同的权重值（从 RoBERTa 权重拷贝），但在训练过程中可以不同。消融实验显示，没有独立投影会导致 WikiHop 性能下降 1.6 个百分点；如果同时去掉独立投影和全局注意力，性能更是暴跌 8.3 个百分点。

---

## 七、独立投影的工程细节与计算流程

从公式角度看，Longformer 的计算过程是：

1. 用 Qs、Ks 计算窗口注意力分数，得到局部上下文的注意力权重
2. 用 Qg、Kg 计算全局注意力分数，得到全局上下文的注意力权重
3. 两类注意力分数分别 softmax，各自加权求和 V 与 Vg
4. 将局部注意力输出和全局注意力输出合并

使用独立投影的关键工程细节：

- Qg、Kg、Vg 从 Qs、Ks、Vs **初始化**（权重相同）
- 训练时它们可以 **独立更新**，模型可以学会为全局注意力分配不同的特征空间
- 这种方式比"用一个投影矩阵同时做局部和全局注意力"更灵活

## 八、复杂度、效率与可扩展性分析

实现 Longformer 的带空洞滑动窗口注意力本质上需要一种 **带状矩阵乘法**（banded matrix multiplication）——矩阵乘法的输出只在某些对角线上非零。PyTorch 和 TensorFlow 都不直接支持这种运算。

论文提供了三种实现，各有优劣（参见论文 Fig. 1）：

### 方法一：Longformer-loop

- **思路**：循环计算每条对角线
- **优点**：内存高效（只计算非零值），支持空洞
- **缺点**：极其慢，不可用于实际训练
- **用途**：仅用于测试验证

### 方法二：Longformer-chunks

- **思路**：将 Q 和 K 切成大小为 w 的重叠块（重叠 w/2），用标准矩阵乘法计算块内注意力，然后 mask 掉不需要的对角线
- **优点**：利用 PyTorch 的标准 matmul，计算效率高（比 loop 快得多）
- **缺点**：不支持空洞模式；内存消耗是理论最优的 2 倍（因为计算了部分零值）
- **用途**：预训练和微调场景（内存占用增加不构成问题）

### 方法三：Longformer-cuda

- **思路**：用 TVM（Tensor Virtual Machine，陈天奇等 2018）编写自定义 CUDA kernel
- **优点**：完全功能的实现，内存最优，速度与高度优化的全注意力相当
- **缺点**：需要 TVM 编译流程，工程复杂度较高
- **用途**：自回归语言模型实验（需要最长序列 + 空洞支持）

### TVM 的角色

论文选择 TVM 而非手动编写 CUDA，是因为 TVM 允许用高级 Python 描述计算逻辑，然后自动生成并编译为优化的 CUDA 代码。这比维护一个用 C++ 写的、绑定特定 TensorFlow 版本的 BlockSparse（Sparse Transformer 使用的方案）要灵活和可维护得多。

## 九、实验结果

### 任务设定

Longformer 在字符级自回归语言模型上做了第一轮评估，使用 text8 和 enwik8 两个基准。这两个数据集都包含来自 Wikipedia 的约 1 亿字符，划分为 90M/5M/5M 的训练/验证/测试集。

任务形式是标准的自回归（左到右）语言模型，即预测下一个字符。

### 模型配置

实现基于 Transformer-XL 的代码库（禁用了记忆机制），使用 **相对位置编码**（正弦权重）。

两种模型大小：

- **小型**：12 层，8 头，512 隐藏层维度（与 Transformer-XL 一致）
- **大型**：30 层，8 头，512 隐藏层维度（与 Sparse Transformer 一致）

训练细节：混合精度训练（fp16 + fp32），注意力部分保持 fp32 以避免数值不稳定性；梯度检查点（gradient checkpointing）减少显存；48GB RTX8000 GPU。

### 评估方式

使用 32,256 字符的序列长度。按照 Dai et al.（2019）的方法，将数据集切分为步长为 512 的重叠序列，仅报告每个序列最后 512 个 token 的困惑度（BPC，bits per character）。

### 小型模型结果

| 模型（text8） | 参数量 | Test BPC |
|---|---|---|
| T12 | 44M | 1.18 |
| Adaptive Span | 38M | 1.11 |
| BP-Transformer | 39M | 1.11 |
| **Longformer** | **41M** | **1.10** |

| 模型（enwik8） | 参数量 | Test BPC |
|---|---|---|
| T12 | 44M | 1.11 |
| Transformer-XL | 41M | 1.06 |
| Reformer | — | 1.05 |
| Adaptive Span | 39M | 1.02 |
| BP-Transformer | 38M | 1.02 |
| **Longformer** | **41M** | **1.00** |

Longformer 小型模型同时在 text8 和 enwik8 上取得了 SOTA。

### 大型模型结果

| 模型 | 参数量 | enwik8 Test BPC |
|---|---|---|
| Transformer-XL (18 层) | 88M | 1.03 |
| Sparse Transformer | ≈100M | 0.99 |
| Transformer-XL (24 层) | 277M | 0.99 |
| Adaptive Span | 209M | 0.98 |
| Compressive Transformer | 277M | 0.97 |
| Routing Transformer | ≈223M | 0.99 |
| **Longformer** | **102M** | **0.99** |

Longformer 大型模型（102M 参数）超过了同规模的 Transformer-XL（88M），与 Sparse Transformer（≈100M）持平，略低于参数量两倍以上的 Adaptive Span 和 Compressive Transformer。但后两者使用的是 **自适应注意力跨度**，这种机制不适合 pretrain-finetune 范式——这是 Longformer 的一个重要区分点。

## 十、训练策略与消融实验

### 分阶段训练

Longformer 的训练策略很有意思：模型需要大量梯度更新来先学会局部上下文，然后才能有效利用长距离上下文。因此论文设计了 **分阶段训练**（staged training），共 5 个阶段：

- **阶段 1**：短序列（2048）、小窗口、高学习率
- **后续阶段**：窗口大小和序列长度逐渐翻倍，学习率逐步减半
- **阶段 5**：序列长度 23,040（接近 GPU 显存极限）

具体配置（小型模型）：

| 阶段 | 序列长度 | 窗口大小范围（从底层到顶层） | LR | Batch size | 步数 |
|---|---|---|---|---|---|
| 1 | 2,048 | 32 → 8,192 | 2.5e-4 | 32 | 430K |
| 2 | 4,096 | 32 → 8,192 | 1.25e-4 | 32 | 50K |
| 3 | 8,192 | 32 → 8,192 | 6.25e-5 | 16 | 50K |
| 4 | 16,384 | 32 → 8,192 | 3.125e-5 | 16 | 35K |
| 5 | 23,040 | 512 → top | 1.5625e-5 | 16 | 5K |

这样做的好处是：大部分训练时间花在短序列上（快、便宜），只有最后阶段才处理最长序列（慢但少量）。

### 消融实验

论文在 text8 上对注意力模式做了系统的消融实验（每个配置训练 150K 步，阶段 1 配置）。

**窗口大小配置**（Tab. 4 top）：

- **递增**（32→512）：BPC 1.21（最佳）
- **固定**（230 平均）：BPC 1.23
- **递减**（512→32）：BPC 1.24

**空洞配置**（Tab. 4 bottom）：

- **无空洞**：BPC 1.21
- **2 个 head 加空洞**：BPC 1.20（小幅改进）

## 十一、预训练与微调范式

### 为何要预训练？

当时的大多数长文档 Transformer（如 Transformer-XL、Adaptive Span、Compressive Transformer、Reformer）只做了自回归语言模型的评估。但这种模式不适合迁移学习场景——下游任务（分类、QA、共指消解）需要的是 **双向上下文** 而不是左到右。

Longformer 是较早一批真正走完整 pretrain-finetune 路线的长文档 Transformer 之一。论文从 RoBERTa 的发布检查点继续预训练，使用掩码语言模型（MLM）目标。

### 预训练数据

为了训练 Longformer 处理长文档的能力，论文自己编译了一个长文档语料：

| 数据源 | token 数量 | 平均文档长度 |
|---|---|---|
| Books（Zhu et al., 2015） | 0.5B | 95.9K |
| English Wikipedia | 2.1B | 506 |
| Realnews（Zellers et al., 2019） | 1.8B | 1.7K |
| Stories（Trinh and Le, 2018） | 2.1B | 7.8K |

其中 Books + Wikipedia 与 RoBERTa 的预训练数据有重叠，这样做是为了尽量保持分布一致性。Realnews 和 Stories 的加入提供了更多的长文档样本（只取了长度 > 1200 token 的部分）。

总 token 数约 6.5B。

### 预训练配置

使用 fairseq 框架，序列长度 4,096，batch size 64（= 2^18 tokens），学习率 3e-5，线性 warmup 500 步后 power-3 多项式衰减。其余超参数与 RoBERTa 一致。训练 65K 梯度更新。

### MLM 困惑度结果

| 配置 | base | large |
|---|---|---|
| RoBERTa（序列长度 512） | 1.846 | 1.496 |
| Longformer（序列长度 4096，随机初始位置编码） | 10.299 | 8.738 |
| Longformer（拷贝初始位置编码，未训练） | 1.957 | 1.597 |
| + 2K 梯度更新 | 1.753 | 1.414 |
| + **65K 梯度更新** | **1.705** | **1.358** |
| Longformer（只训练额外位置编码，冻结 RoBERTa） | 1.850 | 1.504 |

几个关键发现：

1. RoBERTa 的 1.846 BPC 说明构建的预训练语料分布与 RoBERTa 原始语料接近。
2. 从「随机初始化位置编码 → 拷贝初始化位置编码」，BPC 从 10.299→1.957（base），说明 **拷贝初始化极其重要**。
3. 从「拷贝初始化 → 2K 步 → 65K 步」持续收敛，说明模型在逐渐学会利用滑动窗口注意力和长距离上下文。
4. 只训练位置编码的方案（冻结 RoBERTa 权重）BPC 为 1.850，比全参数训练（1.705）高，但比完全不训练（1.957）好。这种配置的动机是完美保留 RoBERTa 在短文档上的表现，但实际效果不如全参数训练。

## 十二、位置编码的拷贝初始化

### 问题

RoBERTa 的绝对位置编码最大只支持到 512，对应它 512 token 的序列长度上限。Longformer 需要支持到 4,096（甚至 16K），所以需要扩展现有的位置编码矩阵。

### 方法

最简单的方案是随机初始化新的位置编码（从 512 扩展为 4096）。但论文发现这会导致灾难性的性能下降（BPC 从 1.846 飙升到 10.299，近乎随机猜测）。

**解决方案**：将 RoBERTa 原有的 512 个位置编码 **重复拷贝** 多次，填满到 4096 个位置。例如，位置 512-1023 拷贝位置 0-511 的编码，位置 1024-1535 再次拷贝，以此类推。

### 为什么有效

BERT 的注意力头分析表明，BERT 学到了一个很强的 **局部注意力偏差**——大部分注意力集中在当前词和相邻 token，尤其是前一个和后一个 token。拷贝位置编码保留了这种局部结构，只是在拷贝边界处（每 512 token 的边界）有轻微的不连续性。

这种简单方法效果惊人：拷贝初始化后 BPC 从 10.299 降到了 1.957，仅比原始的 RoBERTa（1.846）高了 0.11，说明滑动窗口注意力配合拷贝位置编码已经能很好地工作了。之后的预训练进一步降低了这个差距。

这个方法也用于 LED 的 BART 初始化中：BART 的 1024 个位置编码被重复拷贝 16 次，填满 16K。

## 十四、消融研究与分析（WikiHop）

论文在 WikiHop 开发集上做了详细的消融实验（Tab. 10），分解每个设计选择的贡献：

| 配置 | 准确率 | 差值（相对于 Longformer 5 epoch 基线） |
|---|---|---|
| **Longformer（序列长度 4096，15 epoch）** | **75.0** | — |
| RoBERTa-base（序列长度 512） | 72.4 | -1.4 |
| Longformer（序列长度 4096，5 epoch） | 73.8 | — |
| Longformer（序列长度 512，全注意力） | 71.7 | -2.1 |
| Longformer（序列长度 2048） | 73.1 | -0.7 |
| Longformer（无 MLM 预训练） | 73.2 | -0.6 |
| Longformer（无独立 Q/K/V 投影） | 72.2 | -1.6 |
| Longformer（无独立投影 + 无全局注意力） | 65.5 | -8.3 |
| Longformer（只训练位置编码） | 73.5 | -0.3 |

**关键结论**：

1. **更长的序列有帮助**：4096 > 2048 > 512（75.0 > 73.1 > 72.4）。但 2048 就已经比 512 好，说明增长序列长度在大多数情况下都能带来收益。
2. **全局注意力至关重要**：去掉全局注意力 + 独立投影会导致灾难性的性能下降（-8.3 分）。
3. **独立投影不可或缺**：只去掉独立投影（保留全局注意力）下降 1.6 分。说明两套投影各自优化局部和全局注意力是有效的。
4. **MLM 预训练带来提升**：从头训练（无预训练）下降 0.6 分。WikiHop 的数据量较大（约 40K 训练样本），所以即使没有预训练也能学得不错，但预训练仍然有帮助。
5. **长序列本身不是关键**：当序列长度设为 512 并使用全注意力时，Longformer 的表现反而比 RoBERTa-base 差（71.7 vs 72.4），说明性能提升不是来自额外的预训练步数，而是来自 **滑动窗口注意力 + 全局注意力 + 长序列** 的组合。
6. **更多训练 epoch 有帮助**：15 epoch（75.0）优于 5 epoch（73.8）。

## 十五、LED 与总结

### LED 的设计

LED 是 Longformer 的 Encoder-Decoder 变体，专门用于序列到序列（seq2seq）任务，如长文档摘要。

- **Encoder**：使用 Longformer 的高效注意力模式（滑动窗口 + 全局注意力）
- **Decoder**：使用标准的 **全注意力** 处理编码器的输出和已解码的位置

这样，Encoder 的复杂度是 O(n)，Decoder 的复杂度是 O(m²)（m 是输出序列长度，通常远小于输入长度 n）。

### 初始化策略

LED 从 BART 初始化（不是从头训练），因为预训练一个 LED 非常昂贵。具体来说：

- 架构与 BART 一致（LED-base: 6 层 encoder + 6 层 decoder；LED-large: 12 + 12 层）
- 位置编码从 BART 的 1,024 扩展到 16,384，采用与 Longformer 相同的 **拷贝初始化** 方法——BART 的 1024 个位置编码重复拷贝 16 次
- Encoder 窗口大小设置为 1,024
- 全局注意力放在第一个 \<s\> token

### arXiv 摘要结果

论文在 arXiv 长文档摘要数据集上评估 LED。这个数据集的文档长度 90 百分位为 14,500 token，是测试长文档 seq2seq 的理想场景。

| 模型 | ROUGE-1 | ROUGE-2 | ROUGE-L |
|---|---|---|---|
| Discourse-aware (2018) | 35.80 | 11.05 | 31.80 |
| Extr-Abst-TLM (2020) | 41.62 | 14.69 | 38.03 |
| Dancer (2020) | 42.70 | 16.54 | 38.44 |
| Pegasus (2020) | 44.21 | 16.95 | 38.83 |
| **LED-large (seqlen 4096)** | 44.40 | 17.94 | 39.76 |
| BigBird (seqlen 4096) | 46.63 | 19.02 | 41.77 |
| **LED-large (seqlen 16384)** | **46.63** | **19.62** | **41.83** |

LED-large 在序列长度为 16K 时取得了 SOTA，R-1 46.63，R-2 19.62，R-L 41.83。注意：

- BigBird 的结果来自从 Pegasus（专门为摘要预训练的模型）继续预训练的配置，而 LED 只是从 BART 初始化，**没有额外预训练**
- 用 16K 序列长度比 4K 有明显的提升（R-1 +2.23，R-2 +1.68），证明了处理更长输入的能力对长文档摘要至关重要
- 论文指出，如果对 LED 做额外的预训练，性能很可能进一步提升

### 输入长度的影响

Fig. 3 清晰地展示了序列长度对性能的影响：

| 序列长度 | ROUGE-1 | ROUGE-2 |
|---|---|---|
| 1K | 35.21 | 11.54 |

从 1K 到 4K 的提升远大于从 4K 到 16K，说明 **足够长的上下文** 是高质量摘要的必要条件，但收益是递减的。

Longformer 是高效注意力范式中的经典之作，提出了一种简单但有效的局部-全局注意力组合方案。其核心贡献包括：

1. **滑动窗口 + 空洞窗口 + 全局注意力** 的三层注意力设计，在保持线性复杂度的同时兼顾了局部和全局信息。
2. **完整的 pretrain-finetune pipeline**，从 RoBERTa 继续预训练并在多个下游任务上验证有效性——这是在当时大多数长文档 Transformer 没有做到的。
3. **LED 变体** 将 Longformer 推广到 seq2seq 场景，在 arXiv 摘要任务上取得了 SOTA。
4. **工程贡献**：使用 TVM 实现的自定义 CUDA kernel 支持空洞滑动窗口，同时提供了更易用的 chunks 实现用于预训练/微调场景。

Longformer 的设计影响了后续一系列工作，包括 ETC、BigBird 等。它的影响一直持续到今天——Hugging Face 的 `longformer` 和 `LED` 模型仍然是处理长文档 NLP 任务的常用选择，特别是当 BigBird 的随机注意力带来的收益不足以弥补其实施复杂度时，Longformer 的简洁性反而成为优势。
