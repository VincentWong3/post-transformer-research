## 一、先搞清楚坑在哪

Transformer 的核心组件是自注意力（self-attention），它允许每个 token 直接关注序列中的所有其他 token。这种全连接的结构带来了强大的建模能力，但也带来了一个根本性的问题：**计算复杂度是 $O(n^2)$**，其中 $n$ 是序列长度。

当处理短文本时，这个平方复杂度不是大问题。BERT 的标准序列长度是 512，RoBERTa 也是 512，对于大多数句子级别的任务已经足够。但现实世界中有大量需要理解长文档的场景：

- **法律文档**：一份合同可能有数万 token
- **科研论文**：一篇完整的论文通常有 3000-10000 token
- **书籍和长篇小说**：整本书显然无法放进 512 的窗口
- **医疗记录**：患者的完整病历可能跨越多年
- **代码仓库**：整个代码库的分析需要理解长程依赖

**关键问题**：现有的 Transformer 模型无法直接处理超过 512 token 的文本，因为：
1. 显存占用随序列长度平方增长（512²=262k vs 4096²=16.7M）
2. 计算时间同样平方增长
3. 简单的截断策略会丢失长程依赖信息

## 二、现有方法的真正问题

在 Longformer 之前，处理长文档的尝试主要有两条路径：

### 路径一：截断 + 分段

把长文档切成 512 token 的片段，分别送入 BERT，然后聚合结果。

**问题**：
- 跨片段的信息交互完全丢失
- 比如文档开头定义的术语在结尾使用时，模型无法建立联系
- 在 QA 任务中，答案可能依赖分散在多个段落的线索
- [Joshi et al., 2019] 的 TriviaQA 实验显示，分段策略在需要跨段推理的问题上表现很差

### 路径二：稀疏注意力

用各种稀疏注意力模式替代全连接注意力。代表性工作包括：
- **Sparse Transformer** [Child et al., 2019]：固定间隔的稀疏注意力
- **Reformer** [Kitaev et al., 2020]：LSH 哈希实现近似注意力
- **Routing Transformer** [Roy et al., 2020]：聚类路由注意力

**问题**：
- 这些方法的稀疏模式是固定的或基于聚类的，**无法保证覆盖所有重要的局部上下文**
- 许多方法在长文档任务上的实验不够充分
- 实现复杂，往往需要定制化的 CUDA 核函数

### 路径三：层级模型

先用浅层模型编码片段，再用 Transformer 聚合片段表示。

**问题**：
- 信息在层级压缩中丢失
- 细粒度的 token 级别交互被破坏
- 端到端训练困难

**Longformer 的核心洞察**：一个实用的长文档 Transformer 需要同时满足三个条件：
1. **线性复杂度**：随序列长度线性增长，而不是平方
2. **灵活的注意力模式**：既能捕获局部上下文，又能处理全局交互
3. **即插即用**：能直接替代现有 Transformer，无需重新训练

## 三、Longformer 的核心思路

一句话总结：**用多种注意力模式的组合（局部滑动窗口 + 全局稀疏 token + 任务特定的预训练）实现线性复杂度的长文档处理**。

### 高层面板

Longformer 的核心设计是**注意力模式的解耦**：

1. **局部滑动窗口注意力**（Sliding Window Attention）：每个 token 只关注其周围的 $w$ 个邻居（左右各 $w/2$），复杂度从 $O(n^2)$ 降到 $O(n \times w)$
2. **全局注意力**（Global Attention）：选择少量特殊 token（如 `[CLS]`），让它们关注整个序列，同时整个序列也关注它们
3. **膨胀滑动窗口**（Dilated Sliding Window）：在不增加计算量的情况下扩大感受野

这三种模式可以灵活组合，根据任务需求选择不同的配置。

### 4.2 前向过程详解

让我们跟踪一个长度为 $n=4096$ 的序列通过 Longformer 的完整流程：

```
输入序列: [x₁, x₂, ..., x₄₀₉₆]
         ↓
    Token Embedding + Position Embedding
         ↓
    [h₁, h₂, ..., h₄₀₉₆]  shape: (4096, d_model=768)
         ↓
    ┌─────────────────────────────────────────────┐
    │   Longformer Encoder Layer × 12             │
    │                                              │
    │   1. Longformer Self-Attention               │
    │      - Sliding window (w=512)                │
    │      - Global attention (selected tokens)    │
    │      - Output shape: (4096, 768)             │
    │                                              │
    │   2. LayerNorm + Residual                    │
    │      - Output shape: (4096, 768)             │
    │                                              │
    │   3. Feed-Forward (intermediate=3072)        │
    │      - Output shape: (4096, 768)             │
    │                                              │
    │   4. LayerNorm + Residual                    │
    │      - Output shape: (4096, 768)             │
    └─────────────────────────────────────────────┘
         ↓
    最终表示: [z₁, z₂, ..., z₄₀₉₆]  shape: (4096, 768)
```

### 4.3 注意力计算细节

#### 标准注意力 vs Longformer 注意力

**标准注意力**（以 BERT 为例）：
```
Q = XW_Q, K = XW_K, V = XW_V    # shape: (n, d_k)
A = softmax(QK^T / √d_k)         # shape: (n, n)  ← 平方复杂度
O = AV                           # shape: (n, d_v)
```

**Longformer 注意力**：
```
# 对于每个 token i：
# 只关注窗口内的邻居和全局 token
window = [i-w/2, ..., i, ..., i+w/2]  # w 个 token
globals = 预定义的全局 token 集合

attention_set(i) = window ∪ globals
# |attention_set(i)| ≈ w + g  << n
# 复杂度: O(n × (w + g))  ← 线性复杂度
```

### 4.4 为什么滑动窗口有效

这是一个反直觉的设计：为什么限制注意力范围反而能工作？

**直觉**：
1. **局部性先验**：在自然语言中，大多数信息交互发生在局部上下文中。一个词的含义主要由其周围的词决定
2. **层级结构**：浅层关注局部特征，深层通过多层堆叠逐步扩大感受野
3. **计算效率**：将计算资源集中在真正重要的局部交互上，而不是浪费在远距离的弱关联上

**有意思的是**：12 层 Longformer 的滑动窗口感受野是 $w \times 12 = 512 \times 12 = 6144$ token，远超单层的窗口大小。这是通过层间传播实现的——第 2 层能"看到"第 1 层窗口外的信息。

## 五、核心创新点

### 创新点一：滑动窗口注意力

#### (a) 痛点与动机

**问题**：全连接注意力的 $O(n^2)$ 复杂度使长文档处理不可行。

**为什么现有方法失败**：
- [Vaswani et al., 2017] 的标准 Transformer：复杂度 $O(n^2)$，无法扩展到 4096+ token
- [Child et al., 2019] 的固定间隔稀疏：虽然降低了复杂度，但固定的稀疏模式无法适应不同位置的上下文需求

**动机**：需要一个既能保持局部建模能力，又能线性扩展的注意力机制。

#### (b) 方案细节

滑动窗口注意力为每个 token $i$ 定义关注集合：
```
N(i) = {j : |i - j| ≤ w/2}
```

其中 $w$ 是窗口大小。每个 token 只关注其左右各 $w/2$ 个邻居。

**实现细节**：
- 使用对角线带状矩阵（banded matrix）存储注意力分数
- 每个 token 计算 $w$ 个注意力分数（不是 $n$ 个）
- 复杂度从 $O(n^2)$ 降到 $O(n \times w)$


#### (c) 为什么有效

1. **计算效率**：$w=512$ 时，$n \times w = 4096 \times 512 = 2.1M$，远小于 $n^2 = 16.7M$
2. **多层叠加**：12 层的感受野 = $512 \times 12 = 6144$，远大于单层窗口
3. **实验验证**：表 2 显示，滑动窗口在多个任务上接近甚至超过全连接注意力的性能

#### (d) 与 Related Work 的关系

- **Sparse Transformer** [Child et al., 2019]：使用固定步长的稀疏注意力，但 Longformer 的滑动窗口更符合语言局部性
- **Adaptive Span** [Sukhbaatar et al., 2019]：学习每个 head 的注意力范围，但需要额外的参数和训练
- **Image Transformer** [Parmar et al., 2018]：在图像上使用局部注意力，Longformer 将其推广到文本

#### (e) 如果去掉会怎样

表 3 的消融实验显示：
- 去掉滑动窗口（改用全连接）：显存从 4.8GB 暴涨到 40.4GB（8x 增长）
- 去掉所有注意力优化：无法在 GPU 上处理超过 1024 token 的序列

### 创新点二：全局注意力

#### (a) 痛点与动机

**问题**：纯局部注意力无法处理需要全局信息的任务（如文档分类、QA）。

**为什么局部注意力不够**：
- 分类任务：需要 `[CLS]` token 汇聚全局信息
- QA 任务：问题 token 需要关注文档中所有可能包含答案的位置
- [Devlin et al., 2019] 的 BERT 使用 `[CLS]` 分类，但需要全连接注意力支持

**动机**：在不增加整体复杂度的情况下，为关键 token 提供全局视野。

#### (b) 方案细节

全局注意力为选定的 token（如 `[CLS]`、`[SEP]`、问题 token）提供全连接注意力：

```
# 选定全局 token 集合 G
# 对于全局 token g ∈ G：
#   attention_set(g) = 所有 n 个 token
# 对于普通 token i ∉ G：
#   attention_set(i) = 窗口邻居 + 所有全局 token

# 双向关注：
# 全局 token 关注所有普通 token
# 普通 token 也关注全局 token
```

**关键设计**：全局 token 的数量 $g$ 远小于 $n$（通常 $g < 100$），所以增加的复杂度是 $O(n \times g)$，仍然保持线性。

#### (c) 为什么有效

1. **任务自适应**：不同任务选择不同的全局 token
   - 分类：`[CLS]` 作为全局 token
   - QA：问题 token 作为全局 token
2. **最小开销**：$g$ 个全局 token 只增加 $n \times g$ 的计算量
3. **双向交互**：全局 token 能获取全序列信息，同时普通 token 也能获取全局上下文

#### (d) 与 Related Work 的关系

- **BERT** [Devlin et al., 2019]：使用全连接注意力，所有 token 都是"全局"的
- **XLNet** [Yang et al., 2019]：使用双流注意力，但仍然是全连接
- Longformer 的创新在于：**选择性全局**——只在需要的地方使用全连接

#### (e) 如果去掉会怎样

表 3 的消融实验显示：
- 去掉全局注意力：QA 任务的 F1 从 67.8 降到 54.2（下降 13.6 点）
- 去掉全局注意力：分类任务的准确率从 94.0 降到 91.3（下降 2.7 点）

### 创新点三：膨胀滑动窗口

#### (a) 痛点与动机

**问题**：普通滑动窗口的感受野增长太慢。12 层窗口大小为 512 的 Longformer，感受野是 6144 token，但需要 12 层才能达到。

**为什么需要更大的感受野**：
- 某些任务需要更远距离的依赖（如篇章级共指消解）
- 增加窗口大小会线性增加计算量

**动机**：在不增加计算量的情况下扩大感受野。

#### (b) 方案细节

膨胀滑动窗口在每 $d$ 个 token 中只关注一个（类似空洞卷积）：

```
# 普通滑动窗口（窗口大小 7）：
关注: [i-3, i-2, i-1, i, i+1, i+2, i+3]

# 膨胀滑动窗口（窗口大小 7，膨胀率 2）：
关注: [i-6, i-4, i-2, i, i+2, i+4, i+6]
```

**关键设计**：不同层使用不同的膨胀率
- 低层：小膨胀率或无膨胀（捕获局部模式）
- 高层：大膨胀率（捕获长程依赖）

#### (c) 为什么有效

1. **计算量不变**：每个 token 仍然只关注 $w$ 个 token
2. **感受野扩大**：膨胀率 $d$ 使感受野扩大 $d$ 倍
3. **层级配合**：低层捕获局部特征，高层捕获长程依赖

#### (d) 与 Related Work 的关系

- **WaveNet** [van den Oord et al., 2016]：在音频生成中使用膨胀卷积
- **Sparse Transformer** [Child et al., 2019]：使用固定步长的稀疏注意力，但 Longformer 的膨胀是连续的（注意相邻 token）

#### (e) 如果去掉会怎样

表 3 显示膨胀滑动窗口的效果因任务而异：
- 文本分类：膨胀有帮助（+0.5%）
- QA：膨胀帮助不大（+0.1%）
- 核心任务：膨胀有负面影响（-0.3%）

这说明膨胀不是万能的，需要根据任务选择。

### 创新点四：分步预训练策略

#### (a) 痛点与动机

**问题**：Longformer 的注意力模式与 BERT 完全不同，不能直接使用 BERT 的预训练权重。

**为什么不能直接微调**：
- BERT 使用全连接注意力，Longformer 使用稀疏注意力
- 直接替换注意力层会导致严重的性能下降
- [Liu et al., 2019] 的 RoBERTa 显示，预训练对性能至关重要

**动机**：需要一个能有效利用现有预训练权重的训练策略。

#### (b) 方案细节

Longformer 采用两步预训练策略：

**第一步：扩展位置编码**
```
1. 加载 BERT 权重
2. 将位置编码从 512 扩展到 4096
3. 初始化方法：复制最后的位置向量
   pos_4096[i] = pos_512[min(i, 511)]
```

**第二步：分阶段训练**
```
阶段 1（步 1-10000）：
  - 窗口大小 w = 32（小窗口，快速适应）
  - 学习率：5e-5
  - 批次大小：64

阶段 2（步 10000-20000）：
  - 窗口大小 w = 128
  - 学习率：3e-5

阶段 3（步 20000-30000）：
  - 窗口大小 w = 512（最终窗口）
  - 学习率：1e-5

阶段 4（步 30000-40000）：
  - 加入膨胀（不同层不同膨胀率）
  - 学习率：5e-6
```

#### (c) 为什么有效

1. **渐进适应**：从小窗口开始，逐步扩大，让模型慢慢适应稀疏注意力
2. **保持知识**：使用 BERT 的初始化权重，保留已有语言知识
3. **稳定性**：小窗口阶段训练稳定，大窗口阶段性能提升

#### (d) 与 Related Work 的关系

- **BERT** [Devlin et al., 2019]：Longformer 使用其权重初始化
- **RoBERTa** [Liu et al., 2019]：Longformer 使用其训练数据和配置
- **T5** [Raffel et al., 2020]：使用不同的预训练目标，但 Longformer 保持 MLM

#### (e) 如果去掉会怎样

表 4 的消融实验显示：
- 不进行分步训练：验证集困惑度从 4.12 升到 4.89（+18.7%）
- 不进行位置编码扩展：无法处理超过 512 token 的序列
- 不使用 BERT 初始化：需要更多训练数据和时间

## 六、公式详解

### 6.1 注意力复杂度公式

**标准自注意力**：
$$ \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V $$

**复杂度分析**：
- $QK^T$ 计算：$n \times d_k \times n = O(n^2 d_k)$
- softmax：$O(n^2)$
- $AV$ 计算：$n \times n \times d_v = O(n^2 d_v)$
- 总复杂度：$O(n^2 d)$

**Longformer 注意力**：
$$ \text{LongAttention}(Q, K, V) = \text{softmax}\left(\frac{QK^T \odot M}{\sqrt{d_k}}\right)V $$

其中 $M$ 是注意力掩码矩阵，$M_{ij} = 1$ 当且仅当 $j \in \text{attention\_set}(i)$。

**复杂度分析**：
- 每个 token 的注意力计算量：$O(w d)$
- 总复杂度：$O(n w d)$
- 当 $w \ll n$ 时，这是线性复杂度

### 6.2 具体数值计算

以 $n=4096, w=512, d=768$ 为例：

**标准注意力**：
$$ \text{计算量} = n^2 \times d = 4096^2 \times 768 = 12.9 \times 10^9 $$

**Longformer**：
$$ \text{计算量} = n \times w \times d = 4096 \times 512 \times 768 = 1.6 \times 10^9 $$

**加速比**：
$$ \frac{12.9 \times 10^9}{1.6 \times 10^9} = 8.06 \times $$

**显存对比**（表 3 数据）：
| 配置 | 显存占用 | 加速比 |
|------|---------|--------|
| 全连接 | 40.4 GB | 1x |
| 滑动窗口 (w=512) | 4.8 GB | 8.4x |
| 滑动窗口+全局 | 5.2 GB | 7.8x |

### 6.3 窗口大小的选择

$$ \text{感受野}(L, w, d) = w \times L \quad \text{(无膨胀)} $$
$$ \text{感受野}(L, w, d) = w \times L \times d \quad \text{(有膨胀)} $$

其中 $L$ 是层数，$w$ 是窗口大小，$d$ 是膨胀率。

**具体数值**：
- 12 层，窗口 512：感受野 = $512 \times 12 = 6144$
- 12 层，窗口 512，膨胀率 2：感受野 = $512 \times 12 \times 2 = 12288$

**边界情况**：
- $w = n$：退化为全连接注意力（复杂度 $O(n^2)$）
- $w = 1$：退化为逐 token 处理（无上下文信息）
- $d > w/2$：膨胀窗口不再连续，可能丢失局部信息

## 十、局限

### 10.1 窗口大小选择困难

**问题**：窗口大小 $w$ 是一个超参数，不同任务需要不同的窗口大小。
- 分类任务：窗口 512 就够
- QA 任务：可能需要更大的窗口

**影响**：需要针对任务调参，增加使用成本。

**论文是否承认**：是，论文讨论了不同窗口大小的影响（表 3）。

### 10.2 全局 token 的选择

**问题**：全局 token 的选择需要任务特定的知识。
- 分类：`[CLS]`
- QA：问题 token
- 其他任务：需要人工指定

**影响**：限制了方法的通用性。

**后续工作**：Big Bird 使用随机选择全局 token，减少了人工干预。

### 10.3 膨胀滑动窗口的收益不确定

**问题**：表 3 显示膨胀的收益因任务而异，有时甚至有害。
- 分类：+0.5%
- QA：+0.1%
- 核心指代：-0.3%

**影响**：膨胀不是通用的改进，需要谨慎使用。

### 10.4 预训练成本高

**问题**：分步预训练需要 40000 步，在 8 卡 V100 上需要数天。

**影响**：资源受限的团队难以复现。

## 十一、小结

**Longformer 解决了一个核心问题**：如何让 Transformer 高效处理长文档。通过将全连接注意力替换为滑动窗口注意力 + 选择性全局注意力，Longformer 将复杂度从 $O(n^2)$ 降到 $O(n)$，同时保持了强大的建模能力。

**为什么它有效**：
1. 滑动窗口利用了语言的局部性先验
2. 全局注意力为关键 token 提供了全局视野
3. 分步预训练策略有效利用了 BERT 的预训练权重
4. 灵活的组合模式适应不同任务需求

**更广泛的影响**：
- 开启了长文档 Transformer 的研究方向
- 影响了后续的 Big Bird、LED、SLED 等工作
- 在实际应用中（法律、医疗、科研）有重要价值
- 证明了稀疏注意力可以替代全连接注意力而不损失太多性能
