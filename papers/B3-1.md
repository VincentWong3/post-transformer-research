# A ConvNet for the 2020s（2022）

**论文：** [arXiv:2201.03545](https://arxiv.org/abs/2201.03545) · Zhuang Liu, Hanzi Mao, Chao-Yuan Wu, Christoph Feichtenhofer, Trevor Darrell, Saining Xie · FAIR / UC Berkeley · CVPR 2022

**技术分支：** B3 — 视觉 backbone 是否需要局部归纳偏置？ / B3-1 — local inductive bias 回归范式

**代码：** [facebookresearch/ConvNeXt](https://github.com/facebookresearch/ConvNeXt)

---

## 一、先搞清楚"问题在哪里"

2020 年之后，视觉领域发生了一件看起来不可逆转的事：Transformer 开始全面取代 CNN，成为视觉 backbone 的首选。

这件事的起点是 ViT。ViT 在 ImageNet 上证明了纯 Transformer（没有卷积）可以超越当时最好的 CNN。随后 Swin Transformer 进一步证明了层级化 Transformer 在检测、分割等密集预测任务上同样可以大幅超越 ResNet。到 2021 年底，几乎所有顶会论文中，Swin Transformer 都取代了 ResNet 成为默认 backbone。

看起来 CNN 的时代结束了。Transformer 就是更好。

但这里有一个被大多数人忽略的问题：**ViT/Swin 和 ResNet 之间的对比，真的公平吗？**

ViT 用了 AdamW 优化器、余弦学习率衰减、300 epoch 训练、强数据增强（Mixup、CutMix、RandAugment）、Stochastic Depth、Label Smoothing……而 ResNet 的对比基线用的是 90 epoch 的 SGD 训练，数据增强只有水平翻转和随机裁剪。这不是公平对比。这是用 2021 年的训练技术去对比 2015 年的训练技术。

换句话说，如果给 ResNet 换上同样的现代训练技术，它会不会也能赶上 Transformer？

这个问题在 2021 年底是非常尖锐的。因为当时社区的主流叙事是"Transformer 架构本身更好"，很少有人认为训练技术才是真正的拉差因素。这篇论文的动机就是：**把 ResNet 逐步改造成类似 Swin 的架构，用相同的训练设置，看看到底每一步改进了多少。如果每一步提升都不大，说明真正差距在架构之外的地方。**

这就是 ConvNeXt 的出发点。它不打算发明新的模块，也不打算推翻 Transformer，而是要做一次严谨的消融实验。

## 二、旧路线的真正问题

要理解 ConvNeXt 为什么重要，先得搞清楚 2021 年 CNN 和 Transformer 的对比困境。

### 2.1 ResNet 停在 2015 年

ResNet-50 发表于 2015 年。它的训练设置是：

- 优化器：SGD + momentum
- 学习率调度：Step decay（30 epoch 一降）
- 训练 epoch：90
- 数据增强：Random crop + horizontal flip
- 正则化：Weight decay + BN
- 无：Label smoothing、Stochastic Depth、Mixup、CutMix、RandAugment、EMA

到了 2021 年，DeiT 和 Swin 的训练设置完全不同：

- 优化器：AdamW（权重衰减解耦）
- 学习率调度：Cosine annealing
- 训练 epoch：300
- 数据增强：Mixup + CutMix + RandAugment + Random Erasing
- 正则化：Stochastic Depth + Label Smoothing

这两套训练方案的差距有多大？论文实验表明，单纯把 ResNet-50 从原始训练切换到现代训练方案，ImageNet Top-1 准确率就从 76.1% 提升到了 78.8%（+2.7%）。而这个 76.1% 正是很多 ViT/Transformer 论文用作 CNN 基线的数值。

**所以，Transformer 所谓的大幅超越 CNN，很大一部分是训练技术红利，不是架构红利。**

### 2.2 架构层面的差距在哪里

排除了训练技术因素后，ResNet 和 Swin Transformer 之间仍然存在几个真正的架构差异：

**Stage 计算比例不合理。** ResNet-50 的 stage block 分配是 [3,4,6,3]，计算集中在 "res4"（14×14 分辨率），而这个设计是为了兼容 Faster R-CNN 的检测头。Swin-T 的分配是 [1,1,3,1]（小模型）或 [1,1,9,1]（大模型），计算集中在最低分辨率 stage。后来的研究发现，更深的低分辨率 stage 对分类和密集预测都有好处。

**Bottleneck 结构 vs Inverted Bottleneck。** ResNet 使用 Bottleneck（宽→窄→宽），MobileNetV2 和 Transformer 使用 Inverted Bottleneck（窄→宽→窄）。后者在深度可分离卷积的设置下计算效率更高。

**卷积核大小。** ResNet 坚持堆叠 3×3 小卷积核（VGG 传统），而 Swin Transformer 的最小局部窗口是 7×7。更大的感受野直接意味着更强的特征表达能力。

**激活函数和归一化。** ResNet 用 ReLU + BN，Swin 用 GELU + LN。虽然 BN 在小 batch 时有 batch 依赖问题，但在 ImageNet 训练中 BN 的效果其实不差。不过 LN 在 transfer learning 和微调场景下更稳定。

**下采样机制。** ResNet 用 stride=2 的 3×3 conv 在 block 内部下采样，而 Swin 用独立的 Patch Merging（类似于 2×2 卷积 stride=2）。分离下采样层可以让每个 stage 内部保持统一的计算模式。

### 2.3 卷积本身被低估了什么

除了上述工程层面的差异，卷积的两个本质优势被舆论低估了：

**平移等变性（translation equivariance）。** 卷积核在整张图上滑动，同一特征不论出现在图像哪个位置，都能被相同的卷积核检测到。自注意力不具备这个性质——它需要学出来。

**计算共享。** 卷积在滑动窗口模式下，相邻窗口的计算高度共享，因为卷积核参数是共享的。自注意力窗口之间存在信息冗余，每个查询 pixel 都需要重新计算和所有 key 的注意力。

但卷积也有一个核心弱点：**局部感受野。** 标准 3×3 卷积堆叠 50 层，理论感受野可以达到整个图像，但实际有效感受野远小于理论值（ResNet 的有效感受野大约只有图像尺寸的 1/10）。这也是为什么 Transformer 的全局注意力在某些场景下更有效。

### 2.4 问题总结

2021 年底的核心困境是：

> 社区普遍相信 Transformer 架构优于 CNN，但这个信念建立在两个不公平的前提上：
> 1. CNN 的基线使用的是过时的训练设置
> 2. CNN 的架构设计停留在了 2015 年，没有吸收后续的改进

这篇论文要做的，就是在这两个维度上同时修复不公平，然后看 CNN 到底能跑多快。

## 三、这篇论文的核心思路

> **这篇论文的核心思路是：以 ResNet 为起点，以 Swin Transformer 为设计目标，逐步"现代化"每一层设计选择，在保持纯卷积的前提下，最终得到一个性能超越 Swin 的 ConvNet。**

这不是一个"我发明了新模块，所以更好"的故事。这是一个"你告诉我 Transformer 为什么好，我把它翻译成卷积语言"的故事。

具体来说，论文沿着一条严格定义的改造路线前进：

```text
ResNet-50 (2015)              — 起点，训练技术 + 架构都过时
  ↓ 训练技术升级               — 替换为 AdamW + 300 epoch + 强增强 + 正则化
ResNet-50 (modern training)   — 差一点就是公平对比起点
  ↓ stage 计算比例调整          — [3,4,6,3] → [3,3,9,3]
  ↓ stem 改成 patchify          — 7×7 stride2 + maxpool → 4×4 stride4
  ↓ 引入 depthwise conv         — ResNeXt 风格，增加 width
  ↓ inverted bottleneck         — bottleneck → inverted bottleneck
  ↓ 大卷积核 depthwise conv     — 3×3 → 7×7
  ↓ 微观设计调整                 — ReLU → GELU, BN → LN, 分离下采样
ConvNeXt (2022)                — 终点，纯卷积但超越 Swin
```

每一个步骤都是一个独立的设计选择。每一步改造都是在问同一个问题：**"如果把这一层改成和 Swin 一样，ConvNet 能涨多少？"**

这个研究范式的核心价值是：它不是一个黑盒对比（Swin vs ResNet），而是一个白盒消融（每个设计选择的边际贡献）。

## 四、整体结构与 forward 流程

ConvNeXt 的架构几乎完全复用了 Swin Transformer 的宏观设计，只是把自注意力替换成了深度可分离卷积。

### 4.1 整体架构

```text
输入图像: (B, 3, H, W)
    │
    │── Stem: Conv2d(3, 96, kernel_size=4, stride=4)
    │   输出: (B, 96, H/4, W/4)
    │
    │── LayerNorm (channels_first)
    │
    │── Stage 1: ConvNeXt Block × 3, dim=96
    │   输出: (B, 96, H/4, W/4)
    │
    │── Downsample: LayerNorm → Conv2d(96, 192, kernel_size=2, stride=2)
    │   输出: (B, 192, H/8, W/8)
    │
    │── Stage 2: ConvNeXt Block × 3, dim=192
    │   输出: (B, 192, H/8, W/8)
    │
    │── Downsample: LayerNorm → Conv2d(192, 384, kernel_size=2, stride=2)
    │   输出: (B, 384, H/16, W/16)
    │
    │── Stage 3: ConvNeXt Block × 9, dim=384
    │   输出: (B, 384, H/16, W/16)
    │
    │── Downsample: LayerNorm → Conv2d(384, 768, kernel_size=2, stride=2)
    │   输出: (B, 768, H/32, W/32)
    │
    │── Stage 4: ConvNeXt Block × 3, dim=768
    │   输出: (B, 768, H/32, W/32)
    │
    │── Global Average Pooling
    │── LayerNorm
    │── Linear(768, num_classes)
```

### 4.2 和 Swin-T 的对应关系

| 组件 | Swin-T | ConvNeXt-T |
|------|--------|------------|
| Stem | Patch Embedding (4×4, stride 4) | Conv2d (4×4, stride 4) |
| Stage 1 | SwinBlock × 2 | ConvNeXtBlock × 3 |
| Downsample 1 | Patch Merging (2×2, stride 2) | Conv2d (2×2, stride 2) + LN |
| Stage 2 | SwinBlock × 2 | ConvNeXtBlock × 3 |
| Stage 3 | SwinBlock × 6 | ConvNeXtBlock × 9 |
| Stage 4 | SwinBlock × 2 | ConvNeXtBlock × 3 |
| 归一化 | LN | LN |
| 分类头 | LN + Linear | LN + Linear |

核心区别：SwinBlock 中包含 W-MSA/SW-MSA（窗口注意力 + 移位窗口注意力），而 ConvNeXtBlock 中替换为了 Depthwise Conv。

### 4.3 ConvNeXt Block 的 forward 流程

```text
输入: x (B, C, H, W)
    │
    │── Depthwise Conv7×7 (spatial mixing)
    │   groups=C, padding=3
    │   输出: (B, C, H, W)    ← 空间维度信息交互
    │
    │── Permute: (B, C, H, W) → (B, H×W, C)
    │
    │── LayerNorm (channel-wise)
    │
    │── Linear: C → 4C        ← 通道扩展（inverted bottleneck 第一步）
    │
    │── GELU 激活
    │
    │── Linear: 4C → C        ← 通道压缩（inverted bottleneck 第二步）
    │
    │── Layer Scale (可选): scale = gamma × x, gamma 初始化为 1e-6
    │
    │── Permute: (B, H×W, C) → (B, C, H, W)
    │
    │── DropPath (Stochastic Depth)
    │
    │── 残差连接: x = x + block_output
    │
    输出: (B, C, H, W)
```

这个 block 的结构对应关系：

- **Depthwise Conv** → 对应 Transformer 的 Self-Attention（负责 token 间信息交互）
- **Linear(C→4C) + GELU + Linear(4C→C)** → 对应 Transformer 的 FFN（通道维度信息变换）
- **两个 LN** → 对应 Pre-LN Transformer 中 Attention 前和 FFN 前的 LN
- **残差连接** → 标准 Transformer 残差

### 4.4 模型规格

| 模型 | dim | depths | 参数量 | FLOPs (224²) |
|------|-----|--------|--------|--------------|
| ConvNeXt-T | 96 | [3,3,9,3] | 29M | 4.5G |
| ConvNeXt-S | 96 | [3,3,27,3] | 50M | 8.7G |
| ConvNeXt-B | 128 | [3,3,27,3] | 89M | 15.4G |
| ConvNeXt-L | 192 | [3,3,27,3] | 198M | 34.5G |
| ConvNeXt-XL | 256 | [3,3,27,3] | 350M | 60.9G |

与 Swin 一一对应：ConvNeXt-T ↔ Swin-T，ConvNeXt-S ↔ Swin-S，ConvNeXt-B ↔ Swin-B，ConvNeXt-L ↔ Swin-L。

## 五、核心模块逐个解释

### 5.1 Depthwise Convolution（深度可分离卷积）

**它解决了什么问题？** 如何在保持 CNN 归纳偏置的前提下，高效地进行空间信息交互。

**为什么需要它？** 标准卷积同时做两件事：空间信息融合（K×K 邻域内）和通道信息融合（跨 channel）。ResNeXt 的分组卷积思想指出，可以把这两件事分开做，效率更高。Depthwise Conv 把分组数设为 channels，相当于完全去掉通道维度的混合。

**它是怎么工作的？** 每个输出通道只由一个卷积核处理，该卷积核只在一个输入通道上滑动。对于输入 (B, C, H, W)，depthwise conv 有 C 个 kernel，每个 kernel 的 shape 是 (K, K, 1)。

```python
# standard conv: (B, C_in, H, W) → (B, C_out, H', W')
# weights: (K, K, C_in, C_out)
# FLOPs: K² × C_in × C_out × H' × W'

# depthwise conv: (B, C, H, W) → (B, C, H', W')
# weights: (K, K, C)   [or more precisely (K, K, 1, C) in PyTorch idiom]
# FLOPs: K² × C × H' × W'
# groups=C in Conv2d
```

**它和自注意力的联系是什么？** 自注意力中，每个 token 通过 query 和 key 的加权求和聚合其邻域信息，这种聚合是逐通道的（每个头独立算注意力权重）。Depthwise Conv 也是逐通道的：每个通道有各自的空间权重，独立进行空间信息混合。两者都在做"空间混合 + 通道独立"。

**为什么是 7×7？** 因为 Swin Transformer 的窗口大小是 7×7，为了公平对比。更大的 kernel 意味着更大的有效感受野（从 3×3 的 3 扩大到 7 ），计算量从 9C 增加到 49C，但因为 depthwise 的性质，增加的是 C 倍而不是 C² 倍。

**引入后的性能变化：** 在 ResNet-50 规模的消融实验中，加上 7×7 depthwise conv（配合之前的修改）后，准确率从 79.9% 提升到 80.6%（7×7 depthwise conv）。

### 5.2 Inverted Bottleneck（倒瓶颈结构）

**它解决了什么问题？** ResNet 的 bottleneck 结构（C → C/4 → C）在 depthwise conv 场景下效率不高。

**为什么需要它？** 在 depthwise conv 中，空间信息混合只消耗了很少的 FLOPs，大部分计算在 1×1 conv 上。如果用传统的 bottleneck，先把通道压缩，再在窄通道上做 depthwise conv，再扩展，那大部分可学习的参数被浪费在窄通道上。用 inverted bottleneck 先把通道扩展 4 倍，再在宽通道上做 depthwise conv，最后压缩回来，参数效率更高。

**结构对比：**

```text
ResNet bottleneck:            ConvNeXt inverted bottleneck:
    C                              C
    │                              │
 Conv1×1: C → C/4          Linear: C → 4C
    │                              │
 ReLU                          GELU
    │                              │
 Conv3×3: C/4 → C/4  (stride)  DWConv7×7: 4C → 4C
    │                              │
 ReLU                          ─ (no activation after DWConv)
    │                              │
 Conv1×1: C/4 → C            Linear: 4C → C
    │                              │
残差连接                       残差连接
```

**为什么这个设计合理？** 在 depthwise conv 场景下，inverted bottleneck 让大部分 FLOPs 集中在 1×1 卷积上（通道混合），而 depthwise conv（空间混合）因为 C 扩大了 4 倍而消耗更多计算。但这种"让空间混合在宽通道上进行"的方式，实际上加强了每个通道的空间特征表达能力。

**引入后的性能变化：** 在这个步骤，ResNet-50 规模下准确率从 80.5% 微涨到 80.6%（但 FLOPs 从 5.3G 降到 4.6G）。在 ResNet-200/Swin-B 规模下，这个改动带来了更大的提升（81.9% → 82.6%，同时 FLOPs 从 15.0G 降到 14.3G）。这说明 inverted bottleneck 在大模型上收益更明显。

### 5.3 大卷积核（7×7 Depthwise Conv）

**它解决了什么问题？** 3×3 卷积的感受野太小，限制了深层对全局结构的感知。

**为什么之前不用大卷积核？** VGG 用实验证明了"堆叠小卷积核（3×3）等价于大卷积核感受野"这一结论，而且 3×3 卷积在 GPU 上效率极高（cuDNN 高度优化）。AlexNet 和 Inception 用过 7×7、5×5，但都被 VGG 的 3×3 堆叠取代了。

**为什么现在可以用？** 因为 depthwise conv。标准 7×7 conv 的 FLOPs 是 49 × C² × H' × W'，而 depthwise 7×7 conv 的 FLOPs 是 49 × C × H' × W'。C 从 64/128 开始，49 × C² 和 49 × C 差了 C 倍（64~128 倍）。所以 depthwise conv 让大卷积核变得经济可行。

**位置：** 在 ConvNeXt Block 中，depthwise conv 是空间信息混合的核心操作，放在 block 的开头（对应 Swin 的 attention）。

**为什么必须移动 depthwise conv 的位置？** 在标准 ResNeXt block 中，3×3 conv 放在两个 1×1 conv 之间。在 inverted bottleneck 设置下，如果把 7×7 depthwise conv 也放在中间，就要在 4C 的宽通道上执行大核卷积，这会导致 FLOPs 大幅上升。把 depthwise conv 移到 block 开头（在 C 通道上），1×1 conv 放在后面（在 4C 通道上），FLOPs 更低，效果更好。

### 5.4 分离的下采样层

**它解决了什么问题？** ResNet 在每个 stage 的残差连接中通过 stride=2 的 3×3 conv 做下采样，这导致下采样和特征提取耦合在一起。

**为什么需要分离？** 分离后的下采样层（2×2 conv, stride=2）只做一件事情：降低空间分辨率并翻倍通道数。每个 stage 内部的 block 都保持输入分辨率不变。这让每个 stage 的处理模式统一，也便于和 Swin 的 Patch Merging 直接对比。

**在 forward 中的位置：** 在 Stage 1→2, 2→3, 3→4 之间插入。先做 LN（稳定训练），再做 2×2 conv。

### 5.5 Layer Normalization 和 GELU

**LN 替代 BN：** BN 在训练时依赖 batch 统计量，在 batch size 小或 batch 之间统计量差异大的情况下不稳定。LN 在通道维度上归一化，不依赖 batch，在微调和迁移学习中更稳定。虽然 ConvNeXt 在 ImageNet 训练时 batch size 很大（4096），BN 表现不差，但为了和 Swin 公平对比，替换为 LN。

**GELU 替代 ReLU：** GELU（Gaussian Error Linear Unit）可以理解为 ReLU 的平滑版本。ReLU 在 x<0 时完全截止（梯度为零），GELU 在 x<0 时保留少量负值，允许负值缓慢衰减。这在深层网络中提供了更好的梯度流动。

## 六、核心公式与数学直觉

### 6.1 Depthwise Convolution 的计算公式

标准卷积：

$$y_{i,j,k} = \sum_{p=-⌊K/2⌋}^{⌊K/2⌋} \sum_{q=-⌊K/2⌋}^{⌊K/2⌋} \sum_{c=1}^{C_{in}} x_{i+p, j+q, c} \cdot w_{p,q,c,k}$$

其中 $w$ 的 shape 是 $(K, K, C_{in}, C_{out})$。复杂度 $O(K^2 \cdot C_{in} \cdot C_{out} \cdot H \cdot W)$。

Depthwise Conv（设 $C_{in} = C_{out} = C$，groups=C）：

$$y_{i,j,c} = \sum_{p=-⌊K/2⌋}^{⌊K/2⌋} \sum_{q=-⌊K/2⌋}^{⌊K/2⌋} x_{i+p, j+q, c} \cdot w_{p,q,c}$$

其中 $w$ 的 shape 是 $(K, K, C)$。复杂度 $O(K^2 \cdot C \cdot H \cdot W)$。

**这个式子告诉我们什么？** 标准卷积的 FLOPs 与 $C_{in} \cdot C_{out}$ 成正比（约 $C^2$），而 depthwise conv 的 FLOPs 与 $C$ 成正比。当 $C=96$ 时，depthwise 7×7 的 FLOPs 是标准 7×7 的 $1/96$。这就是为什么 depthwise conv 可以承受更大的卷积核。

### 6.2 ConvNeXt Block 的残差结构

$$\mathbf{x}' = \mathbf{x} + \text{DropPath}(\text{ScaleConv}_{1\times1}^{\text{4C→C}} \circ \text{GELU} \circ \text{Conv}_{1\times1}^{\text{C→4C}} \circ \text{LN} \circ \text{DWConv}_{7\times7}^{\text{C}} (\mathbf{x}))$$

其中 $\text{ScaleConv}$ 是带 Layer Scale 的线性层。

对比 Swin Transformer Block：

$$\mathbf{x}' = \mathbf{x} + \text{DropPath}(\text{FFN} \circ \text{LN} \circ \text{W-MSA/SW-MSA} \circ \text{LN}(\mathbf{x}))$$

这里 LN 的位置有细微差别：Swin 是 Pre-LN（norm → attention / FFN），ConvNeXt 也是 Pre-LN，但把 LN 放在了 DWConv 之后（Permute 之后）。实际上 ConvNeXt Block 中 DWConv 前还有一个 LN（在 Permute 之前），所以是"Pre-DWConv LN + Pre-FFN LN"的双 LN 结构。

### 6.3 感受野的计算

堆叠 L 层 kernel size = K 的卷积，理论感受野：

$$\text{RF}(L) = L \cdot (K - 1) + 1$$

堆叠 3 层 3×3 卷积：$\text{RF}(3) = 3 \cdot 2 + 1 = 7$，等于 1 层 7×7 卷积。

但有效感受野（实际能利用的感受野）远小于理论值。对于 3×3 卷积堆叠 50 层，有效感受野大约只有图像尺寸的 10-20%。而直接用 7×7 卷积，第一层就有 7×7 的有效感受野，在深层可以维持更大的有效感受野。

**直觉：** 用大卷积核而不是堆叠小卷积核，相当于每个位置从一开始就能看到更大的局部区域，而不是"层层传递"。

### 6.4 Layer Scale 的数学形式

$$\mathbf{y} = \gamma \cdot \mathbf{x}$$

其中 $\gamma \in \mathbb{R}^C$ 是可学习向量，初始化为 1e-6。

**这解决了什么问题？** Transformer 深层训练时，残差分支的输出可能会越来越大有，导致训练不稳定。Layer Scale 用一个接近零的初始值抑制残差分支的早期输出，让训练从"近似恒等映射"开始。随着训练进行，$\gamma$ 自适应地决定每个通道的贡献强度。

这等价于 ResNet 初始化的一个变体——在残差路径上加一个极小的缩放因子，然后在训练过程中自动调节。Swin V2 和 CaiT 都使用了类似的技术。

## 八、复杂度、效率与可扩展性分析

### 8.1 计算复杂度对比

| 模型 | FLOPs | params | 图像吞吐量 (img/s, 单卡 V100) |
|------|-------|--------|-------------------------------|
| ResNet-50 | 4.1G | 25M | 1226 |
| Swin-T | 4.5G | 28M | 674 |
| ConvNeXt-T | 4.5G | 29M | 780 |

**关键发现：** ConvNeXt-T 的吞吐量比 Swin-T 高约 16%。这是因为：

1. Depthwise Conv 的内存访问模式比窗口注意力更连续（不需要 gather 操作）
2. 不需要窗口移位（cyclic shift）和相对位置编码的计算
3. 不需要 attention mask 的生成和计算

但 ConvNeXt 的吞吐量仍然显著低于 ResNet-50，主要原因是 Depthwise Conv 的硬件利用率不如标准 conv——GPU 上标准 conv 的 Tensor Core 利用率接近 90%，而 depthwise conv 受限于内存带宽（每个通道的计算量太小，无法充分利用 Tensor Core）。

### 8.2 复杂度公式

**Swin-T 窗口注意力：**

$$\text{FLOP}_{\text{window-attention}} = N \cdot (4 \cdot C \cdot M^2 + 2 \cdot (M^2)^2 \cdot C)$$

其中 $N$ 是 token 总数，$M$ 是窗口大小（7），$C$ 是通道数。

第一项 $4CM^2$ 是 QKV 投影，第二项 $2M^4C$ 是 attention 矩阵计算（M² 个 token 的两两注意力 + 对 value 的加权求和）。

**ConvNeXt depthwise conv：**

$$\text{FLOP}_{\text{dwconv}} = K^2 \cdot C \cdot H \cdot W = K^2 \cdot C \cdot N$$

其中 $K=7$, $N=H\cdot W$。

**当 K=M=7 时的复杂度对比：**

Swin-T: $O(N \cdot 49 \cdot C + N \cdot 2401 \cdot C) = O(2450 \cdot N \cdot C)$
ConvNeXt: $O(49 \cdot C \cdot N)$

Swin-T 的 attention 部分比 ConvNeXt 的 depthwise conv 重约 50 倍（2450 vs 49）。但 Swin-T 的绝大多数 FLOPs 在 FFN 部分（占据约 80% 的 FLOPs），卷积和注意力的 FLOPs 占比都很小。所以整体 FLOPs 相近，但实际延迟差异来自内存访问模式而非计算量。

### 8.3 可扩展性

ConvNeXt 的可扩展性体现在三个维度：

1. **处理高分辨率输入**：Depthwise conv 的复杂度是 $O(N)$，与 token 数线性相关。全局注意力是 $O(N^2)$。在检测任务中（800×1333 输入），ConvNeXt 可以维持类似于低分辨率下的效率。

2. **层次结构**：4 级特征金字塔（1/4, 1/8, 1/16, 1/32 分辨率），可以直接接入 FPN、Mask R-CNN、UperNet 等检测/分割框架，不需要额外适配。

3. **模型家族**：从 ConvNeXt-T（29M）到 ConvNeXt-XL（350M），参数规模跨越了 12 倍，且所有规模下性能曲线都平滑增长，没有出现饱和。

### 8.4 效率代价

ConvNeXt 的高效率有一个代价：**Depthwise Conv 在 GPU 上的利用率不如标准 conv**。这是因为 depthwise conv 不适合 Tensor Core 的矩阵乘法加速（它的 FLOPs 太低，受内存带宽瓶颈限制）。

在推理场景中，这可以通过 kernel fusion（将 depthwise conv 和后续的 1×1 conv 融合为一个操作）来缓解，但在训练场景中仍然是实际瓶颈。

### 10.3 核心代码分析

**convnext.py** 中主要的类和函数：

```python
class Block(nn.Module):
    """ConvNeXt Block 的实现"""
    def __init__(self, dim, drop_path=0., layer_scale_init_value=1e-6):
        super().__init__()
        # 7×7 depthwise conv
        self.dwconv = nn.Conv2d(dim, dim, kernel_size=7, padding=3, groups=dim)
        # LN + Inverted Bottleneck
        self.norm = LayerNorm(dim, eps=1e-6)
        self.pwconv1 = nn.Linear(dim, 4 * dim)  # 扩展通道
        self.act = nn.GELU()
        self.pwconv2 = nn.Linear(4 * dim, dim)  # 压缩通道
        # Layer Scale
        self.gamma = nn.Parameter(
            layer_scale_init_value * torch.ones((dim)),
            requires_grad=True
        ) if layer_scale_init_value > 0 else None
        self.drop_path = DropPath(drop_path) if drop_path > 0. else nn.Identity()

    def forward(self, x):
        input = x
        x = self.dwconv(x)
        x = x.permute(0, 2, 3, 1)  # (N, C, H, W) → (N, H, W, C)
        x = self.norm(x)
        x = self.pwconv1(x)
        x = self.act(x)
        x = self.pwconv2(x)
        if self.gamma is not None:
            x = self.gamma * x
        x = x.permute(0, 3, 1, 2)  # (N, H, W, C) → (N, C, H, W)
        x = input + self.drop_path(x)
        return x
```

**实现细节和论文的对应对照：**

| 论文概念 | 代码实现 |
|----------|----------|
| Depthwise Conv 7×7 | `nn.Conv2d(dim, dim, 7, padding=3, groups=dim)` |
| Inverted Bottleneck 扩展 | `nn.Linear(dim, 4*dim)` |
| GELU 激活 | `nn.GELU()` |
| Inverted Bottleneck 压缩 | `nn.Linear(4*dim, dim)` |
| Layer Scale | `nn.Parameter(1e-6 * ones(dim))` |
| Drop Path | `DropPath(drop_path)` |
| LayerNorm (C, H, W→H, W, C 后) | `LayerNorm(dim)` |

```python
class ConvNeXt(nn.Module):
    """完整的 ConvNeXt 模型"""
    def __init__(self, in_chans=3, num_classes=1000,
                 depths=[3, 3, 9, 3], dims=[96, 192, 384, 768],
                 drop_path_rate=0., layer_scale_init_value=1e-6):
        super().__init__()

        self.downsample_layers = nn.ModuleList()
        # Stem: 4×4 conv, stride 4 + LN
        stem = nn.Sequential(
            nn.Conv2d(in_chans, dims[0], kernel_size=4, stride=4),
            LayerNorm(dims[0], eps=1e-6, data_format="channels_first")
        )
        self.downsample_layers.append(stem)

        # 3 个下采样层: LN → 2×2 stride=2 conv
        for i in range(3):
            downsample_layer = nn.Sequential(
                LayerNorm(dims[i], eps=1e-6, data_format="channels_first"),
                nn.Conv2d(dims[i], dims[i+1], kernel_size=2, stride=2),
            )
            self.downsample_layers.append(downsample_layer)

        # 4 个 stage
        self.stages = nn.ModuleList()
        dp_rates = [x.item() for x in torch.linspace(0, drop_path_rate, sum(depths))]
        cur = 0
        for i in range(4):
            stage = nn.Sequential(
                *[Block(dim=dims[i],
                        drop_path=dp_rates[cur + j],
                        layer_scale_init_value=layer_scale_init_value)
                  for j in range(depths[i])]
            )
            self.stages.append(stage)
            cur += depths[i]

        self.norm = nn.LayerNorm(dims[-1], eps=1e-6)
        self.head = nn.Linear(dims[-1], num_classes)

    def forward_features(self, x):
        for i in range(4):
            x = self.downsample_layers[i](x)
            x = self.stages[i](x)
        return self.norm(x.mean([-2, -1]))  # GAP + LN

    def forward(self, x):
        x = self.forward_features(x)
        x = self.head(x)
        return x
```

**代码中的三个设计细节值得注意：**

1. **LayerNorm 有两种 data format**：`channels_first` 用于下采样层（保持 (B,C,H,W) 格式），默认 `channels_last` 用于 block 内部（(B,H,W,C) 格式）。这是因为 LN 需要在 channel 维度上归一化，而 conv 操作效率最高的是 channels_first 格式。

2. **Drop Path 线性增加**：从 0 到 `drop_path_rate`（默认 0.1），按 block 在总 block 列表中的位置线性插值。早期 stage 的正则化更强，后期 stage 更弱。

3. **Layer Scale 初始化为 1e-6**：这个极小值让残差分支在训练初期几乎不起作用，起到类似"warmup"的效果。随着训练进行，gamma 逐渐增大。

## 十三、范式意义

### 13.1 这到底改变了什么？

ConvNeXt 的范式贡献不在于发明了什么，而在于 **重构了一个话语权**。

在 2021 年，"ConvNet 过时了、Transformer 更好"是主导叙事。谁想用一个 ConvNet backbone，都必须解释"为什么不直接用 Swin"。ConvNeXt 改变了这个叙事：

> 它证明了：ConvNet 不是过时了，而是被用错了（训练方式）和被设计得不合适了（架构细节）。

这个改变的影响有多大？它让整个视觉 backbone 设计空间重新开放了。不再只有"用 Transformer"或"用 CNN"的二元选择，而是"借鉴 Transformer 的设计原则来改进 ConvNet"——以及反过来，"从 ConvNet 的视角审视 Transformer 设计中的冗余"。

### 13.2 贡献层级分类

ConvNeXt 的贡献可以归类为：

- **架构改进**（Architecture improvement）：将 ResNet 升级为可以匹敌 Transformer 的现代化 ConvNet
- **训练策略改进**（Training strategy improvement）：系统性地证明了现代训练技术对 ConvNet 的有效性
- **范式澄清**（Paradigm clarification）：揭示了 ConvNet 和 Transformer 性能差距的真实来源

它不属于范式转变（Paradigm shift），因为它没有引入新的计算原语，没有改变视觉 backbone 的基本范式。但它是一个重要的 **bridge node**（桥梁节点）——它连接了 CNN 和 Transformer 两条此前被认为互斥的技术路线。

### 13.3 它的核心洞见

> ConvNet 和 Transformer 不是对立的。把 Transformer 中有效的设计选择（更好的 stage 比例、inverted bottleneck、大卷积核、现代训练技术）迁移回 ConvNet，ConvNet 可以做得更好。

这个洞见看起来简单，但它挑战了当时的主流认知：即 Transformer 的成功归因于注意力机制本身的优越性。ConvNeXt 的实验证据表明，注意力机制可能并不比卷积"更好"，它只是在一个更好的整体设计框架中被使用的。

### 13.4 它在上游/下游任务上的意义

对于下游任务（检测、分割），ConvNeXt 提供了一个可以直接替换 ResNet/Swin 的高性能 backbone。因为：

1. 它保持全卷积结构，与现有检测/分割框架无缝对接
2. 它的层级特征金字塔与 FPN 完美契合
3. 它的推理效率高于 Swin

## 十四、局限性与后续方向

### 14.1 缺少理论解释

论文没有深度解释为什么 Depthwise Conv + Inverted Bottleneck 的组合优于 Self-Attention + FFN。它只是通过消融实验展示了"它更好"，但没有回答"为什么更好"。

可能的解释是：Depthwise Conv 的权重是静态的（在推理时固定），而 Self-Attention 的注意力权重是动态的（取决于输入对）。在数据量有限的场景下（ImageNet-1K），静态权重通过更多 FLOPs/epoch 的学习可能比动态权重更有效。但这只是一个猜测，论文没有涉及。

### 14.2 训练成本

ConvNeXt 使用了非常重的训练设置（300 epoch + 强数据增强 + 正则化）。这个训练成本（约 10 × 传统的 ResNet 训练）是否在所有场景下都值得，需要根据应用场景判断。

### 14.3 未验证大规模数据场景

论文没有测试 ConvNeXt 在 JFT-300M 或 Instagram 数据等大规模数据集上的表现。在数据量极大时（几亿张图片），Transformer 的全局注意力可能会展现出比 ConvNet 更强的表达能力（这是 ViT 论文的核心结论之一）。所以 ConvNeXt 在极大规模场景下的竞争力仍需验证。

### 14.4 Depthwise Conv 的硬件效率问题

虽然 Depthwise Conv 的理论计算量低，但在实际 GPU 上，它的吞吐量远低于标准卷积。这是因为：

1. Depthwise Conv 的 compute-to-memory-access 比率很低
2. Tensor Core 无法加速 Depthwise Conv（因为它本质上是逐通道操作）
3. Depthwise Conv 的内存访问模式不规则（非连续访问）

在移动端和边缘设备上，这个效率问题更明显。

### 14.5 视频任务未验证

论文没有测试 ConvNeXt 在视频任务上的表现。视频领域的时序建模可能仍然需要注意力机制来捕获帧间依赖关系。

### 14.6 后续方向

1. **ConvNeXt V2**（2023）：将 MAE 的自监督框架适配到 ConvNeXt，解决了 CNN 做 MIM（Masked Image Modeling）时遇到的架构不兼容问题
2. **更高效的 Depthwise Conv 实现**：通过 kernel fusion、depthwise conv 专用优化等方式提升硬件效率
3. **更大尺度验证**：在大规模数据上验证 ConvNeXt 的可扩展性是否真的可以和 Transformer 竞争
4. **与其他模态结合**：ConvNeXt 是否可以替代 CLIP 中的 ViT 做多模态学习？

## 十五、小结

ConvNeXt 的价值不在于它提出了多么新颖的模块，而在于它做了视觉社区最需要的一件事：**系统性地分离了训练技术和架构设计对性能的贡献**。

直白说，这篇论文的核心贡献不是 ConvNeXt 这个架构本身，而是它回答了一个更根本的问题：

> 如果你用 2022 年的训练技术去训练一个 2015 年的架构（ResNet），再用 2022 年的架构设计原则去改进它，它能不能打败 2022 年的 Transformer（Swin）？

答案是能。ConvNeXt 在所有模型规模和所有任务上都超过了 Swin。

这告诉我们两件事：
1. **训练技术是性能差异的主要来源**，而非架构本身
2. **局部归纳偏置在视觉任务上仍然有价值**——卷积不一定比注意力差

在技术树上，ConvNeXt 是一个 bridge node。它不是一个新范式的开端，但它拆掉了一堵墙——那堵人们以为隔在 ConvNet 和 Transformer 之间的墙。拆掉之后，两条路线开始互相借鉴、融合发展，这可能是它更深远的意义。
