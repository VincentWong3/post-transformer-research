## 一、先搞清楚"问题在哪里"

2020–2021 年，视觉 Transformer 这条路走出了一条清晰的轨迹。先是 ViT 证明纯 Transformer 架构可以在大规模数据上追上乃至超越 CNN，然后是 DeiT 证明通过知识蒸馏和数据增强，可以从头在 ImageNet 上训练 ViT。看起来 Transformer 作为视觉通用 backbone 这件事似乎稳了。

但这时候出现了一个尴尬的局面：**Transformer 确实准，但它跑得慢。**

不是说 FLOPs 高——实际上 ViT-B/16 的 FLOPs（17.6G）比 ResNet-152（11.3G）高一截，但准多了。问题是**实际硬件的推理速度**。ViT-B/16 在 GPU 上推理速度是 429 img/s，而 ResNet-50 能做到 1227 img/s。差距近 3 倍。

更别提 CPU 了。对于很多实际部署场景——数据中心 feature extraction、移动端推理、边缘设备——GPU 不是唯一的平台。在这些场景里，CNN 一直是无冕之王，因为卷积运算可以被深度优化：BatchNorm 融合、INT8 量化、MKL 加速，大量工程积累。

而 Transformer 呢？它的核心运算是大矩阵乘法和 softmax，前者对 GPU 友好，但对 CPU 不太友好；后者完全是内存带宽瓶颈。GELU 激活函数要算 erf，LatyerNorm 没法融合到前一层——这些看似微小的代价在推理时累积起来就成了大问题。

所以 2021 年初的现实是：**Transformer 在精度上能打过 CNN，但在"实际跑得快"这件事上吃力得很。** 尤其当目标是在中等规模模型（10M–40M 参数）的区间内做速度-精度权衡时，纯 Transformer 设计（ViT/DeiT）并不是最优解。

这就引出了 LeViT 要回答的问题：**能不能把 CNN 那些"在被 GPU 喂大前就积累了二十年的工程技巧"搬进 Transformer 里，造一个既准又快的混合架构？**

## 二、旧路线的真正问题

### 2.1 ViT 的"慢"不是偶然的

先说一个反直觉的事实：ViT 的 FLOPs 比 ResNet-50 高，但它的计算结构其实对 GPU 更友好——因为 Transformer 的核心计算（矩阵乘法）在 GPU 上的计算密度远高于卷积步。所以按理说 Transformer 应该更快才对。

那为什么实际测出来反而慢？问题出在几个细节上：

**第一，GELU 激活函数。** 这不是一个简单操作。GELU 要计算高斯误差函数 $erf(x)$，这个超越函数在小批量、低维度时可能和卷积本身一样耗时。DeiT 论文自己承认在 DeiT-Tiny 上，如果不用 GELU 而用 Hardswish，MLP 占用的推理时间可以显著降低。

**第二，LayerNorm。** ViT 在注意力和 MLP 子层之前都做 LayerNorm，但 LayerNorm 需要为每个 token 单独计算均值和方差，不能像 BatchNorm 那样在推理时融合到前一层。BatchNorm 在推理时可以被吸收进卷积或线性层的权重里，不产生额外开销。

**第三，位置编码。** ViT 的可学习位置编码只加在输入层，但在深层网络中，位置信息可能会在后续计算中被稀释。有消融实验表明去掉位置编码会导致精度大幅下降——但这意味着位置信息占用了表征容量。

**第四，固定序列长度。** ViT 对所有层使用相同的序列长度（例如 196 个 token）。这意味着即使到了深层，token 数依然不减。而 CNN 的典型做法是逐步降低空间分辨率、增加通道数——深层 token 包含冗余空间信息，计算效率低。

### 2.2 为什么 CNN 的"多尺度金字塔"重要

ResNet 的一个核心设计是用 4 个 stage 逐步降低分辨率：输入从 224×224 下采样到 112×112、56×56、28×28、14×14、7×7。每个 stage 的特征图分辨率不同，通道数递增。

这种设计的实际好处是：**计算量主要集中在低分辨率的深层，而不是高分辨率的浅层。** 早期层虽然分辨率高但通道数少，后期层通道数多但分辨率低。整体 FLOPs 被摊平，不会出现某个阶段的计算量爆炸。

而 ViT 的 transformer 块对所有 token 一视同仁地做全局注意力——无论这些 token 来自浅层还是深层。深层 token 已经包含高级语义信息，不需要再以同样成本处理所有空间位置。

### 2.3 更致命的问题：高分辨率输入让 ViT 崩掉

ViT 的标准输入是 224×224（patch size 16），token 数为 196。全局注意力的复杂度是 $O(N^2)$，196² = 38,416 个 attention pair——还行。

但如果输入是 384×384，token 数变成 576，576² = 331,776——接近 9 倍。如果做检测任务用到 800×1333 的输入，token 数超过 4000——attention 矩阵有 1600 万个元素，一块 V100 的显存也装不下。

所以 ViT 的一个根本痛点是：**它被设计为定长序列的处理器，但图像天生是多尺度的。** 不解决这个问题，Transformer 就只能在分类这种固定分辨率任务上生存。

## 三、这篇论文的核心思路

> **这篇论文的核心思路是：把 CNN 的多尺度金字塔结构嫁接到 Transformer 上，再用注意力下采样替代池化做跨阶段衔接。——用 ConvNet 的骨架来穿 Transformer 的衣服。**

这句话不是随便说的。LeViT 不是简单地在 CNN backbone 后面接 Transformer（像 ViT 那样用 ResNet 做 feature extractor 再喂给 transformer 层），而是从 stem 开始就用卷积做快速下采样，然后每个 stage 内部用 Transformer 块做特征提取，stage 之间用"下采样注意力"做分辨率缩减。

三个关键设计：

1. **Convolutional stem**：用 4 层 stride-2 的 3×3 卷积替代 ViT 的 16×16 patch embedding。快速把 224×224 图像压缩到 14×14 分辨率，同时保留足够信息。

2. **下采样注意力（shrinking attention）**：stage 之间的过渡不使用池化，而是把下采样后的 token 做 query、原始 token 做 key 和 value 来计算注意力——既降低了分辨率，又保持了内容相关的动态特征聚合。

3. **注意力偏置（attention bias）**：在每个注意力头中引入可学习的相对位置偏置，替代 ViT 的绝对位置编码。参数更少，而且隐式引入了平移不变性——这在视觉任务里天然占优。

用一句话概括 LeViT 的设计哲学：**CNN 给了它骨架和速度，Transformer 给了它内容和精度。**

这是 LeViT-256 的架构（注：论文中 LeViT-256 的 stem 输出是 256×14×14，总 FLOPs 为 1120M）。不同变体的 stage 数、head 数和通道数不同，但整体拓扑一致。

### 4.2 前向过程详解

以 LeViT-256 为例，batch size = 1，输入 3×224×224。

**Step 1: CNN Stem**

用 4 层 3×3 卷积（stride 2）把输入快速下采样到 14×14，总共 16 倍下采样（和 ViT 的 patch size 16 等价），但用了更细粒度的逐步下采样。每层后接 BatchNorm 和 Hardswish。

**Step 2: Stage 1（14×14）**

把 256×14×14 的特征图视为序列，即 196 个 token，每个 256 维。通过 4 个 LeViT Block：每个 block 包含一个多头注意力子层（4 head）和一个 MLP（扩展因子 2），均使用预归一化的 BatchNorm + 残差连接。

**Step 3: Shrinking Attention（14×14 → 7×7）**

这是关键模块。输入 256×14×14，先对特征图做 2×2 平均池化得到 256×7×7（下采样），然后这个下采样后的特征图在注意力计算中充当 **query**。原始的 256×14×14 特征图同时作为 **key** 和 **value**。

因为 query 只有 49 个 token，而 key/value 有 196 个，所以注意力权重矩阵大小是 49×196，而不是 196×196——计算量骤降。同时由于这层没有残差连接，为了防止信息丢失，value 的通道数扩展到 4D（D 是 key 维度）。

输出 384×7×7，通道数从 256 增到 384。

**Step 4: Stage 2（7×7）**

4 个 LeViT Block，384 通道，6 head。

**Step 5: Shrinking Attention（7×7 → 4×4）**

同理，query 来自下采样后的 384×4×4，key/value 来自 384×7×7。输出 512×4×4。

**Step 6: Stage 3（4×4）**

4 个 LeViT Block，512 通道，8 head。

**Step 7: 分类头**

全局平均池化（GAP）把 4×4×512 → 512 维向量 → 线性层 → 1000 维分类 logits。

这里有个微妙的设计：LeViT 不使用 ViT 的 class token。因为 GAP 在推理时不需要额外的 token 处理和位置信息，而且当输入分辨率变化时，GAP 天然适应不同大小的 feature map。

### 4.3 每个设计的"为什么"

**为什么用 BN 而不是 LN？**
BN 在推理时可以和前面的卷积或 1×1 卷积融合，零额外开销。LN 不行。论文实测用 LN 替代 BN 后推理速度下降约 20%。

**为什么用 Hardswish 而不是 GELU？**
Hardswish 只需要 ReLU6 和简单乘除，GELU 需要 erf（超越函数）。实测 GELU → Hardswish 只掉 0.1% 精度，但推理速度快约 15%。

**为什么 MLP 扩展因子是 2 而不是 4？**
ViT 的 MLP 扩展因子 4 是其计算量的主要来源（约 2/3 的 FLOPs）。LeViT 砍到 2，用更宽的网络（更大 C）和更强的注意力来补偿。消融实验显示 4× 只提升 0.3% 精度，但速度下降 22%。

**为什么不用 class token？**
Class token 是 ViT 的遗产；它在推理时需要一个额外的可学习参数，且对 GAP 没有优势。LeViT 用 GAP 替代，训练时用双头（分类头 + 蒸馏头）策略。

## 五、核心模块逐个解释

### 5.1 下采样注意力（Shrinking Attention / Downsampling Attention）

**解决的问题：** ViT 的固定序列长度导致深层仍然处理大量冗余 token，计算效率低。

**为什么需要它：** CNN 用池化或 stride 卷积做下采样，但这些操作是固定的（不是内容相关）。如果在下采样的同时也用注意力来聚合信息，既降低分辨率又保留动态权重的好处。

**工作原理：**

```
输入: C×H×W
     │
     ▼ 对输入做平均池化（2×2, stride 2）
  query: C×H/2×W/2  （下采样后的特征图，充当 Q）
     │
  key  : C×H×W       （原始特征图，充当 K）
  value: 4C×H×W      （原始特征图经过线性投影，通道扩展为 4 倍）
     │
     ▼
  Attention(Q, K, V) = softmax(QK^T/√d) · V
     │
     ▼
  输出: 4C×H/2×W/2   （分辨率减半，通道翻倍 ∝ 4C）
```

注意这里 Q 和 K 都只包含前 C 维（用 D 表示 key 维度），而 V 扩展到 4D。这种不对称设计是故意的：key 只需要编码位置信息（smaller keys），而 value 需要保留内容信息。

**为什么没有残差连接？** 因为输出和输入的分辨率不同、通道数不同，无法直接相加。去掉残差连接换取速度提升，通过增加 head 数来补偿（shrinking attention 的 head 数 = C/D，是普通 block 的两倍）。

### 5.2 注意力偏置（Attention Bias）

**解决的问题：** ViT 的绝对位置编码只加在输入层，在深层可能被稀释；且位置编码参数随输入分辨率变化需要插值。

**工作原理：**

在标准点积注意力 $QK^T$ 的基础上，对每个 query-key 对加一个可学习的偏置：

$$A_{(x,y),(x',y')}^h = Q_{(x,y)}\cdot K_{(x',y')} + B_{|x-x'|,|y-y'|}^h$$

这里 $B^h$ 是一个 $(2H-1) \times (2W-1)$ 的矩阵，每个 head h 各有一个，参数仅依赖于相对位移 $(|Δx|, |Δy|)$，与绝对位置无关。

**为什么有效：**
- **平移不变性**：偏置只依赖相对坐标，天然适合视觉任务（物体在不同位置应该被同样处理）
- **参数效率**：14×14 特征图的偏置矩阵只有 27×27=729 个参数 per head，而绝对位置编码要 196×196=38,416 维
- **注入位置信息的时机更好**：在每个注意力层都加，而不是只在输入层加一次

消融实验显示去掉 attention bias 会导致约 0.8% 的精度下降。

### 5.3 激活函数与归一化的选择

LeViT 的块结构如下：

```
输入 C×H×W
     │
     ▼
  BatchNorm（pre-norm）
     │
     ▼
  [多头注意力 / 下采样注意力]
     │
     ▼ 残差连接 +
  BatchNorm（pre-norm）
     │
     ▼
  Conv 1×1 (C → 2C) → Hardswish → Conv 1×1 (2C → C)
     │
     ▼ 残差连接 +
  输出 C×H×W
```

关键选择：
- **BN 替代 LN**：推理时融合，不增加开销
- **Hardswish 替代 GELU**：速度快，精度损失极小
- **Zero-init BN**：每个残差连接的 BN 的 γ 初始化为 0，确保训练初期残差连接不生效，稳定训练。这是 [Goyal et al., 2017] 在大 batch size 训练论文中的技巧。

## 六、核心公式与数学直觉

### 公式 1：带偏置的缩放点积注意力

$$A_{(x,y),(x',y')}^h = Q_{(x,y)}\cdot K_{(x',y')} + B_{|x-x'|,|y-y'|}^h$$

**意义：** 把"两个像素在空间上的距离"作为一个可学习的偏移量加入注意力得分，相当于告诉模型"相邻像素应更相关"这个先验。

**直觉：** 左边 $QK^T$ 是根据内容计算的相似度（"这两个像素的视觉特征像不像"），右边 $B$ 是固定的空间偏好（"这两个像素是不是靠在一起"）。两者相加决定了最终的注意力权重。模型可以通过调整 $B$ 来学会"这个 head 应该只看局部"或"这个 head 应该看全局"。

### 公式 2：Hardswish

$$\text{Hardswish}(x) = x \cdot \frac{\text{ReLU6}(x+3)}{6}$$

**意义：** Swish 的分段线性近似，去掉 erf 计算换速度。

**直觉：**
- 当 $x < -3$：输出 0（饱和）
- 当 $-3 ≤ x ≤ 3$：输出非线性（$x(x+3)/6$）
- 当 $x > 3$：输出 $x$（线性）

相比 ReLU（负半轴完全为 0），Hardswish 在小负值区域有软的饱和，这有助于训练稳定性。相比 GELU，它只需要简单算术指令，在 CPU 和推理芯片上快得多。

### 公式 3：下采样注意力的复杂度

两层之间的下采样注意力（shrinking attention），设原始 token 数为 N，下采样后为 M（通常 M = N/4），key 维度 D：

- 注意力权重计算 $QK^T$：$O(M \cdot N \cdot D)$
- 加权聚合 $AV$：$O(M \cdot N \cdot 4D)$
- 总复杂度：$O(5MND)$

而如果使用标准的全局注意力（不做下采样），复杂度是 $O(N^2 D + N^2 \cdot 4D) = O(5N^2 D)$。

因为 M = N/4，下采样注意力节省了 4 倍的计算量。关键在于这个节省**是在保持注意力机制的核心优势（内容相关的特征聚合）下实现的**，而不是简单地用池化替代注意力。

## 八、复杂度、效率与可扩展性分析

### 8.1 计算量与序列长度的关系

LeViT 的核心效率优势来自金字塔架构：

| Stage | 序列长度 | 通道数 | FLOPs 占比 |
|-------|---------|-------|-----------|
| Stem | — | — | ~16% |
| Stage 1 (14×14) | 196 | 256 | ~45% |
| Stage 2 (7×7) | 49 | 384 | ~25% |
| Stage 3 (4×4) | 16 | 512 | ~14% |

有意思的是，Stage 1（最大分辨率）占了最多的 FLOPs，不是因为注意力贵，而是因为 MLP（1×1 卷积）在 token 多的时候计算量大。随着分辨率降低，MLP 的计算量也急剧下降。

### 8.2 与 DeiT 的对比

DeiT-Tiny（C=192, 3 heads）和 LeViT-256（C=256, 4 heads）在 14×14 分辨率下，LeViT 更宽但总耗时接近（2474 μs vs 2365 μs）。关键差异：

| 组件 | DeiT-Tiny | LeViT-256 |
|------|-----------|-----------|
| LayerNorm | 49 μs | —（BN 融合后为零） |
| Q/K 投影 | 299 μs | 275 μs |
| $QK^T$ | 228 μs | 159 μs |
| AV 聚合 | 161 μs | 206 μs |
| 注意力投影 | 175 μs | 310 μs |
| MLP | 1390 μs | 1140 μs |
| **总计** | **2474 μs** | **2365 μs** |

数据来自论文表 S1（Supplementary）。注意 LeViT 虽然更宽（C=256 vs 192），但总耗时反而略低，主要节省来自 MLP 的 2× 扩展因子（vs DeiT 的 4×）。

### 8.3 实际硬件速度

LeViT 相比 EfficientNet 和 DeiT 在不同硬件上的速度优势：

| 类别 | 模型 | GPU (img/s) | Intel CPU (img/s) | ARM CPU (img/s) |
|------|------|:--------:|:--------------:|:--------------:|
| 最快速 | LeViT-128S | 12880 | 131.1 | 39.1 |
| 同精度 | EfficientNet B0 | 4754 | 30.1 | 3.5 |
| 中速 | LeViT-256 | 6582 | 42.5 | 16.4 |
| 同精度 | DeiT-Tiny | 3973 | 39.1 | 16.8 |

关键数字：**LeViT-256 在 GPU 上比 DeiT-Tiny 快 65%，在 CPU 上差不多。LeViT-128S 在 CPU 上比 EfficientNet B0 快 4.4 倍。**

### 9.2 消融实验

论文表 5 在 LeViT-128S 上做了系统的消融（训练 100 epoch）：

| # | 消融项 | 准确率 | 相对基线差异 |
|---|-------|:-----:|:----------:|
| — | 基线 LeViT-128S | 71.9% | 0 |
| A1 | 去掉金字塔结构 | 56.5% | -15.4% |
| A2 | 去掉 CNN stem（用 16×16 patch） | 65.3% | -6.6% |
| A3 | BN → LN | 66.6% | -5.3% |
| A4 | 去掉蒸馏（no distillation） | 69.7% | -2.2% |
| A5 | 去掉注意力偏置 | 70.4% | -1.5% |
| A6 | 用 DeiT 风格的块（4× MLP, 对称 QKV） | 70.9% | -1.0% |
| A7 | 去掉注意力中的 Hardswish | 71.1% | -0.8% |

A1 是最惊悚的——去掉金字塔后精度狂泻 15%（71.9 → 56.5）。这说明金字塔结构**不是锦上添花的优化，而是 LeViT 的基石**。没有它，网络无法有效控制 FLOPs，被迫在极窄的宽度下工作。

A2 说明了 CNN stem 的必要性——用单个 16×16 patch embedding 替代 4 层 3×3 卷积导致 6.6% 的精度下降。在低容量区间，CNN stem 可以更高效地压缩高维像素输入。

A3 和 A5 表明 BN 和 attention bias 各自贡献约 1.5-5%——因为训练只跑了 100 epoch，差异被放大。全训（1000 epoch）下各组件贡献约 0.5-1%。

### 9.3 跨数据集的泛化

论文还在 ImageNet-Real 和 ImageNet-V2 上做了验证。LeViT-256 在 -Real 上和 EfficientNet B3 持平（86.8%），在 -V2 上略低（70.0% vs 70.6%）。说明 LeViT 的精度优势主要来自蒸馏和训练技巧，而不是在 ImageNet 验证集上过拟合。

### 关键类

**LeViT 模型类**（`models.py`）：

```python
class LeViT(nn.Module):
    def __init__(self, ...):
        # stem: 4层 3×3 conv + BN + Hardswish
        # stages: list of AttentionBlock
        # shrinking attention: SubsampleBlock
        # 分类头: GAP + Linear
    
    def forward(self, x):
        # x: B×3×224×224
        for stage in self.stages:
            x = stage(x)
        x = x.mean(dim=2).mean(dim=2)   # GAP
        x = self.head(x)
        return x
```

**AttentionBlock**（非下采样版本）：

```python
class AttentionBlock(nn.Module):
    def forward(self, x):
        # x: B×C×H×W
        shortcut = x
        x = self.bn1(x)
        x = self.attention(x)    # 多头注意力，含 attention bias
        x = shortcut + x
        shortcut = x
        x = self.bn2(x)
        x = self.linear1(x)      # Conv 1×1, C→2C
        x = self.hardswish(x)
        x = self.linear2(x)      # Conv 1×1, 2C→C
        x = shortcut + x
        return x
```

**Attention bias 的实现**（`models.py` 中的 `attention_bias` 函数）：

```python
# 对每个 head，Bias 是一个 (2H-1)×(2W-1) 的可学习参数
# 在注意力计算时，根据 (Δx, Δy) 索引到偏置值
# 然后加到注意力得分上
```

LeViT 不是第一个想到用卷积做 vision Transformer stem 的。在它之前已有工作：

- **ViT** 自己试过用 ResNet 做 feature extractor + Transformer 层（grafting），但只是作为实验
- **PVT（Pyramid Vision Transformer, Wang et al., 2021）** 提出了类似的金字塔结构，但 PVT 是标准注意力 + 下采样，且主要面向 dense prediction 任务
- **CvT（Wu et al., 2021）** 在注意力中引入了卷积操作（深度可分离卷积做 QKV 投影）

LeViT 的独特之处在于：
1. 它不是简单地把 ViT 改成金字塔，而是**为推理速度做了系统级优化**（BN 融合、Hardswish、smaller keys）
2. 它关注的是**中等规模模型在实际硬件上的吞吐量**，而不是 FLOPs 或参数量
3. 它引入的 attention bias 后来被 Swin Transformer 等后续工作广泛采用

## 十三、范式意义

LeViT 的范式意义可以概括为：**Bridge node（桥梁节点）**。

它不是范式级别的变革——它没有发明注意力、没有发明 Transformer、没有发明知识蒸馏。它做的是**系统化的工程融合**：把 CNN 二十多年积累的工程技巧（BN 融合、金字塔结构、Hardswish 激活、stem 设计）系统性地搬进 Transformer 架构。

从这个角度看，LeViT 回答了一个更深层的问题：**"Transformer 能不能不依赖大规模预训练和大 GPU，而能在实际部署场景里跑得又快又准？"** 答案是"能"，前提是你愿意向 CNN 学到的东西低头。

后来的工作（EfficientViT, MobileViT, FastViT）继承并扩展了这条路线。甚至 ConvNeXt（2022）可以被看作这个方向的一个反向版本——从 ConvNet 的角度吸收 Transformer 的设计（ConvNeXt 本质上是 CNN 化的 Swin Transformer）。

所以 LeViT 的技术角色更像是：
- **不是** ViT（范式变革）
- **不是** DeiT（训练策略优化）
- **是** 让 Transformer "放下身段"接受 CNN 工程学院派的桥梁

## 十四、局限性与后续方向

### 局限性

**1. 对下游任务的适应性有限**

LeViT 被设计为专门做分类的模型。它的 stage 3 输出分辨率只有 4×4，对于需要多尺度特征的检测/分割任务来说太低了。虽然可以接特征金字塔，但 LeViT 的 stage 内部没有像 Swin 那样保留跨窗口交互的能力。这限制了它在 dense prediction 任务上的直接应用。

**2. 依赖蒸馏，训练成本高**

LeViT 的精度在很大程度上依赖 DeiT 风格的知识蒸馏——教师模型是 RegNetY-16GF（更大的网络）。训练本身需要 1000 epoch、32 个 GPU、3–5 天。这个训练成本对于尝试新架构的实验来说是沉重的。

**3. 注意力偏置的灵活性有限**

Attention bias 是固定的（不随输入内容变化），虽然引入了局部先验，但对需要长距离依赖或全局上下文的任务（如视频理解、高分辨率文档分析）可能不够灵活。Swin 的 shifted window 和 PVT 的空间缩减注意力在这一点上提供了更多的灵活性。

**4. FLOPs 分布不均匀**

从论文的 FLOPs 占比看，Stage 1（14×14）占了大约 45% 的 FLOPs。这意味着网络的瓶颈在前端压缩阶段，而不是在更语义化的深层。如果有进一步的优化空间，可能在于 Stage 1 的 MLP 计算。

### 后续方向

EfficientViT (Cai et al., 2022) 和 MobileViT (Mehta & Rastegari, 2022) 直接继承了 LeViT 的混合架构思路，分别在不同方向上做了优化（EfficientViT 进一步简化 MLP，MobileViT 把 Transformer 移到更低分辨率）。FastViT (Vasu et al., 2023) 引入了重参数化技术，在保持 LeViT 风格的同时进一步加速推理。

也许最有趣的后续工作是**ConvNeXt**：它本质上反向吸收了 LeViT 的设计（LayerNorm → BN 的反向替换），从 ConvNet 的角度重新引入 Transformer 的设计要素。ConvNeXt 证明了一个事实：经过 LeViT 的融合实验后，CNN 和 Transformer 之间的设计空间已经基本被打通。

## 十五、小结

LeViT 是一篇"在正确的时间做了正确的事情"的论文。它的价值不只在提出了一个具体架构（虽然 LeViT-256 在当时的精度-速度 Pareto 前沿上确实创造了新纪录），更在于它**系统性地证明了 CNN 和 Transformer 可以在工程层面深度融合**。

直白说，这篇论文做了三件被后续大量验证正确的事：
1. **金字塔结构**对视觉 Transformer 至关重要——后来的 PVT、Swin、MViT 全都内置了
2. **Relative position bias** 比绝对位置编码更适合视觉任务——后续 Swin、ConvNeXt 都在用
3. **工程优化（BN 融合、Hardswish、smaller keys）** 对推理速度的贡献比架构创新更大——这启发了后来的 EfficientViT 等工作去考虑"系统级"效率

LeViT 在技术树上是一个**桥梁节点**：它连接了 CNN 和 Transformer 两条被一度看作竞争关系的路线，证明了这两条路线不是非此即彼，而是可以取长补短。从这个意义上说，LeViT 为后来的"CNN vs Transformer"大融合思潮提供了重要的工程依据和实验佐证。
