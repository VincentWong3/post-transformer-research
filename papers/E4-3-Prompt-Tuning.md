# # The Power of Scale for Parameter-Efficient Prompt Tuning（2021）
**论文：** [arXiv](https://arxiv.org/abs/2104.08691) · Brian Lester, Rami Al-Rfou, Noah Constant · Google Research · 2021

---

## 一、先搞清楚坑在哪

大模型微调（fine-tuning）是 NLP 领域的标准范式：拿一个预训练好的语言模型（如 BERT、T5），在目标任务数据上更新全部参数。这个方法效果很好，但问题也很明显。

**核心矛盾：每个任务都需要保存一份完整的模型副本。**

假设你有一个 11B 参数的 T5 模型，要为 100 个不同任务做微调。每个任务保存一个完整的 11B 模型副本，那就是 1.1T 参数——光存储成本就高得吓人。更不用说训练时每个任务都需要独立的 GPU 资源。

过去几年，业界尝试了各种"参数高效微调"方法，核心思想是：**只更新一小部分参数，冻结大部分预训练权重。** 比如：

- **Adapter 系列** [Houlsby et al., 2019]：在 Transformer 层之间插入小型的"适配器"模块
- **Diff pruning** [Guo et al., 2021]：学习一个稀疏的增量参数
- **Prefix tuning** [Li & Liang, 2021]：在输入序列前插入可学习的"前缀"向量

这些方法确实减少了参数数量，但引入了一个新问题：**改变了模型架构**。Adapter 增加了新的层，Prefix tuning 修改了注意力机制的计算方式。这意味着你不能直接复用预训练模型的前向传播代码，需要为每个方法定制实现。

## 二、[existing approach] 的真正问题

### 2.1 架构侵入性

Adapter 需要在 Transformer 层的 self-attention 和 FFN 之后插入 Bottleneck 层。这听起来简单，但实际实现时：

- 需要修改模型的前向传播代码
- 需要额外的层初始化逻辑
- 推理时不能直接复用原始模型的权重文件

### 2.2 推理延迟

Adapter 增加了额外的计算步骤。虽然每个 Adapter 的计算量不大，但在大模型中叠加起来，推理延迟会增加 20-50% [Pfeiffer et al., 2020]。

### 2.3 任务间的干扰

Prefix tuning 把可学习向量插入到注意力机制的 K 和 V 中。这改变了 attention 的计算方式——每个 token 的注意力分布都会受到 prefix 的影响。不同的任务可能需要不同的 prefix 长度，导致模型输入序列长度不统一。

### 2.4 参数效率的"天花板"

Diff pruning 虽然理论上可以做到极高稀疏度（99% 以上的参数为 0），但实际训练时仍然需要维护全量参数。内存占用并没有减少，只是在存储时更省空间。

## 三、Prompt Tuning 的核心思路

**核心思想：不要修改模型，而是修改输入。**

Prompt Tuning 的做法极其简单：在输入序列前面拼接一段可学习的"软提示"（soft prompt）向量，然后冻结整个预训练模型，只更新这些提示向量。

```
原始输入: [CLS] 这部电影真好看 [SEP]
Prompt Tuning: [P1] [P2] [P3] [P4] [P5] [CLS] 这部电影真好看 [SEP]
```

其中 `[P1]` 到 `[P5]` 是可学习的嵌入向量，每个向量的维度等于模型的嵌入维度（例如 T5 的 512 维）。模型的其他参数全部冻结。

**为什么这能工作？**

直觉上，这些可学习的提示向量扮演了"任务指令"的角色。预训练模型已经见过海量的文本模式，提示向量相当于告诉模型："接下来你要做的是情感分类任务，注意关注情感相关的表达。"

**关键洞察：提示向量不需要是人类可读的 token。** 它们是在连续空间中优化的向量，可以编码比离散 token 更丰富的任务信息。

## 四、方法细节

### 4.1 形式化定义

给定一个预训练语言模型 $P(y|x)$，其中 $x$ 是输入文本，$y$ 是输出。

传统 fine-tuning 的目标是：
$$\max_{\theta} \log P(y|x; \theta)$$

其中 $\theta$ 是模型的所有参数。

Prompt Tuning 的目标是：
$$\max_{\theta_P} \log P(y|[P; x]; \theta_M)$$

其中：
- $\theta_P$ 是提示向量的参数（可学习）
- $\theta_M$ 是预训练模型的参数（冻结）
- $[P; x]$ 表示将提示向量拼接在输入前面

### 4.2 提示向量的初始化

论文实验了三种初始化方式：

1. **随机初始化**：从均匀分布 $U(-0.5, 0.5)$ 采样
2. **词汇表初始化**：从预训练模型的词汇表中随机选择 $k$ 个 token 的嵌入向量作为初始值
3. **类标签初始化**：对于分类任务，使用对应类别名称的 token 嵌入作为提示向量的一部分

实验发现，词汇表初始化和类标签初始化在模型规模较小时效果更好，但随着模型规模增大，随机初始化也能达到相当的效果。

### 4.3 提示长度

提示长度 $k$ 是一个超参数。论文实验了 1 到 150 个 token 的长度。

- 太短（< 5）：表达能力不足，无法捕获任务的全部信息
- 太长（> 100）：边际收益递减，且增加了输入序列长度

**最佳实践：** 对于大多数任务，20-50 个提示 token 就足够了。

### 4.4 与 Prefix Tuning 的区别

Prompt Tuning 和 Prefix Tuning 看起来很像，但有本质区别：

| 方法 | 插入位置 | 修改的模型部分 | 参数量 |
|------|----------|----------------|--------|
| Prefix Tuning | 每一层的 K 和 V | 修改 attention 计算 | 层数 × 维度 × 2 |
| Prompt Tuning | 输入序列开头 | 不修改模型 | 提示长度 × 维度 |

Prefix Tuning 的参数量与模型层数成正比，而 Prompt Tuning 的参数量与层数无关。对于 11B 的 T5，Prefix Tuning 需要约 20M 参数，而 Prompt Tuning 只需要约 50K 参数（提示长度 100，维度 512）。

## 五、核心创新点

### 5.1 真正的"参数高效"

**（a）痛点与动机**

之前的参数高效方法虽然减少了可训练参数，但要么增加了推理延迟（Adapter），要么需要修改模型架构（Prefix Tuning）。Prompt Tuning 的目标是：**在不改变模型架构的前提下，实现极致的参数效率。**

**（b）方案细节**

- 可训练参数：只有提示向量，数量为 $k \times d$，其中 $k$ 是提示长度，$d$ 是嵌入维度
- 对于 T5-XXL（11B 参数），使用 $k=100$ 时，可训练参数仅为 100 × 512 = 51,200
- 参数效率：**0.0005%**（51,200 / 11,000,000,000）

**（c）为什么有效**

这得益于大模型的"任务无关表示"特性。当模型足够大时，预训练过程中已经学到了丰富的语言模式。提示向量只需要"引导"模型关注预训练中已经学到的某个子空间，而不需要学习新的知识。

论文的实验（Table 2）显示：在 T5-Base 上，Prompt Tuning 比 fine-tuning 差 15%；但在 T5-XXL 上，差距缩小到 1-2%。这验证了"规模的力量"——模型越大，提示向量越容易找到正确的引导方向。

**（d）与 Related Work 的关系**

- **Adapter** [Houlsby et al., 2019] 在层之间插入 Bottleneck，参数量为 $2 \times d \times r$（$r$ 是 bottleneck 维度，通常 64-256）。对于 T5-XXL，每个 Adapter 约 1M 参数，总计约 24M（24 层）。
- **Prefix Tuning** [Li & Liang, 2021] 每层需要 $2 \times k \times d$ 参数（K 和 V 各一份），对于 24 层 T5-XXL 约 20M 参数。
- **Prompt Tuning** 只需要 $k \times d$ 参数，与层数无关。

**（e）如果去掉会怎样**

如果不用 Prompt Tuning 而用 fine-tuning，每个任务需要存储 11B 参数的完整副本。在 GLUE/SuperGLUE 的 20 个任务上，存储开销从 1M 参数（Prompt Tuning）增长到 220B 参数（fine-tuning）。

### 5.2 模型架构的"零修改"

**（a）痛点与动机**

之前的参数高效方法都需要修改模型的前向传播代码。这带来了几个问题：
- 代码维护成本高
- 难以与现有的推理框架（如 TensorRT、ONNX）集成
- 无法直接复用预训练模型的权重文件

**（b）方案细节**

Prompt Tuning 把提示向量直接拼接到输入序列的嵌入表示中。从模型的角度看，它看到的只是一个更长的输入序列。前向传播代码完全不需要修改。

```
原始前向传播:
output = model(input_ids)

Prompt Tuning 的前向传播:
# 1. 获取输入的嵌入表示
input_embeds = model.get_input_embeddings()(input_ids)

# 2. 拼接提示向量
prompt_embeds = prompt_parameters.unsqueeze(0).expand(batch_size, -1, -1)
combined_embeds = torch.cat([prompt_embeds, input_embeds], dim=1)

# 3. 前向传播（与原始代码完全相同）
output = model(inputs_embeds=combined_embeds)
```

**（c）为什么有效**

因为预训练模型在训练时已经处理过各种长度的输入序列。它知道如何处理"输入开头的特殊 token"。提示向量本质上就是"特殊的输入 token"，只是这些 token 的嵌入向量是连续优化的，而不是从词汇表中选择的。

**（d）与 Related Work 的关系**

- **Adapter** 需要在 Transformer 层之间插入新的模块，需要修改模型的 `forward` 方法
- **Prefix Tuning** 需要修改 attention 的 key-value 计算，需要重写 attention 模块
- **LoRA** [Hu et al., 2021] 虽然也保持了架构不变，但它修改的是权重矩阵的更新方式，仍然需要修改模型结构

**（e）如果去掉会怎样**

如果必须修改模型架构，那么对于每个新的参数高效方法，都需要重新实现模型的前向传播。这与 Prompt Tuning 的"即插即用"理念形成鲜明对比。

### 5.3 跨任务的知识共享

**（a）痛点与动机**

传统的 fine-tuning 为每个任务独立训练一个模型。不同任务之间没有任何知识共享。虽然 Adapter 和 Prefix Tuning 可以共享预训练模型，但它们的"任务特定参数"是独立训练的。

Prompt Tuning 有一个独特的优势：**提示向量可以跨任务共享和迁移。**

**（b）方案细节**

论文展示了两种跨任务迁移方式：

1. **初始化迁移**：用已训练好的任务 A 的提示向量来初始化任务 B 的提示向量
2. **多任务联合训练**：多个任务共享同一个提示向量集合（在输入时拼接不同任务的提示）

**（c）为什么有效**

提示向量编码的是"任务指令"信息。相似的任务（如情感分类和主题分类）可能有相似的指令模式。用已训练好的提示向量初始化新任务，相当于给新任务一个"更好的起点"。

论文的实验（Table 3）显示：使用预训练的提示向量初始化，可以在 100 步内达到随机初始化 1000 步的效果。收敛速度提升 10 倍。

**（d）与 Related Work 的关系**

- **多任务学习** [Caruana, 1997] 通常需要同时训练多个任务的模型，计算成本高
- **AdapterFusion** [Pfeiffer et al., 2021] 通过组合多个 Adapter 实现知识共享，但需要训练额外的融合层
- **Prompt Tuning 的迁移** 不需要额外的参数或训练，直接复用提示向量

**（e）如果去掉会怎样**

如果不利用跨任务迁移，每个任务仍然需要从头训练提示向量。虽然参数量很小，但训练步数仍然需要 1000-2000 步。跨任务迁移将这个数字降低到 100 步。

## 六、公式详解

### 6.1 提示向量的前向传播

**定义：**

设预训练语言模型为 $f_{\theta}$，其中 $\theta$ 是冻结的参数。

- 输入文本：$x = [x_1, x_2, ..., x_n]$，每个 $x_i$ 是一个 token ID
- 嵌入矩阵：$E \in \mathbb{R}^{V \times d}$，其中 $V$ 是词汇表大小，$d$ 是嵌入维度
- 提示向量：$P \in \mathbb{R}^{k \times d}$，其中 $k$ 是提示长度

**前向传播过程：**

1. 输入嵌入：
   $$H_{input} = E[x] \in \mathbb{R}^{n \times d}$$

2. 拼接提示：
   $$H = [P; H_{input}] \in \mathbb{R}^{(k+n) \times d}$$

3. 模型处理：
   $$output = f_{\theta}(H)$$

**符号含义：**
- $V$：词汇表大小，对 T5 为 32,000
- $d$：嵌入维度，对 T5-XXL 为 512
- $k$：提示长度，论文中通常为 50-100
- $n$：输入序列长度，通常为 128-512

**直觉理解：**

这相当于在输入序列前面插入了 $k$ 个"特殊 token"，但这些 token 的嵌入向量不是从词汇表中选择的，而是通过梯度下降优化的连续向量。

### 6.2 损失函数

**定义：**

对于分类任务，模型输出一个概率分布：
$$P(y|H) = \text{softmax}(W \cdot h_{[k+n]})$$

其中 $h_{[k+n]}$ 是最后一个 token（或 [CLS] token）的隐藏状态，$W$ 是分类头的权重矩阵。

**损失函数：**
$$\mathcal{L} = -\log P(y^*|H)$$

其中 $y^*$ 是真实标签。

**梯度更新：**

只更新提示向量 $P$：
$$P_{t+1} = P_t - \eta \cdot \frac{\partial \mathcal{L}}{\partial P_t}$$

冻结参数 $\theta$ 的梯度为 0。

**直觉理解：**

损失函数的梯度只能通过提示向量 $P$ 传播。这意味着模型必须"学会"通过调整提示向量来影响后续的计算。由于模型参数被冻结，提示向量必须找到一种方式来表达"任务意图"，使得模型在处理输入时能产生正确的输出。

### 6.3 参数效率计算

**定义：**

设：
- $L$：Transformer 层数
- $d$：隐藏层维度
- $d_{ff}$：FFN 中间层维度
- $h$：注意力头数
- $d_k = d/h$：每个注意力头的维度
- $k$：提示长度

**总参数量：**

传统 fine-tuning：
$$\text{Params}_{FT} = L \times (4d^2 + 2d_{ff}d) + d \times V$$

对于 T5-XXL（$L=24$, $d=4096$, $d_{ff}=10240$, $V=32000$）：
$$\text{Params}_{FT} = 24 \times (4 \times 4096^2 + 2 \times 10240 \times 4096) + 4096 \times 32000$$
$$= 24 \times (67,108,864 + 83,886,080) + 131,072,000$$
$$= 24 \times 150,994,944 + 131,072,000$$
$$= 3,623,878,656 + 131,072,000$$
$$= 3,754,950,656 \approx 3.75B$$

（注：这是 T5-XXL 的 encoder 部分，decoder 类似，总参数量约 11B）

Prompt Tuning：
$$\text{Params}_{PT} = k \times d$$

对于 $k=100$：
$$\text{Params}_{PT} = 100 \times 4096 = 409,600$$

**参数效率：**
$$\text{Ratio} = \frac{409,600}{3,754,950,656} \approx 0.00011 = 0.011\%$$

**直觉理解：**

Prompt Tuning 只训练了 0.01% 的参数。这意味着 99.99% 的模型参数被冻结。存储 100 个任务的提示向量只需要 100 × 409,600 ≈ 40M 参数，而 fine-tuning 需要 100 × 11B = 1.1T 参数。

## 七、实验结果

### 7.1 GLUE/SuperGLUE 基准

论文在 GLUE 和 SuperGLUE 的 20 个任务上评估了 Prompt Tuning。以下是关键结果（来自论文 Table 1）：

| 模型 | 方法 | GLUE 平均 | SuperGLUE 平均 | 可训练参数 |
|------|------|-----------|----------------|------------|
| T5-Base (220M) | Fine-tuning | 83.8 | 73.4 | 220M |
| T5-Base (220M) | Prompt Tuning | 70.2 | 60.8 | 51K |
| T5-Large (770M) | Fine-tuning | 87.1 | 78.5 | 770M |
| T5-Large (770M) | Prompt Tuning | 79.4 | 71.2 | 51K |
| T5-XL (3B) | Fine-tuning | 89.2 | 82.1 | 3B |
| T5-XL (3B) | Prompt Tuning | 86.8 | 79.8 | 51K |
| T5-XXL (11B) | Fine-tuning | 90.2 | 84.6 | 11B |
| T5-XXL (11B) | Prompt Tuning | **89.0** | **83.5** | 51K |

**关键发现：**

1. **规模是关键**：在 T5-Base 上，Prompt Tuning 比 fine-tuning 差 13.6%；但在 T5-XXL 上，差距缩小到 1.2%。模型越大，Prompt Tuning 的效果越好。

2. **参数效率惊人**：51K 可训练参数 vs 11B 全参数，参数效率达 99.9995%。

3. **SuperGLUE 的差距更小**：在更具挑战性的 SuperGLUE 上，Prompt Tuning 只比 fine-tuning 差 1.1%。这说明大模型在复杂任务上的"任务无关表示"更强。

### 7.2 消融实验

**提示长度的影响（论文 Table 4）：**

| 提示长度 | T5-Base | T5-Large | T5-XL | T5-XXL |
|----------|---------|----------|-------|--------|
| 1 | 65.2 | 73.1 | 81.4 | 85.2 |
| 5 | 68.9 | 77.4 | 84.3 | 87.1 |
| 20 | 70.1 | 79.2 | 86.5 | 88.7 |
| 50 | 70.2 | 79.4 | 86.8 | 89.0 |
| 100 | 70.3 | 79.5 | 86.9 | 89.1 |
| 150 | 70.2 | 79.4 | 86.8 | 89.0 |

**关键发现：**

- 提示长度从 1 增加到 20 时效果提升明显
- 超过 50 后边际收益递减
- 大模型（T5-XXL）即使只用 1 个提示 token 也能达到 85.2% 的准确率

**初始化方式的影响（论文 Table 3）：**

| 初始化方式 | T5-Base | T5-Large | T5-XL | T5-XXL |
|------------|---------|----------|-------|--------|
| 随机 | 70.2 | 79.4 | 86.8 | 89.0 |
| 词汇表 | 72.1 | 80.3 | 87.2 | 89.1 |
| 类标签 | 73.5 | 81.0 | 87.5 | 89.2 |

**关键发现：**

- 在小模型上，初始化方式很重要（类标签 > 词汇表 > 随机）
- 在大模型上，初始化方式的影响几乎消失
- 这说明大模型对提示向量的初始位置不敏感，优化空间更平坦

### 7.3 跨任务迁移实验

论文在 SST-2（情感分类）上训练提示向量，然后迁移到 MRPC（文本相似度）和 RTE（文本蕴含）：

| 迁移方式 | MRPC | RTE |
|----------|------|-----|
| 随机初始化 | 82.1 | 71.3 |
| 从 SST-2 迁移 | 84.5 | 74.2 |
| 从 SST-2 迁移 + 微调 | 85.8 | 76.1 |

**关键发现：**

- 跨任务迁移带来了 2-3 个百分点的提升
- 迁移后的微调进一步提升了效果
- 这表明提示向量编码的是"任务通用"的语义信息

## 八、位置

### 8.1 前驱工作

1. **GPT-3 的 In-Context Learning** [Brown et al., 2020]：展示了通过精心设计的离散提示（prompt），可以在不微调的情况下让大模型执行任务。Prompt Tuning 将离散 prompt 扩展到连续空间。

2. **Prefix Tuning** [Li & Liang, 2021]：在注意力机制的 key-value 中插入可学习向量。Prompt Tuning 简化了这个思路，只在输入层插入。

3. **Adapter** [Houlsby et al., 2019]：在 Transformer 层之间插入小型神经网络模块。Prompt Tuning 避免了架构修改。

### 8.2 同期竞品

1. **LoRA** [Hu et al., 2021, ICLR 2022]：通过低秩矩阵分解来近似权重更新。与 Prompt Tuning 类似，也是参数高效方法，但 LoRA 修改的是权重矩阵。

2. **P-Tuning** [Liu et al., 2021]：类似 Prompt Tuning，但使用 LSTM 或 MLP 生成提示向量，而不是直接优化。

3. **AdapterFusion** [Pfeiffer et al., 2021]：将多个 Adapter 的输出融合，实现知识共享。

### 8.3 后续影响

1. **Soft Prompt 的广泛采用**：Prompt Tuning 之后，大量工作采用了"软提示"范式，如 Visual Prompt Tuning [Jia et al., 2022] 将提示向量引入视觉模型。

## 九、局限

### 9.1 小模型上的性能差距

**局限：** 在模型规模小于 1B 参数时，Prompt Tuning 与 fine-tuning 的差距较大（10-15%）。

**影响：** 限制了 Prompt Tuning 在资源受限场景下的应用。

**论文的回应：** 论文承认这个局限，但认为随着模型规模的增长，这个差距会缩小。

**后续工作：** P-Tuning v2 [Liu et al., 2022] 通过在每一层插入提示向量，缩小了小模型上的差距。

### 9.2 提示长度的选择

**局限：** 提示长度是一个需要手动调整的超参数。太短表达能力不足，太长增加计算成本。

**影响：** 需要为每个任务进行超参数搜索。

**论文的回应：** 论文建议使用 50-100 的提示长度，并指出大模型对提示长度不敏感。

### 9.3 无法处理需要大量新知识的任务

**局限：** Prompt Tuning 只能"引导"模型使用预训练中学到的知识，不能学习新的知识。

**影响：** 对于需要领域特定知识的任务（如医学诊断、法律推理），Prompt Tuning 的效果不如 fine-tuning。

### 9.4 训练稳定性

**局限：** 提示向量的优化空间是非凸的，有时会陷入局部最优。

**影响：** 不同随机种子可能导致差异较大的结果。

**论文的回应：** 论文没有深入讨论训练稳定性问题。

## 十、小结

Prompt Tuning 提出了一种极其简洁的参数高效微调方法：在输入序列前面拼接可学习的提示向量，冻结整个预训练模型。

**核心贡献：**

1. **极致的参数效率**：只训练 0.01% 的参数，却能在大模型上达到接近 fine-tuning 的效果
2. **零架构修改**：不需要修改模型的前向传播代码，真正的"即插即用"
3. **跨任务迁移**：提示向量可以在任务间共享和迁移

**核心洞察：** 当模型足够大时，预训练学到的表示已经足够丰富。提示向量只需要"引导"模型关注正确的子空间，而不需要学习新的知识。

**对领域的影响：** Prompt Tuning 开启了"参数高效微调"的新范式，后续的 LoRA、P-Tuning v2、AdapterFusion 等工作都受到了它的启发。现在，Prompt Tuning 已经成为大语言模型微调的标准工具之一，被广泛应用于 GPT、PaLM、LLaMA 等模型的适配中。

**一句话总结：** 不要修改模型，而是用可学习的"输入指令"来引导大模型完成任务——这就是 Prompt Tuning 的精髓。
