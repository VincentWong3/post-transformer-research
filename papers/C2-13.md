## 一、先搞清楚"问题在哪里"

### 1.1 从三个独立模型到"一个模型"

图像分割一直是个分裂的世界。

语义分割要把图片中每个像素标上类别（"这是路、这是天空、这是建筑"），实例分割要区分出每个独立物体（"第一辆车、第二辆车、第三辆车"），全景分割把两者合并——为背景区域（stuff）打标签，为前景物体（thing）做实例区分。

这三个任务长期以来各自为战。每个任务有专门的架构、专门的训练、专门的模型权重。2019 年 Kirillov 等人提出全景分割（panoptic segmentation）的概念，本意是"统一语义和实例分割"。但实际发展事与愿违——全景分割催生了一波新的全景专用架构（Panoptic-FPN、Panoptic-DeepLab），而不是把已有的语义和实例模型统一起来。

### 1.2 "架构统一了，模型没有"

2021-2022 年出现了转机。K-Net、MaskFormer、Mask2Former 这些新的全景架构，**确实做到了一个架构适配三个任务**——不需要换 backbone，不需要改 decoder，只需要换一个 training config 就能分别训练出三个高性能模型。

但这距离"真正统一"还差一步。Mask2Former 在 ADE20K 上，语义、实例、全景每个任务要单独训练 **160K 次迭代**，总共 **480K 次迭代**，产出**三套模型权重**，推理时需要**三个不同的部署副本**。

论文原文说得很直白：

> "Such panoptic architectures can be trained on all three tasks and obtain high performance without changing architecture. They **do** need to, however, be trained individually on each task to achieve the best performance."

作者管这叫 **semi-universal approach**（半统一方案）。架构统一了，但训练和模型没有统一。

### 1.3 为什么 Mask2Former 不能"一次训练管三个任务"？

如果 Mask2Former 的架构本身就能做三个任务，那为什么不能同时在三种 ground truth 上一起训练，得到一个模型搞定所有任务？

作者做了实验（Table 7）：直接拿 Mask2Former 做 joint training——每次随机采样语义/实例/全景的 GT 来训练——结果效果大幅下降。PQ 从各自单独训练的 48.1/34.2/54.5（取决于任务）掉到 joint training 的 48.7（PQ）、33.7（AP）、56.2（mIoU）。

**问题出在哪里？**

作者的假设是：**架构中没有"当前在做什么任务"的引导信号。** 模型面对同一张图片，语义和实例任务需要的输出完全不同——语义任务是"每类一个 mask"，实例任务是"每个物体一个 mask"。模型无从知道当前 batch 要用哪种方式解读 ground truth，因此 joint training 时无法有效学习任务间的差异。

这就像给一个学生同时布置三道不同的题，但不告诉他现在在答哪道。

## 二、核心思路：Task Token

### 2.1 一句话概括

**OneFormer 的核心创新是在模型的输入中增加了一个 task token，告诉模型"现在要做的是语义/实例/全景任务"，然后让模型在三种任务上联合训练。** 训练时随机采样 task token 和对应的 GT，推理时给定 task token 决定输出格式。一次训练，一个模型，三个任务全做。

### 2.2 具体怎么做

**Step 1 — Task Token 生成**

每次训练时，从 {panoptic, instance, semantic} 中均匀采样一个任务（概率各 1/3）。构造一个文本输入："the task is {task}"，用 tokenizer 编码后经过一个线性层 + LayerNorm，得到一个 1 维的 task token 向量 $\mathbf{Q}_{\text{task}}$ 。

**Step 2 — Task-conditioned Query 初始化**

OneFormer 使用 N 个 object queries（object queries = transformer decoder 的输入查询向量）。其中 N-1 个 queries 被初始化为 task token 的重复（ $\mathbf{Q'} = \text{repeat}(\mathbf{Q}_{\text{task}}, N-1)$ ），第 N 个 query 就是 $\mathbf{Q}_{\text{task}}$ 本身。这使得**每个 query 都知道当前任务是什么**。

**Step 3 — Task-specific GT 生成**

所有标注从全景标注推导而来。例如当 task = "instance" 时，只提取 thing 类别的非重叠 mask，忽略 stuff 区域；当 task = "semantic" 时，每个类别只生成一个整体 mask（不区分实例）；当 task = "panoptic" 时，stuff 区域整体 mask + thing 区域非重叠 mask。

这意味着 **OneFormer 训练只需要一套全景标注**——语义和实例标注是从中推导出来的。

**Step 4 — Query-Text Contrastive Loss**

在训练时还引入了一个 **query-text contrastive loss**。对于当前 task 的 GT，提取每类的 mask 数量，构造一个文本列表：每个 mask 对应 "a photo with a {CLS}"。再加一些 "a/an {task} photo" 作为 no-object 填充项。这个文本列表通过一个 6 层 transformer text encoder 编码为 text queries $\mathbf{Q}_{\text{text}}$ 。

然后计算 object queries 和 text queries 之间的对比学习损失：

$$\mathcal{L}_{\mathbf{Q} \leftrightarrow \mathbf{Q}_{\text{text}}}$$

这个损失的意图是：
1. 让同任务的 queries 和 text 距离更近 → **学习任务间的差异**
2. 让同类别但不同 mask 的 queries 和对应的 text 距离更近 → **减少类别误判**

论文的消融实验（Table 5）显示，去掉这个对比损失，PQ 从 67.2 暴跌到 **58.8（-8.4%）**，AP 从 45.6 降到 **42.4（-3.2%）**。

**Step 5 — 推理时可以省掉 text mapper**

text mapper（包括 text encoder 和 text context embeddings）只在训练时使用。推理时直接丢掉，模型大小和 FLOPs 与 Mask2Former 相当。这是一个优雅的设计——训练时用文本辅助学习，推理时零开销。

### 2.3 与 Mask2Former 的关键区别

| 特性 | Mask2Former | OneFormer |
|------|-------------|-----------|
| 训练方式 | 每个任务单独训练 160K 次 | 一次联合训练 160K 次 |
| 总训练量 | 480K 次迭代 | 160K 次迭代 |
| 模型数量 | 3 个 | 1 个 |
| 任务引导 | 无 | Task token 输入 |
| Query 初始化 | 可学习 | 由 task token 重复 |
| 文本辅助 | 无 | Query-text 对比损失 |
| 标注需求 | 语义/实例/全景各一套 | 仅需全景标注 |

**核心区别：Mask2Former 的架构做到了"一个架构兼容三个任务"，OneFormer 做到了"一个模型做三个任务"。**

## 三、整体架构与 Forward 流程

### 3.1 整体结构

```text
输入图像: H×W×3     任务输入: "the task is {task}"
    │                      │
    ▼                      ▼
┌─────────────────────┐    Tokenizer + Linear + LN
│ Backbone +           │    └──→ Q_task (1-D task token)
│ Pixel Decoder        │
│ (6× MSDeformAttn)     │
│ → multi-scale features│
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Transformer Decoder  │  ← 输入: multi-scale features
│ (L=3 stage, 9层)    │        + N 个 object queries
│                      │
│ Stage 1: 3 层 decoder│  Object queries 初始化:
│ Stage 2: 3 层 decoder│  N-1 个 = repeat(Q_task)
│ Stage 3: 3 层 decoder│  第 N 个 = Q_task
│                      │  
│ masked cross-attn    │
│ self-attn + FFN      │
└─────────┬───────────┘
          │
          ▼
┌─────────────────────┐
│ Task-dynamic 输出     │
│ ┌─ class logits      │
│ └─ binary masks      │
└─────────────────────┘
```

### 3.2 训练时 vs 推理时的差异

**训练时完整流程：**

```text
图像 + task token → backbone + pixel decoder → 特征图
task token → 重复 → object queries
↓
Transformer Decoder → mask + class 预测
↑
对比学习 ← text queries ← text encoder ← "a photo with a {CLS}..."
                          ↑
                      从当前 task GT 提取
```

**推理时简化流程：**

```text
图像 + task token → backbone + pixel decoder → 特征图
task token → 重复 → object queries
↓
Transformer Decoder → mask + class 预测
                      （text mapper 已丢弃）
```

推理时不需要文本编码器，也不需要 GT 信息。给定 task token（"the task is panoptic" / "the task is instance" / "the task is semantic"），模型动态调整输出。

## 四、核心实验与结果

### 4.1 主要结果

**ADE20K val 数据集（Table 1 & Table 7, Swin-L backbone, 160K 迭代, 单尺度推理）：**

| 方法 | PQ | AP | mIoU | 参数量 |
|------|-----|-----|------|-------|
| Mask2Former-联合训练 | 48.7 | 33.7 | 56.2 | 216M |
| **OneFormer** | **49.8** | **35.9** | **57.0** | **219M** |
| OneFormer 相对于联合训练提升 | +1.1 | +2.2 | +0.8 | |

即使只做 160K 次迭代（一次性训练 = Mask2Former 三个任务总训练量的 1/3），OneFormer 在三个指标上全部超越 Mask2Former 的联合训练版本。

**Cityscapes val（Table 2, Swin-L backbone, 90K 迭代, 单尺度）：**

| 方法 | PQ | AP | mIoU |
|------|-----|-----|------|
| Mask2Former（各自最佳） | 66.6 | 43.7 | 83.3 |
| **OneFormer（单模型联合训练）** | **67.2** | **45.6** | **83.0** |
| 提升 | +0.6 | +1.9 | -0.3 |

在 Cityscapes 上，OneFormer 的 PQ 和 AP 超越了各自单独训练的 Mask2Former 最佳模型，虽然 mIoU 略有差距但 AP 的提升非常显著（+1.9%）。

**COCO val2017（Table 3, Swin-L backbone, 100 epochs）：**

| 方法 | PQ | PQ\textsuperscript{Th} | PQ\textsuperscript{St} | AP | mIoU |
|------|-----|------|------|-----|------|
| Mask2Former（各自最佳） | 57.8 | 64.2 | 48.1 | 48.7 | 67.4 |
| **OneFormer（单模型联合训练）** | **57.9** | **64.4** | **48.0** | **49.0** | **67.4** |

COCO 上 OneFormer 在 PQ 和 AP 上均超越各自单独训练的 Mask2Former。

OneFormer 发布时正赶上 ConvNeXt（纯卷积，2022）和 DiNAT（邻域注意力，2022）的出现。换上这些 backbone 后性能进一步提升：

### 4.3 资源节省

论文的核心主张之一：**相比 Mask2Former 分别训练三个模型，OneFormer 将训练时间和存储需求降低至 1/3。**

具体计算：
- Mask2Former 在 ADE20K 上训练总量：160K × 3 = 480K 次迭代
- OneFormer 在 ADE20K 上训练总量：160K × 1 = 160K 次迭代
- 存储：3 套模型权重 → 1 套
- 推理部署：3 个独立副本 → 1 个（用 task token 切换）

## 五、消融实验解析

### 5.1 架构组件消融（Table 4, Cityscapes, Swin-L）

| 变体 | PQ | AP | mIoU |
|------|-----|-----|------|
| 完整 OneFormer | 67.2 | 45.6 | 83.0 |
| 去掉 task token | 66.5 (-0.7) | 43.3 (-2.3) | 82.9 (-0.1) |
| 去掉 learnable text context | 62.7 (-4.5) | 45.0 (-0.6) | 82.8 (-0.2) |
| 去掉 task-guided query 初始化 | 65.8 (-1.4) | 44.5 (-1.1) | 83.1 (+0.1) |

三个组件各有贡献：
- **Task token 对 AP 影响最大（-2.3%）**——说明没有任务引导时，实例分割的物体检测能力大打折扣
- **Learnable text context 对 PQ 影响最大（-4.5%）**——文本上下文 embedding 帮助模型理解图像中不同 mask 的语义关系
- **Task-guided query 初始化贡献 PQ +1.4%**

### 5.2 损失函数消融（Table 5, Cityscapes, Swin-L）

| 变体 | PQ | AP | mIoU |
|------|-----|-----|------|
| Query-text contrastive loss（完整） | 67.2 | 45.6 | 83.0 |
| Query classification loss（替代） | 66.4 (-0.8) | 44.7 (-0.9) | 82.6 (-0.4) |
| 无对比损失 | 58.8 (-8.4) | 42.4 (-3.2) | 82.5 (-0.5) |

对比损失的关键性在 PQ 上体现得淋漓尽致——**去掉之后 PQ 直接掉了 8.4 个点**。用分类损失替代也不行，效果仍然下降。

原因是：分类损失只能让模型区分不同类别，但**对比损失同时区分了不同任务**——同任务的 queries 之间相似度高，跨任务的差异大。

### 5.3 任务 Token 输入消融（Table 8, ADE20K, Swin-L）

| 推理时 task 设置 | PQ | PQ\textsuperscript{Th} | PQ\textsuperscript{St} | AP | mIoU |
|-----------------|-----|------|------|-----|------|
| "the task is panoptic" | 49.3 | 49.6 | 50.2 | 35.8 | 57.0 |
| "the task is instance" | 33.1 | 48.8 | 1.5 | 35.9 | 26.4 |
| "the task is semantic" | 40.4 | 35.5 | 50.2 | 25.3 | 57.0 |

这个消融验证了 OneFormer 的 **task-dynamic** 特性：

- 当 task = "instance" 时，PQ\textsuperscript{St}（stuff 区域的 PQ）直接掉到 **1.5%**——模型知道"现在是实例分割任务"，背景区域几乎不输出 mask。
- 当 task = "semantic" 时，PQ\textsuperscript{Th}（thing 区域的 PQ）掉到 **35.5%**——模型知道"现在是语义分割任务"，不区分不同实例。
- 当 task = "panoptic" 时，三项指标都正常——模型同时处理 stuff 和 thing。

**一个模型，同一个 forward 过程，仅仅因为输入字符串不同，产生完全不同的输出模式。** 这就是 task token 的威力。

### 5.4 Joint Training 对比（Table 7, ADE20K, Swin-L, 160K 迭代）

| 方法 | PQ | AP | mIoU |
|------|-----|-----|------|
| Mask2Former-Joint | 48.7 | 33.7 | 56.2 |
| **OneFormer** | **49.8 (+1.1)** | **35.9 (+2.2)** | **57.0 (+0.8)** |

在同一数据、同一训练设置下，OneFormer 的架构设计带来了 AP +2.2%、PQ +1.1%、mIoU +0.8% 的提升。考虑到两个模型的参数量几乎相同（219M vs 216M），这个差距完全来自 **task token + task-conditioned query + contrastive loss**。

## 六、与 Mask2Former 的技术对比

### 6.1 架构对比

两者共享的基本结构：
- Transformer decoder + mask classification 范式
- 多尺度特征提取（MSDeformAttn）
- 逐层 auxiliary loss

**OneFormer 独有的设计：**

1. **Task token 输入**：Mask2Former 的输入只有图像，OneFormer 多了一个文本形式的 task 输入
2. **Task-conditioned query 初始化**：Mask2Former 的 queries 是可学习的（learnable），OneFormer 的 queries 由 task token 重复生成 + 一个 $\mathbf{Q}_{\text{task}}$ 组成
3. **Query-text 对比损失**：Mask2Former 只有分类 + mask 损失，OneFormer 多了一个 text 侧的对比损失
4. **Text mapper（训练时）**：Mask2Former 没有文本分支，OneFormer 在训练时用文本辅助，推理时丢弃

### 6.2 为什么 Mask2Former 在 joint training 上效果差？

论文 Table 7 的 Mask2Former-Joint 直接证明了：**没有 task guidance 的 joint training 效果差**。

原因分析：Mask2Former 的 queries 在训练过程中对每个任务学到不同的权重更新模式。当训练数据在三种任务间切换时，queries 的梯度方向互相拉扯——当前 batch 做语义时要求 queries 关注 stuff 类别的整体 mask，下一个 batch 做实例时又要求 queries 关注 thing 类别的个体 mask。**没有显式的 task signal，queries 不知道自己在为什么任务优化，最终学到的是一种"折中"而非"精通"的表征。**

OneFormer 的 task token 提供了这个显式信号。queries 以 task token 为初始化起点，加上 task-specific text queries 的对比学习约束，使得同一个模型的不同 forward 实际上激活了不同的参数子空间。

### 6.3 关于 query 数量的差异

OneFormer 默认用 250 个 queries（ADE20K/Cityscapes）或 150 个 queries（COCO），而 Mask2Former 默认用 200 个。这不是本质差异——作者消融了不同数量的 queries（附录 Table I），发现 150 和 200 效果接近。关键区别是 OneFormer 的 queries 是 **task-conditioned 的**，而 Mask2Former 的 queries 是 **全局可学习的**。

论文还发现 Mask2Former 用 250 个 queries 时性能反而下降（论文脚注："Mask2Former's performance with 250 queries is not listed, as its performance degrades with 250 queries"），而 OneFormer 用 250 个 queries 效果很好。这说明 **task-conditioned queries 在处理更多 queries 时有更好的扩展性**——每个 query 从一开始就知道自己的"任务角色"，不会因为输入空间变大而混淆。

## 八、局限性与后续方向

### 8.1 标注需求：仍需要全景标注

OneFormer 只需要全景标注就能训练，这部分是好的。但全景标注本身就很昂贵——COCO 的 133 类全景标注需要大量人工。如果能做到只用部分标注或弱标注就能联合训练，会更进一步。

### 8.2 推理时 task token 的人为选择

推理时需要手动设定 task token。这意味着部署时需要外部逻辑决定当前场景用哪个 task。如果能自动检测任务类型（例如从输入数据格式推断），会更"统一"。

但这其实是一个实用层面的问题——部署时硬编码 task token 很简单，真正的问题在于模型只是一个工具，任务选择本身始终需要外部逻辑。

### 8.3 COCO 的标注内部不一致

论文在附录 F 中详细分析了 COCO 数据集的问题：COCO 的全景标注和实例标注不是从同一源头生成的，存在大量不一致。例如某些图片中，实例标注把"领带"合并到"人"里，把"狗"和"船"合并成一个实例。这种情况下一套标注训练的模型反而更统一——因为 OneFormer 只看全景标注，不受实例标注的不一致性影响。

这说明 OneFormer 的"只需要一套标注"策略不仅简化了数据需求，还在数据质量层面带来了一致性优势。

### 8.4 对 backbone 的依赖

OneFormer 的效果严重依赖 backbone 的质量——从 Swin-L（49.8 PQ）到 ConvNeXt-L（50.0 PQ）到 DiNAT-L（51.5 PQ），PQ 的提升基本来自 backbone 的提升而非 OneFormer 本身的改进。这意味着 OneFormer 作为框架的"天花板"很大程度上由 backbone 决定。

## 九、技术树中的意义

站在 C2 分支的视角看，OneFormer 完成了 mask classification 范式的一次关键突破：

**C2-11 (MaskFormer)** 把语义分割重新定义为 mask classification，证明了**一个架构**可以做多种分割任务。

**C2-12 (Mask2Former)** 加了 masked cross-attention 和 deformable attention，把性能拉到 SOTA，但还需要**三个模型**。

**C2-13 (OneFormer)** 加了 task token 和 query-text contrastive，把三个模型缩成**一个模型**。

这条技术路线的演进本身就在回答一个问题：**mask classification 范式的尽头是什么？** 如果 MaskFormer 证明了"架构统一"，Mask2Former 证明了"性能统一"，OneFormer 就是证明了"模型统一"。

### OneFormer 没能做到的事

它没有提出新的 backbone，没有改变 Transformer decoder 的基本结构，没有新的 attention 机制。它的创新在于**输入信号的设计**——用一个简单的文本 token 引导模型在多个任务间切换。

这个设计的优雅之处在于它的**轻量**：不需要改 backbone，不需要改 decoder 结构，不需要额外的推理开销。在训练时加入文本辅助，推理时丢掉——对比损失带来的好处是"免费的"（只要训练时有文本能力）。

### 对后续工作的影响

OneFormer 的理念——用一个输入 token 切换任务——在后来的视觉通用模型中有所体现。Mask DINO 等统一检测和分割的工作借鉴了这种"task token + query conditioning"的思路。

但这篇论文更大的影响在于：**它让社区看到"统一的极限在哪"**。在 OneFormer 之后，"一次训练搞定所有分割任务"不再是梦，而是可复现的事实。下一个问题变成了：能不能用一个模型搞定**检测 + 分割 + 深度估计**甚至更多任务？这是一个开放的课题。

## 十、小结

OneFormer 解决的是一个看起来简单但做起来困难的问题：**能不能只训练一次，让一个模型同时精通三个分割任务？**

答案是可以。方法也很直观——给模型一个"任务 ID"（task token），告诉它当前在做什么，然后让它在三种任务数据上一起训练。

但这个"直观"的方法在之前没人做成，是因为它依赖几个关键洞察的组合：

1. **Task token** 提供任务信号（没有这个，joint training 学不好）
2. **Task-conditioned query 初始化** 让每个 query 从一开始就知道任务角色
3. **Query-text contrastive loss** 让模型理解任务间的语义差异（不只是一个 one-hot 编码）
4. **从全景标注推导其他标注** 让数据准备只有一套

在 C2 分支的技术树上，OneFormer 是 mask classification 范式迈出的"统一最大化"一步。它不是 bridge node——它没有连接不同的研究领域；但它是一个重要的 **refinement node**——在同一范式下把"统一"推到了极致。而实现这个极致的核心武器，不过是一个文本 token。
